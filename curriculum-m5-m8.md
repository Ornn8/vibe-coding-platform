# M5-M8 课程架构 — Unit-Step 详细设计

> 延续 curriculum-v3.md 的设计标准：
> - Unit-Step 三层结构（导航栏只显示 Unit）
> - 四大方法论原则贯穿（规划驱动/上下文固定/胶水编程/闭环交付）
> - 每个 Step 包含知识点讲解 + 场景示例 + L1→L3 递进练习
> - M7-M8 标记为"进阶选修"

---

## M5：迭代交付与 Debug（5 个 Unit）

> 本模块的核心转变：从"一次性 vibe 出成品"到"分步迭代、持续改进"。
> 方法论重心：闭环交付 + 上下文固定。

### Unit 5.1：分步实施——不要一口吃成胖子
预计时长：30 分钟 ｜ 方法论标签：闭环交付

| Step | 知识点 | 场景/示例 | 练习 |
|------|--------|-----------|------|
| Step 1 | 为什么"一个 prompt 搞定一切"注定失败 | 场景对比：一次性 prompt 做完整功能 vs 分 5 步逐步实现的产出质量差异（给出两段真实代码/截图对比） | L1：两段产出哪个质量更高？为什么？ |
| Step 2 | 步骤粒度控制：每步 15-30 分钟可完成 | 同一个功能的三种拆法对比——太粗（一步搞定）/ 太细（20 步）/ 合适（5 步），各自的问题 | L1：判断 3 种拆法哪个粒度最合适；L2：把"用户登录注册功能"拆成 4-6 步 |
| Step 3 | "在我验证前不要开始下一步"——给 AI 的关键纪律指令 | 展示带和不带这句指令的 AI 行为差异 | L2：在你的项目中实践一次分步实施（先做第一步并验证）；L3：记录一次完整的 3 步分步实施过程 |

### Unit 5.2：Debug 方法论——用 AI 修 AI
预计时长：35 分钟 ｜ 方法论标签：闭环交付

| Step | 知识点 | 场景/示例 | 练习 |
|------|--------|-----------|------|
| Step 1 | AI 产出的代码报错了怎么办：不要慌，不要自己猜，把错误信息喂给 AI | 演示完整 debug 流程：看到报错 → 复制错误信息 → 粘贴给 AI → AI 定位原因 → 修复 → 验证 | L1：以下 3 个 debug 做法，哪个最有效？ |
| Step 2 | 三类常见错误的处理方式 | JavaScript 运行时错误（打开 F12 复制）、视觉样式问题（截图给 AI）、逻辑错误（描述"预期 vs 实际"） | L2：故意引入一个 CSS 问题（比如把颜色值写错），用截图方式让 AI 修复 |
| Step 3 | 什么时候该放弃当前对话重新开始 | 场景：对话已经 20 轮了越修越乱——上下文污染的信号和处理方式 | L2：判断 3 个对话场景是否应该重启；L3：在自己的项目中完成一次完整的 debug 并记录排查日志 |

### Unit 5.3：Memory Bank 与上下文管理
预计时长：40 分钟 ｜ 方法论标签：上下文固定

| Step | 知识点 | 场景/示例 | 练习 |
|------|--------|-----------|------|
| Step 1 | 为什么 AI 会"忘记"你之前说过的事——上下文窗口的局限性 | 类比：AI 的记忆像一块白板，写满了就得擦掉旧内容。对话越长，早期内容越容易被"挤出去" | L1：判断题——以下哪些情况下 AI 可能"忘记"之前的上下文 |
| Step 2 | Memory Bank 工作流：用文件把项目记忆固化下来 | 展示 memory-bank 的文件结构：progress.md（当前进度）、decisions.md（已做的技术决策）、context.md（项目背景） | L2：为你的 M4 项目创建一个 memory-bank 目录，写出 progress.md |
| Step 3 | 跨会话恢复上下文的 prompt 模板 | 展示标准 prompt："阅读 memory-bank 所有文件，阅读 progress.md 了解之前的工作进度，然后继续实施计划第 X 步。在我验证测试前不要开始下一步。" | L2：用这个模板在一个新对话中恢复项目上下文；L3：在你的项目中连续两天使用 memory-bank 工作流，记录体验 |
| Step 4 | 何时重置对话 vs 何时继续 | 决策表格：对话轮次 > 15 / AI 开始重复错误 / 切换到完全不同的任务 → 重置；在同一功能内迭代 / AI 理解力还在 → 继续 | L3：回顾你过去的项目对话，分析有没有应该重置但没重置的情况 |

### Unit 5.4：版本控制与工具协作
预计时长：35 分钟 ｜ 方法论标签：闭环交付

| Step | 知识点 | 场景/示例 | 练习 |
|------|--------|-----------|------|
| Step 1 | 代码改崩了怎么办——Git 基础和 Checkpoint | 场景："你让 AI 改了一个功能，结果整个页面白屏了。如果没有版本记录，你只能从头来。" Git init / add / commit 的最简用法 + Claude Code 的 checkpoint 功能 | L2：在你的项目中做 3 次 commit，每次对应一个功能变更 |
| Step 2 | Codex 入门：异步代码生成 | Codex 和 Claude Code 的定位差异——Claude Code 是交互式（边对话边做），Codex 是异步式（给 spec 让它跑，你去喝杯咖啡回来看结果） | L2：用 Codex 生成一个独立的功能模块 |
| Step 3 | 什么场景用什么工具 | 决策表格：探索性开发/debug → Claude Code；明确 spec 的批量任务 → Codex；快速原型 → GUI 工具；日常编辑 → Cursor | L1：给 5 个场景选工具；L3：同一个任务分别用两种工具完成，对比体验并记录 |

## M6：扩展能力——Skill / MCP / 上下文工程（5 个 Unit）

> 本模块的核心转变：从"会用工具"到"驾驭工具"——理解 AI 工具背后的能力架构。
> 方法论重心：胶水编程 + 上下文固定。

### Unit 6.1：从提示词工程到上下文工程
预计时长：35 分钟 ｜ 方法论标签：上下文固定

| Step | 知识点 | 场景/示例 | 练习 |
|------|--------|-----------|------|
| Step 1 | 提示词工程的局限：单次对话的 prompt 写得再好，也解决不了长期项目的上下文管理问题 | 场景对比：一个短项目（1 小时完成）prompt 够用 vs 一个长项目（2 周）上下文不断退化 | L1：判断以下场景适合"纯 prompt 管理"还是需要"上下文工程" |
| Step 2 | 上下文工程是什么：管理 AI 决策所需的全部信息——系统指令 + 工具配置 + 外部数据 + 对话历史 | 用一个信息层次图说明：系统层（CLAUDE.md）→ 项目层（memory-bank）→ 任务层（当次 prompt）→ 工具层（MCP/Skill） | L2：为你的项目画出这四层信息分别是什么 |
| Step 3 | 上下文退化的信号和应对策略 | 信号清单：AI 开始重复之前的错误、忘记你定的规则、产出风格突然变化、答非所问。应对：压缩历史 / 重新注入关键上下文 / 开新对话 | L2：回顾你的项目对话记录，找出 1-2 个上下文退化的例子；L3：用应对策略处理一次退化情况 |

### Unit 6.2：Skill 系统——给 AI 装操作手册
预计时长：40 分钟 ｜ 方法论标签：胶水编程

| Step | 知识点 | 场景/示例 | 练习 |
|------|--------|-----------|------|
| Step 1 | MCP 提供"手"，Skill 提供"操作手册" | 类比："MCP 让 AI 能连接数据库（有了手），但 Skill 告诉 AI 怎么写高效安全的 SQL（有了操作手册）。光有连接能力不等于知道怎么用好。" | L1：MCP 和 Skill 分别解决什么问题？（选择题） |
| Step 2 | Skill 的结构：SKILL.md 详解 | 展示 vibe-coding-cn 中 postgresql Skill 的真实结构，逐段解释：触发条件、约束、模式、示例 | L2：分析一个 Skill 的结构——标注每个部分的作用 |
| Step 3 | 在 Claude Code 中使用现有 Skill | 操作演示：如何在项目中加载 Skill、Skill 被触发时 AI 行为的变化 | L2：在你的项目中加载一个现有 Skill 并观察效果 |
| Step 4 | Skill 就是胶水编程的高级形态 | 回到胶水编程原则：Skill 不是让 AI "发明"新知识，而是把已经验证的最佳实践"注入"给 AI | L3：找到一个你常用的技术/库，思考它需要什么样的 Skill（列出 3-5 条应该告诉 AI 的最佳实践） |

### Unit 6.3：编写你自己的 Skill
预计时长：45 分钟 ｜ 方法论标签：胶水编程

| Step | 知识点 | 场景/示例 | 练习 |
|------|--------|-----------|------|
| Step 1 | SKILL.md 编写规范：从需求到可复用 Skill | 模板详解：name / description（触发条件）/ 约束 / 模式（do & don't）/ 示例 | L2：用模板为"Tailwind CSS"写一个简单的 Skill 骨架 |
| Step 2 | 好 Skill vs 差 Skill：什么决定质量 | 对比表格：触发条件太模糊 vs 精确、约束太少 vs 恰当、示例缺失 vs 充分 | L1：判断 3 个 Skill 片段哪个质量更高 |
| Step 3 | 元技能：用 AI 生成 Skill | 介绍"生成 Skill 的 Skill"概念——给 AI 一份技术文档/API 参考，让它帮你提炼为 SKILL.md | L3：为你在项目中使用的某个库/框架，用元技能方法生成一个 Skill |
| Step 4 | 测试和迭代你的 Skill | 怎么验证 Skill 是否有效：在真实任务中使用 → 观察 AI 行为变化 → 根据不足改进 | L3：在你的项目中使用你自建的 Skill 完成一个任务，记录效果和改进点 |

## M7：Agentic Engineering 与多 Agent 协作（4 个 Unit）⚡ 进阶选修

> 本模块的核心转变：从"你指挥一个 AI"到"你编排多个 AI Agent 协作"。
> 方法论重心：全部四大原则的高级应用。
> 前置要求：完成 M1-M6。

### Unit 7.1：从 Vibe Coding 到 Agentic Engineering
预计时长：35 分钟 ｜ 方法论标签：规划驱动

| Step | 知识点 | 场景/示例 | 练习 |
|------|--------|-----------|------|
| Step 1 | Karpathy 的进化论：为什么 Vibe Coding 需要升级 | Vibe Coding 用于生产时暴露的问题——AI Slop（表面正确的垃圾代码）、安全漏洞规模化、认知负债累积 | L1：识别 3 段代码中哪段是 AI Slop |
| Step 2 | Agentic Engineering 的定义：编排 Agent 而不是写代码 | 角色转变对比表：Vibe Coder（描述需求 → 接受产出）vs Agentic Engineer（设计系统 → 编排 Agent → 审查产出） | L2：用自己的话描述两者的核心区别 |
| Step 3 | PEV 循环的工程化：从个人习惯到系统化流程 | 展示 PEV 在 Agentic 场景中的变化——Plan 由规划 Agent 完成、Execute 由编码 Agent 完成、Verify 由测试 Agent 完成 | L2：为一个功能设计 PEV 的 Agent 分工方案 |

### Unit 7.3：质量守卫与认知负债管理

### Unit 8.2：上线前的安全检查
预计时长：30 分钟 ｜ 方法论标签：闭环交付

| Step | 知识点 | 场景/示例 | 练习 |
|------|--------|-----------|------|
