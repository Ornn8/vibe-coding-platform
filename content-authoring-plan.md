## 编写规范

### MDX 文件结构

```mdx
---
title: "课节标题"
description: "一句话描述"
module: "m1"
order: 1
---

正文内容...

<Exercise id="ex-1.1.1" title="练习标题">
练习描述和要求...
</Exercise>

---

**参考来源**
- [来源名称](URL) — 引用了什么内容
```

### 写作原则

1. **口语化但不随意**：像一个有经验的朋友在跟你聊天，不是在念教科书
2. **概念先行，工具跟进**：先让读者理解"为什么"，再讲"怎么做"
3. **每个知识点后紧跟练习**：不要把练习都堆到文章末尾
4. **用真实例子**：不要用 foo/bar/hello world，用"记账 App"、"待办清单"这类具体场景
5. **代码块标注语言**：所有代码块必须指定语言（如 ```bash, ```typescript, ```markdown）
6. **图文结合**：适当使用表格对比、流程描述，避免纯文字大段落
7. **每篇 1500-3000 字**：不宜太短（没有深度）也不宜太长（读不完）
8. **引用来源**：课程内容基于开源仓库重新编写，在文末注明参考来源

### 1.2 核心概念辨析

**文件**: `src/content/zh/m1/1.2-core-concepts.mdx`

**核心知识点**:
- **Vibe Coding**：用自然语言描述需求，AI 生成代码，你不逐行审查，信任"氛围"。适合原型、个人项目、快速验证
- **Spec Coding**：写详细的规格说明（PRD/Plan），AI 严格按规格执行，人类审查每一步。适合生产级项目
- **Agentic Engineering**：编排多个 AI Agent 协作完成任务，人类作为监督者和架构师。是 Vibe Coding 的专业化进化
- 三者不是互斥的，而是一个成长光谱：Vibe → Spec → Agentic
- 本课程的路径：M1-M3 教你 Vibe，M4-M6 教你 Spec，M7-M8 教你 Agentic

**写作要点**:
- 用一个具体例子贯穿三种模式的对比（比如"做一个天气查询应用"，分别用三种方式会怎么做）
- 用表格对比三种模式的适用场景、优缺点、所需技能
- 强调：不存在"最好的模式"，只有"最合适的模式"

**练习**:

```
<Exercise id="ex-1.2.1" title="判断适合哪种模式">
以下三个场景，分别适合 Vibe Coding、Spec Coding 还是 Agentic Engineering？说明理由。

1. 周末想做一个生日倒数计时器送朋友
2. 公司要开发一个处理用户支付的功能模块
3. 团队需要同时重构 5 个微服务的 API 接口

请在下方输入你的判断和理由。
</Exercise>
```

**参考来源**:
- datawhalechina/vibe-vibe 第 1.2 节（Vibe Coding 与 Spec Coding 概念）
- Karpathy Agentic Engineering 阐述

### 1.3 工具全景图

**文件**: `src/content/zh/m1/1.3-tool-landscape.mdx`

**文件**: `src/content/zh/m1/1.4-hello-world.mdx`

**核心知识点**:
- 实操演示：用 Bolt.new（或 Lovable）在 3 分钟内生成一个网页
- 演示用的 prompt 示例（逐步展示输入什么、AI 输出什么）

- 给出具体的 prompt 示例（做一个个人介绍页面的完整 prompt）
- 强调：第一次做出来的效果不重要，重要的是"你用自然语言让 AI 帮你做了一个东西"

## M2：需求定义与产品思维（4 节课）

### 2.1 灵魂三问

**文件**: `src/content/zh/m2/2.1-soul-questions.mdx`

**核心知识点**:
- 为什么大多数人的第一个 vibe coding 项目都失败了？因为他们不知道自己要做什么
- 灵魂三问框架：
  1. **用户是谁？**——不是"所有人"，要具体到一个画像（年龄、职业、痛点场景）
  2. **痛点在哪？**——用户现在怎么解决这个问题？为什么现有方案不够好？
  3. **为何用你？**——你的方案比现有方案好在哪里？差异化是什么？
- 灵魂三问不是写给投资人看的，是写给 AI 看的——AI 需要明确的上下文才能产出好代码
- 实例演示：用"背单词 App"走一遍灵魂三问

**写作要点**:
- 开头用一个反面例子："帮我做一个背单词软件"→ AI 产出了一个什么都有但什么都不好用的东西
- 然后用同一个需求走一遍灵魂三问，展示差别
- 强调：这不是产品经理的专属技能，是和 AI 有效沟通的前提

**练习**:

```
<Exercise id="ex-2.1.1" title="为'背单词 App'回答灵魂三问">
假设你要做一个背单词 App，请回答：
1. 用户是谁？（描述一个具体的人：年龄、职业、英语水平、使用场景）
2. 痛点在哪？（他/她现在怎么背单词？有什么不满？）
3. 为何用你？（你的 App 和市面上的百词斩/墨墨有什么不同？）

每个问题至少写 2-3 句话。
</Exercise>
```

**参考来源**:
- datawhalechina/vibe-vibe 第 2.3 节（灵魂三问）

### 2.2 MVP 思维

**文件**: `src/content/zh/m2/2.2-mvp-thinking.mdx`

**核心知识点**:
- 什么是 MVP（Minimum Viable Product）：能跑的最小版本，不是"缩水版"，是"验证核心假设的最短路径"
- 功能优先级框架 P0/P1/P2：
  - **P0**（必须有）：没有它产品无法使用的核心功能
  - **P1**（应该有）：提升体验但不影响核心使用
  - **P2**（可以有）：锦上添花，后续迭代再加
- 为什么功能堆砌是 vibe coding 的大敌：AI 擅长做单个功能，但一次给太多功能会导致上下文爆炸、产出质量下降、代码结构混乱
- 实例：一个电商 App 的 20 个功能，怎么分 P0/P1/P2
- 关键原则：对 AI 说"这个先不做"是一种能力

**写作要点**:
- 用一个具体的功能列表做分级演示
- 强调"砍功能"不是降低标准，而是聚焦价值
- 讲一个"因为一次给太多功能导致 AI 产出崩溃"的真实场景

**练习**:

```
<Exercise id="ex-2.2.1" title="功能优先级分级">
以下是一个"记账 App"的功能列表，请按 P0/P1/P2 分级：

- 添加支出记录（金额、类别、日期）
- 查看支出列表
- 按月统计总支出
- 支持多种货币
- 添加收入记录
- 预算设置和超支提醒
- 数据导出为 Excel
- 拍照识别小票
- 多人共享账本
- 支出趋势图表

说明你分级的理由：哪些是"没有它就不能用"的？
</Exercise>
```

**参考来源**:
- datawhalechina/vibe-vibe 第 2.1-2.2 节（MVP 思维、拒绝功能堆砌）

### 2.3 用户旅程地图

**文件**: `src/content/zh/m2/2.3-user-journey.mdx`

**核心知识点**:
- 什么是用户旅程地图：从用户视角描述"使用产品的完整流程"，从打开到完成目标的每一步
- 为什么 AI 需要它：AI 不懂你脑子里的"用户应该怎么用这个东西"，旅程地图把隐性知识变成显性描述

- 实例：为"记账 App"画一个简单的用户旅程（从打开到记完一笔账）
- 旅程地图如何转化为给 AI 的 prompt：把每一步变成一个功能描述

- 用文字描述即可，不需要画图工具（保持零门槛）

### 2.4 反面教材分析

**文件**: `src/content/zh/m2/2.4-anti-patterns.mdx`

**核心知识点**:
- 需求表达的常见反模式（用真实的"差需求"示例）：
  1. **太模糊**："帮我做一个电商网站" → AI 不知道做多大、什么品类、什么风格
  2. **太贪心**："做一个和淘宝一样的网站" → 淘宝有几千个功能，AI 无法一次完成
  3. **只说结果不说场景**："要好看" → 什么叫好看？谁的审美标准？
  4. **没有约束**："随便做" → AI 会做出一个中庸的、什么特点都没有的东西
  5. **跳过需求直接要代码**："用 React + Node.js + MongoDB 做一个..." → 在需求不清的时候就定技术栈
- 对比修改：每个反面案例对应一个改进版本
- 核心观点：好的需求描述 = 明确的用户 + 明确的场景 + 明确的 P0 功能 + 明确的约束

**写作要点**:
- 用"差 vs 好"的 before/after 对比格式，视觉效果清晰
- 语气可以轻松幽默（"如果你对 AI 说'随便做'，AI 也会'随便做'"）
- 结尾总结一个检查清单：发给 AI 之前，检查你的需求是否包含了这些要素

**练习**:

```
<Exercise id="ex-2.4.1" title="改写差需求">
以下三个需求描述都有问题，请改写为更好的版本：

1. "帮我做一个博客"
2. "做一个好看的登录页面"
3. "我要一个和小红书差不多的 App"

每个改写后的版本应该包含：目标用户、核心功能（P0）、风格/约束描述。
</Exercise>
```

**参考来源**:
- datawhalechina/vibe-vibe 第 2.1 节（为什么 AI 写不出复杂的 App）
- 2025Emma/vibe-coding-cn 方法论（规划驱动原则）

## M3：提示词工程（5 节课）

### 3.1 Context is King

**文件**: `src/content/zh/m3/3.1-context-is-king.mdx`

**核心知识点**:
- 核心论点：同一个需求，不同的 prompt 可以让 AI 产出从"垃圾"到"惊艳"
- 什么是上下文（Context）：AI 做决策的所有输入信息——你说了什么、之前对话的历史、系统级指令
- 为什么上下文比指令重要：AI 不是"执行命令的机器"，而是"基于上下文做推理的模型"
- 对比实验：展示同一个任务（"做一个待办清单"），分别用简单 prompt 和富上下文 prompt 的产出差异
- 上下文的三个层次：任务级（这次对话的内容）→ 项目级（CLAUDE.md 等配置）→ 系统级（AI 的内建知识）

**写作要点**:
- 开头用一个震撼对比：差 prompt vs 好 prompt 的真实产出截图/代码对比
- 不要抽象讲理论，全部用具体例子说明
- 引出后续课节的预告：3.2 讲提示词结构，3.3 讲系统提示词

**练习**:

```

### 3.2 提示词结构：四要素框架

**文件**: `src/content/zh/m3/3.2-prompt-structure.mdx`

**核心知识点**:
- 高质量 prompt 的四要素：
  1. **角色设定**（Role）："你是一个有 5 年经验的前端工程师"——让 AI 进入正确的知识域
  2. **任务描述**（Task）：具体要做什么，产出什么——越具体越好
  3. **约束条件**（Constraints）：技术栈、风格、不能做什么——画框框
  4. **输出格式**（Format）：要代码还是文档？分几个文件？用什么结构？
- 四要素不是每次都必须全写，但越复杂的任务越需要完整
- 每个要素的模板句式和示例
- 常见错误：角色设定太空洞（"你是专家"没有意义）、约束条件和任务矛盾

**写作要点**:
- 用一个任务（"做一个番茄钟"）从零到完整地构建一个四要素 prompt
- 每添加一个要素，展示 AI 产出的变化
- 给出一个可复制的 prompt 模板

**练习**:

```
<Exercise id="ex-3.2.1" title="用四要素改写 Prompt">
把以下三个简单 prompt 改写为包含四要素（角色/任务/约束/格式）的完整版本：

1. "做一个天气查询页面"
2. "帮我写一个 API"
3. "做一个漂亮的 404 页面"

每个改写后的 prompt 应该不少于 5 行。
</Exercise>
```

**参考来源**:
- 2025Emma/vibe-coding-cn 提示词构建原则
- datawhalechina/vibe-vibe 第 3.1 节

### 3.3 系统提示词：CLAUDE.md 的设计逻辑

**文件**: `src/content/zh/m3/3.3-system-prompts.mdx`

- 什么是系统提示词：不是每次对话时写的，而是"预设的背景规则"，相当于 AI 的"工作手册"

  - 项目概述（一句话说清做什么）
  - 技术栈声明
  - 代码规范与约束
  - 文件结构约定
  - 常用命令
- 为什么它重要：让 AI 在整个项目中保持一致的行为，而不是每次对话都重新教一遍
- 实例：展示一个真实的 CLAUDE.md 片段

### 3.4 Prompt 模板与提示词库

**文件**: `src/content/zh/m3/3.4-prompt-templates.mdx`

**核心知识点**:
- Prompt 模板的价值：不是每次都从零写 prompt，好的模板可以复用和改进
- 常用模板分类：
  - **需求澄清模板**："我要做 XX，请帮我确认以下问题：目标用户是谁？核心功能有哪些？技术约束是什么？"
  - **代码生成模板**："作为 [角色]，请按照以下要求生成代码：[任务] [约束] [格式]"
  - **Debug 模板**："我遇到了以下错误：[错误信息]。上下文：[我做了什么操作]。请帮我分析原因并给出修复方案。"
  - **代码审查模板**："请审查以下代码，从 [安全性/性能/可读性] 角度给出改进建议。"
- 提示词库的建设方法：每次写出好的 prompt，保存到个人库中，标注场景和效果
- 介绍平台的 Prompt Portfolio 功能

**写作要点**:
- 给出 4-5 个可直接复制使用的 prompt 模板
- 强调"收集好 prompt"是一种持续积累的习惯
- 介绍 vibe-coding-cn 的提示词库作为参考资源

**练习**:

```

从你之前的学习过程中，整理出至少 3 个你用过的最好的 prompt。

每个 prompt 记录：
1. 使用场景（做什么任务时用的）
2. 完整 prompt 内容
3. 效果如何（AI 产出质量打分 1-5）
4. 改进空间（下次可以怎么改进）

整理好后提交到你的 Prompt Portfolio。
</Exercise>

### 3.5 常见反模式

**文件**: `src/content/zh/m3/3.5-anti-patterns.mdx`

**核心知识点**:
- 五个最常见的提示词反模式：
  1. **需求模糊**：说了"做一个页面"但没说做什么内容的页面 → 解决方案：灵魂三问 + 四要素
  2. **一次给太多**：一个 prompt 里塞了 10 个功能需求 → 解决方案：P0 先行，分步提需求
  3. **缺少约束**：没有告诉 AI 技术栈、风格、不能做什么 → 解决方案：在约束条件中明确
  4. **上下文污染**：对话太长，前面的错误信息影响后续产出 → 解决方案：及时开新对话，或明确告诉 AI 忽略之前的错误
  5. **幻觉引导**：AI 编造了不存在的 API 或功能，你没察觉就继续用了 → 解决方案：对关键信息保持怀疑，验证 AI 给出的 API/库/函数是否真实存在
- 每个反模式给出：症状 → 根因 → 改进方案 → 示例

**写作要点**:
- 用"你是不是经常遇到这种情况？"的方式引入每个反模式
- 读者看到反模式后应该有"啊对，我就是这样"的共鸣感
- 结尾给一个"发送 prompt 前的检查清单"

**练习**:

```
<Exercise id="ex-3.5.1" title="识别反模式">
以下 5 个 prompt 各有什么问题？分别属于哪种反模式？请给出改进版本。

1. "做一个网站"
2. "用 React + TypeScript + Tailwind + Prisma + PostgreSQL + NextAuth + Stripe 做一个 SaaS 平台，包含用户注册登录、订阅支付、仪表盘、数据分析、团队管理、权限系统、邮件通知、API 接口"
3. "做一个好看的页面，用最新的技术"
4. （在一个已经讨论了 50 轮的对话中）"继续做下一个功能"
5. "用 React 的 useServerQuery hook 来获取数据"（注：这个 hook 不存在）
</Exercise>
```

**参考来源**:
- 2025Emma/vibe-coding-cn 开发经验总结
- datawhalechina/vibe-vibe 第 4.5 节（Debug 指南中的反模式）

### 4.1 为什么不写 Plan 必翻车

**文件**: `src/content/zh/m4/4.1-why-plan-matters.mdx`

**核心知识点**:
- 展示一个真实案例：一个没有规划就开始 vibe coding 的项目，中途失控的全过程
- "AI Slop"的概念：看起来能跑但实际上架构混乱、安全漏洞百出、无法维护的代码
- 认知负债（Cognitive Debt）：AI 产出的代码你不理解，堆积到一定程度就完全失控
- 核心观点引用（重新表述）：规划就是一切。不谨慎规划就让 AI 自主开发，代码库会变成无法管理的混乱
- 规划不是浪费时间，是节省时间：5 分钟的规划可以避免 2 小时的返工

**练习**:

```
<Exercise id="ex-4.1.1" title="分析翻车案例">
以下是一段 vibe coding 对话记录的摘要：

"第 1 轮：做一个博客系统 → AI 生成了完整的博客
第 5 轮：加一个评论功能 → AI 重写了一半代码
第 8 轮：评论功能有 bug → AI 修复时破坏了文章列表
第 12 轮：加用户登录 → AI 引入了新框架，和之前的代码不兼容
第 15 轮：整个项目无法运行"

请分析：这个过程中犯了哪些错误？如果重来一次，应该怎么规划？
</Exercise>
```

### 4.2 PRD 编写方法

**文件**: `src/content/zh/m4/4.2-prd-writing.mdx`

**核心知识点**:
- PRD（Product Requirements Document）不需要很正式，但需要结构化
- 适合 vibe coding 的轻量 PRD 模板：
  1. 项目概述（做什么、为谁做、解决什么问题）
  2. 功能列表（P0/P1/P2 分级）
  3. 技术约束（用什么技术栈、有什么限制）
  4. 非功能需求（性能、安全、兼容性）
  5. 设计约束（风格、配色、参考网站）
- 怎么写出 AI 能"读懂"的 PRD：结构化、避免歧义、用列表而非长段落

3. **P0 功能**：必须有的功能（3-4 个）
4. **P1 功能**：应该有的功能（2-3 个）
5. **技术约束**：用什么技术

### 4.3 Implementation Plan：分步拆解

**文件**: `src/content/zh/m4/4.3-implementation-plan.mdx`

**核心知识点**:
- PRD 说的是"要做什么"，Implementation Plan 说的是"按什么顺序做"
- 为什么要分步：AI 一次处理的信息量有限，分步可以保证每步的产出质量
- Implementation Plan 模板：
  - 每步包含：任务描述、预期产出、验证条件
  - 步骤粒度控制：每步 15-30 分钟可完成为宜
  - 步骤之间有明确的依赖关系
- 关键纪律："在我验证测试前不要开始下一步"
- 实例：把番茄钟 PRD 拆成 6 步 implementation plan

**练习**:

```
<Exercise id="ex-4.3.1" title="拆解实施计划">
基于你在 4.2 写的番茄钟 PRD，把它拆成 5-8 步的实施计划。

每步包含：
- **任务**：这一步要做什么（1-2 句话）
- **产出**：完成后应该看到什么
- **验证**：怎么确认这一步完成了

示例：
Step 1: 创建基础页面布局
- 产出：一个空白页面，有标题栏和居中的计时器占位区域
- 验证：打开页面能看到布局，响应式正常
</Exercise>

### 4.4 PEV 循环

**文件**: `src/content/zh/m4/4.4-pev-loop.mdx`

**核心知识点**:
- PEV 循环：Plan → Execute → Verify，是 vibe coding 工程化的核心工作流
- **Plan**：明确这一步要做什么，写清楚给 AI 的指令
- **Execute**：让 AI 执行，产出代码/文档
- **Verify**：你来验证产出是否符合预期，而不是盲目接受
- PEV 的关键：Verify 不是"看一眼就行"，要实际运行、测试边界情况
- 嵌套 PEV：大任务拆成小任务，每个小任务内部也是 PEV
- 常见失败模式：跳过 Plan 直接 Execute、跳过 Verify 直接下一步

**练习**:

```
<Exercise id="ex-4.4.1" title="实践一次 PEV 循环">
选择你的番茄钟实施计划中的第一步，严格按照 PEV 循环执行：

1. **Plan**：写出你要给 AI 的完整 prompt（包含四要素）
2. **Execute**：把 prompt 发给 AI，获得产出
3. **Verify**：检查产出是否符合预期。记录：
   - 符合预期的部分
   - 不符合预期的部分
   - 需要修改什么

提交你的 Plan prompt、AI 的产出截图/代码、以及你的 Verify 记录。
</Exercise>
```

### 4.5 Ask/Plan 模式

**文件**: `src/content/zh/m4/4.5-ask-plan-mode.mdx`

**核心知识点**:
- "先确认再执行"的纪律：在让 AI 写代码之前，先让它告诉你它打算怎么做
- Ask 模式：问 AI "如果我让你做 XX，你会怎么做？"，AI 描述方案但不执行
- Plan 模式（Claude Code 中按 shift+tab）：AI 输出实施计划，你确认后它再执行
- 为什么这很重要：
  - 避免 AI 用你不想要的方式实现
  - 提前发现理解偏差
  - 你作为"审批者"保持对项目的控制
- 什么时候用 Ask vs 什么时候直接 Execute：
  - 复杂/关键功能 → Ask first
  - 简单/低风险修改 → 直接 Execute

**练习**:

```
<Exercise id="ex-4.5.1" title="练习 Ask 模式">
在你使用的 AI 工具中，用 Ask 模式确认一个任务的实施方案：

Prompt："我想为番茄钟添加一个'历史记录'功能，记录每次完成的番茄钟。
在你开始写代码之前，请先告诉我：
1. 你打算用什么方式存储数据？
2. UI 上会添加什么元素？
3. 需要修改哪些现有代码？
请只描述方案，不要写代码。"

记录 AI 的回答，评估方案是否合理。如果不合理，提出修改意见再让它重新规划。
</Exercise>
```

### 4.6 Claude Code 上手

**文件**: `src/content/zh/m4/4.6-claude-code-intro.mdx`

**核心知识点**:
- Claude Code 是什么：终端中运行的 AI 编程助手，直接在你的项目目录中工作
- 安装和配置：npm install -g @anthropic-ai/claude-code（或 npm 安装方式），需要 Anthropic API key 或 Claude Pro/Max 订阅
- 基本使用：
  - 在项目目录中运行 `claude` 进入交互模式
  - 直接描述任务，Claude Code 会读取项目文件并修改
  - Plan 模式（shift+tab）：先看方案再执行
  - Ask 模式：只问不做
- 核心优势：理解你的整个项目上下文（读取所有文件），不需要你复制粘贴代码
- 第一个任务：用 Claude Code 为你的番茄钟项目做一个小改动
- 对比体验：同一个任务，用 GUI 工具和 Claude Code 分别做，感受区别
- 注意事项：Claude Code 消耗 API 额度，注意用量

**写作要点**:
- 这是一个实操课，重点是"跟着做"
- 安装步骤要详细，考虑到 Windows/Mac/Linux 差异
- 用截图或代码块展示终端中的交互过程
