<Callout title="本单元方法论焦点" tone="info">
工具选择本身就是规划。你用什么工具，取决于你这轮要验证什么，而不是单纯追最新。
</Callout>

## Step 1: 浏览器工具适合什么
Bolt.new、Lovable、Replit 这类浏览器工具最适合做“先把东西做出来”的阶段。它们启动快、反馈快、部署快，适合个人主页、着陆页、小型 demo 和第一版 MVP。它们不要求你先理解项目目录，也不要求你先掌握本地环境。

但浏览器工具也有边界。一旦需求变成复杂重构、跨文件协作、严格约束或细致验证，你会越来越需要一个能读完整仓库上下文的环境。换句话说，工具不是越强越好，而是越贴合任务越好。

| 工具 | 更适合的场景 |
|---|---|
| Bolt.new | 首版页面、快速原型 |
| Lovable | 个人项目、着陆页 |
| Replit | 小型 demo、在线试验 |

## Step 2: IDE 工具与 CLI 工具的区别
Cursor、Windsurf 这类 IDE 工具适合你已经开始面对真实代码库，但仍希望保留可视化编辑体验的阶段。Claude Code、Codex 这类 CLI 工具则更接近工程化协作: 它们直接工作在仓库和终端里，更强调约束、计划、验证和多步执行。

可以把三层工具理解成一个升级路径。浏览器工具解决“敢开始”；IDE 工具解决“能持续写”；CLI 工具解决“能在真实项目里稳定交付”。没有哪一层天然更高级，只有当前任务值不值得这层复杂度。

| 任务 | 浏览器工具 | IDE 工具 | CLI 工具 |
|---|---|---|---|
| 做首版主页 | 适合 | 也可以 | 不必一开始就上 |
| 改真实仓库 | 不太适合 | 适合 | 更适合 |
| 做多步验证 | 不稳定 | 可以 | 最稳 |

## Step 3: 你的第一把武器怎么选
起步工具选择可以用三个问题判断。第一，我现在是验证想法，还是修改真实项目？第二，我是否需要 AI 理解整个代码库？第三，我有没有精力同时管理环境、依赖和验证流程？如果前两个答案偏简单，先上浏览器工具；如果任务已接近工程协作，再进入 IDE 或 CLI。

最常见的错误是“还在验证想法，却一开始就把自己推进复杂终端工作流”，或者“已经在改真实仓库，却还想靠浏览器工具硬撑”。正确做法不是炫工具，而是让工具复杂度刚好匹配任务复杂度。

可以把这个判断记成一张小卡片:

| 现在的状态 | 建议 |
|---|---|
| 只想先试试 | 浏览器工具 |
| 已经在真实仓库里改动 | IDE 或 CLI |
| 要看计划、要验证、要交付 | CLI 优先 |

## 本单元小结

- 浏览器工具适合低门槛起步和快速验证。
- IDE 工具适合进入真实代码协作但仍需界面辅助。
- CLI 工具适合项目级上下文、计划与验证要求更高的任务。

## 方法论回顾

工具选择也是规划驱动的一部分。你先选对战场，再决定怎么打。
