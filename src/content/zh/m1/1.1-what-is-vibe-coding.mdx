## Step 1: Vibe Coding 的起源和定义
Vibe Coding 不是某一个工具，也不是一句固定 prompt，而是一种以自然语言驱动 AI 快速产出原型、再根据结果持续修正的工作方式。它的关键不在“写得像不像命令”，而在“你是否先把目标说清楚，再让 AI 帮你往前推进”。

可以把它理解成从“先写每一行代码”转向“先说清楚要去哪里”。AI 现在能读更长的上下文，能理解页面、流程和文档，也能根据反馈继续调整，所以学习 Vibe Coding，本质上是在学习如何和会产出的系统协作。

对比一下会更直观：

| 说法 | 结果倾向 |
|---|---|
| “帮我写个网页” | AI 需要大量猜测 |
| “帮我做一个面向新手的课程首页，先出一个能看懂的首版” | AI 更容易进入协作节奏 |

## Step 2: Vibe / Spec / Agentic 三种模式
三种模式对应三种复杂度。Vibe 适合快速试错和做原型，Spec 适合需求更明确、需要先写 proposal 或 design 的任务，Agentic 则适合把任务交给多个角色或自动化流程并行推进。它们不是互相替代，而是同一条能力链上的不同层级。

一个好用的判断方式是: 如果你还在探索“做什么”，先用 Vibe；如果你已经知道“做什么”，但怕返工，就切到 Spec；如果任务已经复杂到需要分工、审查和追踪，就开始考虑 Agentic。这个判断会在后面的模块里反复使用。

| 场景 | 更合适的模式 |
|---|---|
| 做个人主页首版 | Vibe |
| 为学习平台写 PRD | Spec |
| 多代理并行写文档并审查 | Agentic |

## Step 3: 为什么 2026 年是最好的学习时机
现在学 Vibe Coding 的窗口很特殊。工具已经成熟到足以承担真实任务，但方法论还没有完全固化，所以最容易掌握的人，往往不是最早接触的人，而是最先建立正确习惯的人。你越早学会“先规划、再执行、再验证”，后面的 Prompt Engineering、Claude Code 和多代理协作都会轻松很多。

课程最后会要求你主动去搜索一条最近的 vibe coding 新闻，再用“用户、痛点、方案”这三个问题分析它。这样做不是为了追热点，而是为了让你看见: 这套方法不是抽象理论，而是正在快速变化的行业现场。

比如同样是“AI 编程热”，有人只看工具热度，有人会继续追问: 这波热潮到底在帮谁省时间，谁的痛点被放大了，为什么这类方案现在能成立。后者才是课程要求你的分析方式。

## 本单元小结

- Vibe Coding 是一种自然语言驱动的协作方式，不是单一工具。
- Vibe、Spec、Agentic 对应不同复杂度的任务。
- 现在学的价值，不只在工具成熟，更在方法论还没完全固定。

## 方法论回顾

本单元把“规划驱动”作为全课程入口: 你不是一上来就问 AI 做什么，而是先判断这件事该用什么协作模式。
