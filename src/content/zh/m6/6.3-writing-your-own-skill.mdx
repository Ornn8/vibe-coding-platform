<Callout title="本单元方法论焦点" tone="success">
写 Skill 不是炫技，而是把你已经验证过的方法压缩成 AI 下次还能稳定执行的流程。
</Callout>

## Step 1: 先用一个具体主题，而不是空谈模板
我们直接用 `Tailwind CSS` 作为主题。假设你已经发现自己反复需要告诉 AI：

| 重复说明 | 为什么适合做 Skill |
|---|---|
| 优先用现有类名模式 | 是稳定规则 |
| 不随意写大段自定义 CSS | 是明确边界 |
| 修改后要检查移动端 | 是固定验证动作 |

这就说明，问题已经不再是“这次怎么说”，而是“这类任务怎么固定下来”。

<Separator />

## Step 2: 从最小可用骨架开始写 `SKILL.md`
下面是一份 Tailwind Skill 的最小骨架：

```md
# Tailwind Layout Editing

## Trigger
- When editing existing Tailwind-based UI

## Boundaries
- Reuse current utility style first
- Avoid introducing custom CSS unless necessary
- Keep mobile-first responsive behavior

## Workflow
1. Read the current component structure
2. Identify layout or spacing issue
3. Prefer utility-class edits
4. Check mobile and desktop behavior

## Example
- Fix uneven card spacing without rewriting the component tree
```

这还不完美，但已经足够让 AI 知道：什么时候触发、什么不能乱做、按什么顺序工作。

## Step 3: 再把骨架补成真正能执行的说明书
真正可用的 Skill 往往还要补两类内容：

| 要补的内容 | 作用 |
|---|---|
| 触发例子 | 帮 AI 更稳定识别场景 |
| 明确禁止项 | 防止它自由发挥到别的层面 |

```text
Before：编辑 UI 时尽量用 Tailwind。
After：当任务涉及现有 Tailwind 组件的布局、间距、响应式问题时触发本 Skill；不要新增自定义 CSS 文件，不要改组件结构除非明确要求。
```

<Callout title="场景卡片：为什么禁止项不能省" tone="warning">
情境：AI 明明只该调类名，却顺手改了组件结构或写了一堆 CSS。做法：把禁止项明确写进 Skill。原因：没有禁止项，AI 会把‘帮忙’理解成‘我可以顺手多做一点’。
</Callout>

## Step 4: Skill 必须放进真实任务里测试
Skill 和代码一样，不测试就不知道它是不是真的好用。

| 测试问题 | 你要观察什么 |
|---|---|
| AI 是否更少偏航 | 是否还会乱改不该动的部分 |
| AI 是否更少需要重复解释 | 你有没有少说很多老规则 |
| 输出是否更稳定 | 结果是否更接近你想要的工作流 |

<Exercise id="ex-6.3.4" level="L3" title="用真实任务测试你的 Skill" description="把你写好的 Skill 用在一个真实任务里，记录它让 AI 少犯了什么错、还缺什么约束，并据此迭代一次。" />

## 本单元小结
- Skill 最适合承接重复出现的稳定任务规则。
- 最小 `SKILL.md` 至少要有 Trigger、Boundaries、Workflow、Example。
- 不测试的 Skill 只是文档，测试过并迭代过的 Skill 才是资产。

## 方法论回顾
胶水编程在这里变成‘把经验产品化’：不是重复讲规则，而是把规则做成可反复调用的能力包。
