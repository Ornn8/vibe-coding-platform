<Callout title="本单元方法论焦点" tone="success">
MCP 不是一堆工具名，而是让 AI 从‘只能聊天’升级成‘能连接真实外部系统’的能力层。
</Callout>

## Step 1: 把 MCP 想成 AI 获得了一项新能力
想象两个 AI：

| 状态 | AI 能做什么 |
|---|---|
| 没有 MCP | 只能根据聊天和已有上下文回答 |
| 接入 GitHub MCP 后 | 能直接读取仓库、Issue、PR 等真实内容 |

这就是 MCP 最直观的价值。它不是让 AI “更会说”，而是让 AI 真正多了一双手，能碰到仓库、文档、数据库或其他系统。

```text
Before：AI 只能猜你仓库里有什么。
After：AI 能直接去读仓库、Issue 和 PR，再基于真实内容行动。
```

<Exercise id="ex-6.4.1" level="L1" title="用自己的话解释 MCP 让 AI 多了什么能力" description="不要用协议定义，直接用‘没有 MCP / 有 MCP’的对比方式说明 MCP 让 AI 多了什么。" />

<Separator />

## Step 2: 连接上了，不代表会用得稳
即使 AI 能连到 GitHub、数据库或文档系统，也不代表它知道怎么安全、正确地使用它们。

| 只有 MCP | MCP + Skill |
|---|---|
| AI 能连上系统 | AI 知道什么时候连、怎么连、连完后要按什么流程操作 |
| 可能会乱问、乱用 | 更容易遵守边界和步骤 |

所以最稳的组合不是“只有 MCP”，而是“MCP 提供能力，Skill 规定流程”。

<Callout title="场景卡片：为什么只连上还不够" tone="warning">
情境：AI 已经能访问仓库，但每次读取顺序和使用方式都不稳定。做法：给它配一份说明什么时候读、先读什么、读完怎么输出的 Skill。原因：能力没有流程，就很难稳定复用。
</Callout>

<Separator />

## Step 3: 真正有价值的是完整链路，而不是单次调用
一个有用的 MCP 工作流通常长这样：

| 环节 | 问题 |
|---|---|
| 触发 | 什么时候要调用这个 MCP |
| 访问 | AI 去读或写什么 |
| 处理 | 读取后怎么组织结果 |
| 验证 | 谁来确认结果是对的 |

如果只关注“能不能连”，你往往会忽略“连完之后怎么用才稳”。真正的胶水编程价值，就在这条完整链路上。

<Exercise id="ex-6.4.3" level="L3" title="设计一个真实的 MCP + Skill 工作流" description="围绕一个真实系统，设计从触发、访问、处理到验证的完整工作流，并说明 MCP 和 Skill 在其中各自负责什么。" />

## 本单元小结
- MCP 让 AI 获得连接外部系统的新能力。
- 只有能力不够，还要有 Skill 规定使用方式。
- 真正稳定的不是‘一次连上’，而是‘形成可反复执行的完整链路’。

## 方法论回顾
胶水编程在这里进入双层结构：连接能力由 MCP 负责，稳定流程由 Skill 负责。
