<Callout title="本单元方法论焦点" tone="warning">
这是你第一次完整实践“闭环交付”: 需求文档不是配套材料，而是 MVP 的施工蓝图。
</Callout>

## Step 1: 先做一份能给 AI 看的需求文档
一份够用的需求文档不需要很长，但必须包含项目背景、目标用户、核心痛点、核心流程、P0/P1/P2 边界和验收标准。只要这些要素完整，你就已经具备了把想法转成首版产品的最低条件。

这一步的关键不是“写得像产品经理”，而是写得足够清晰，让你自己和 AI 都不会在第二天忘记为什么要这样做。

你可以把模板理解成下面这种结构：

| 部分 | 作用 |
|---|---|
| 背景 | 说明为什么要做 |
| 用户 | 说明给谁做 |
| 流程 | 说明怎么用 |
| 范围 | 说明做什么、不做什么 |

## Step 2: 用需求文档驱动 MVP
现在再回到 GUI 工具，你会发现自己比在 M1 时更能控制结果。因为你已经有了明确的用户、流程和范围。第一轮 prompt 可以直接基于需求文档摘要生成，而不是临场想到什么说什么。

执行时请坚持一件事: 第一轮只做 P0。如果工具开始自动扩展出很多额外功能，及时拉回边界。

比如:

```text
先只做登录后的记账主流程，不要加分享、统计和多主题切换。
```

## Step 3: 提交结果与过程
本模块的交付不仅是一个 MVP 链接，还包括你的需求文档和 prompt 日志。因为我们要审查的不只是“做出来没有”，还要看你是否真的通过方法降低了混乱。你可以在复盘里说明: 哪一步最有帮助？哪里仍然容易跑偏？下一轮会补什么约束？

这份复盘会直接为 M3 的 Prompt Engineering 做准备。

交付时最好至少包含这三项:

| 交付物 | 说明 |
|---|---|
| 需求文档 | 记录你做什么 |
| MVP 链接/截图 | 记录你做成什么 |
| prompt 日志 | 记录你怎么做的 |

## 本单元小结

- 需求文档让 AI 协作从“拍脑袋”进入“有蓝图”。
- MVP 执行时要坚守 P0 边界。
- 结果和过程必须一起提交，才能真正复盘。
