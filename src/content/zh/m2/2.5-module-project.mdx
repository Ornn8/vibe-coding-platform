<Callout type="methodology" title="本单元方法论焦点">
需求文档不是配套材料，而是 MVP 的施工蓝图。先写清，再开始生成。
</Callout>

## Step 1: 先做一份 AI 也能看懂的需求文档

M2 的需求文档不追求长，而追求完整。最少要包含：

- 项目背景
- 目标用户
- 核心痛点
- 核心流程
- P0 / P1 / P2
- 验收标准

| 部分 | 你要解决什么问题 |
|---|---|
| 背景 | 为什么现在要做这个项目 |
| 用户 | 给谁做 |
| 痛点 | 他现在在忍受什么低效 |
| 流程 | 从打开到完成会发生什么 |
| 范围 | 本轮做什么、不做什么 |
| 验收 | 什么结果算首版合格 |

<Callout type="scenario" title="场景卡片：一页需求文档的正确长度">
情境：你担心文档写太短不专业，写太长又没人看。

做法：先写一页内能讲清的版本。

原因：这份文档的目标不是展示写作能力，而是成为接下来 AI 执行的稳定输入。
</Callout>

## Step 2: 用需求文档驱动首版 MVP

现在再打开 GUI 工具，你的第一轮 prompt 不应该从空白开始，而应该从需求文档摘要开始。

```text
请根据以下需求做一个首版 MVP：
目标用户：第一次参加读书会的新用户。
核心流程：看到活动价值 -> 查看流程和时间地点 -> 完成报名。
本轮只做：活动介绍、流程说明、报名表。
本轮不做：支付、评论、个人中心。
验收标准：用户能在 1 分钟内理解活动并完成报名。
```

<Callout type="warning" title="before / after：临场发挥 vs 文档驱动">
错误做法：打开工具后想到什么说什么。

正确做法：从需求文档提炼一段首版摘要，直接作为 prompt 起点。

差异：前者每一轮都可能换方向，后者每一轮都有基线可回看。
</Callout>

| 执行方式 | AI 第一轮更可能出现什么 |
|---|---|
| 临场发挥 | 功能和结构容易漂 |
| 文档驱动 | 更容易贴着需求边界产出 |

## Step 3: 提交结果，也提交过程

本模块要提交的不只是一个链接，而是一个完整项目包：

- 需求文档
- MVP 链接或截图
- 首轮 prompt
- 关键迭代记录
- 一段复盘

<Callout type="scenario" title="场景卡片：为什么过程证据同样重要">
情境：两个项目都做出了页面，但一个人能解释怎么控制范围，另一个不能。

做法：把需求文档、prompt 记录和迭代证据一起保留下来。

原因：课程要训练的是“可复盘的交付能力”，不是只看最后有没有页面。
</Callout>

| 交付物 | 最低要求 |
|---|---|
| 需求文档 | 一页内可读，包含边界和验收标准 |
| MVP 结果 | 链接或截图能展示核心流程 |
| prompt 记录 | 首轮 prompt + 至少一轮边界修正 |
| 复盘 | 哪一步最有帮助，哪一步仍然会跑偏 |

## 本单元小结

- 一页需求文档已经足够驱动首版 MVP。
- 文档驱动的 prompt 比临场发挥稳定得多。
- 结果和过程必须一起提交，才能真正复盘。

**方法论回顾**：本单元把规划驱动和闭环交付第一次连成完整路径：需求文档 -> MVP -> 过程证据。
