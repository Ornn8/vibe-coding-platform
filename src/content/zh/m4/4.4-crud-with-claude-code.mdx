## Step 1: 开工前先交齐文档
在真正动手写 CRUD 前，至少准备三样东西: PRD、Implementation Plan、CLAUDE.md。它们分别解决目标、步骤和长期约束问题。没有这些文档，Claude Code 就只能根据临时对话做短期决策。

很多返工都源自“边做边想边补”，而这一步的价值正是把思考往前提。

可以把它理解为三份不同的说明书:

| 文档 | 回答的问题 |
|---|---|
| PRD | 要做什么 |
| Plan | 怎么做 |
| CLAUDE.md | 长期遵守什么 |

## Step 2: 分步执行，不要跨多步跳跃
一个 CRUD 项目看起来不复杂，但照样应该按计划逐步推进: 先搭基础结构，再做数据模型，再做列表与详情，再补创建和编辑，最后做删除和验证。每一步完成后都要回到 Verify，确保没有把问题带到下一步。

如果你试图让 AI 一口气做完整项目，它通常会给你一个表面完整、内部松散的结果。

比如:

| 步骤 | 验证 |
|---|---|
| 建表 | 能否正常迁移 |
| 列表页 | 能否正确显示数据 |
| 创建页 | 能否成功提交 |

## Step 3: 继续推进更复杂的步骤
前几步完成后，后面的修改往往会涉及更多文件、更复杂的状态和更细的交互。这个时候最重要的不是速度，而是持续回到 PEV: 先看方案，再执行，再验证。复杂度一旦上来，跳步就会变成返工。

后续步骤常见的差异是: 前几步只动单点，后几步会同时碰页面、数据和验证逻辑，所以更不能省略验证。

## Step 4: 用 memory-bank 记录进展
项目推进到第三步以后，人和 AI 都容易忘记前面已经决定过什么。这时就需要用 progress.md 或 memory-bank 记录当前状态、完成项、待办项和问题。它让协作有了长期记忆，也让交接和复盘成为可能。

你可以把它理解为工作流的外部记忆系统。

一个很实用的记录格式是:

| 字段 | 内容 |
|---|---|
| 已完成 | 哪些步骤已结束 |
| 当前状态 | 现在卡在哪 |
| 下一步 | 下一轮要做什么 |

## 本单元小结

- 完整项目执行前必须交齐关键文档。
- CRUD 也要分步做，每一步都要验证。
- memory-bank 让项目协作具备长期记忆。
