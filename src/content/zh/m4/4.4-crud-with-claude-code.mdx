<Callout title="本单元方法论焦点" tone="warning">
这不是“让 AI 一次做完 CRUD”，而是把四大原则都压进同一个真实项目闭环里。
</Callout>

## Step 1: 先交齐开工文档，再做数据模型
开工前至少准备：PRD、Implementation Plan、`CLAUDE.md`。它们分别在回答：

| 文档 | 回答什么 |
|---|---|
| PRD | 要做什么 |
| Plan | 先做哪一步 |
| `CLAUDE.md` | 长期规则是什么 |

然后先做**数据模型**，而不是一上来就做页面。因为 CRUD 的核心骨架在数据层。

| 阶段 | 本轮目标 | 验证方式 |
|---|---|---|
| 数据模型 | 定字段、关系、最小存储结构 | schema 合理、迁移可跑 |

<Separator />

## Step 2: 第二段只做后端 API，不急着碰页面
数据模型定好后，再做**后端 API**。比如一个笔记 CRUD 项目，这一段只解决：能不能创建、读取、更新、删除笔记。

| 阶段 | 本轮目标 | 验证方式 |
|---|---|---|
| 后端 API | 先让增删改查通路跑通 | 用请求或日志确认返回正确 |

```text
Before：把 CRUD 应用一次做完。
After：先定数据模型，再跑通 API，再做页面，再补交互。
```

这一步的重点是把“数据能通”这件事单独验证掉，不让页面问题和接口问题搅在一起。

<Exercise id="ex-4.4.2.1" level="L2" title="为你的 CRUD 写后端阶段计划" description="围绕一个真实 CRUD 项目，写出数据模型阶段和后端 API 阶段各自要完成什么、怎么验证。" />

<Separator />

## Step 3: 第三段做前端页面，第四段补交互和细节
当数据层和 API 都稳了，再做**前端页面**和**交互补全**。

| 阶段 | 本轮目标 | 验证方式 |
|---|---|---|
| 前端页面 | 列表页、详情页、表单页先出现 | 页面能展示真实数据 |
| 交互补全 | loading、错误提示、确认删除、表单校验 | 真实操作时反馈正确 |

这就是为什么内容计划里要求把 CRUD 明确拆成四段。因为越往后，页面、数据和交互会同时碰，跳步就更容易返工。

<Callout title="场景卡片：为什么先不要沉迷‘页面已经能看’" tone="warning">
情境：你看到列表页已经出来了，很想继续堆 UI。做法：先验证它展示的是不是正确数据，再决定下一步。原因：页面‘长得像对的’不等于数据流真的通了。
</Callout>

<Separator />

## Step 4: 用 memory-bank 或 progress.md 记录整个执行过程
项目推进到第三步以后，人和 AI 都会开始忘记已经决定过什么。这时就要把状态写进 `progress.md` 或 memory-bank。

| 字段 | 你应该写什么 |
|---|---|
| 已完成 | 数据模型、API、页面、交互中哪些已结束 |
| 已验证 | 跑过哪些命令、做过哪些手动检查 |
| 当前状态 | 卡在哪里 |
| 下一步 | 下一轮准备推进什么 |

```text
已完成：笔记模型、列表接口、创建接口
已验证：接口返回正确，列表页能展示真实数据
当前状态：编辑表单还没接上
下一步：补编辑接口和表单提交验证
```

<Callout title="before / after：没有进度记录 vs 有 progress 记录" tone="info">
错误做法：每次都靠聊天记录回忆做到哪。正确做法：用 progress.md 记已完成、已验证和下一步。差别在于，前者很快失去上下文，后者才有长期协作记忆。
</Callout>

<Exercise id="ex-4.4.4.1" level="L3" title="真实记录一个 CRUD 执行日志" description="为你的 CRUD 项目写一段真实 `progress.md` 记录，至少写出数据模型、后端、前端、交互四段里已经完成的部分和下一步计划。" />

## 本单元小结

- CRUD 也必须分段做：数据模型、后端、前端、交互。
- 每一段都要单独验证，不要把问题带进下一段。
- `progress.md` 或 memory-bank 是项目级协作的外部记忆。
