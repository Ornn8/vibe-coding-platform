<Callout title="本单元方法论焦点" tone="success">
PRD 不是写给别人看的文档，Plan 也不是形式主义。它们是在替未来的执行和返工成本做前置决策。
</Callout>

## Step 1: 先用一个番茄钟案例感受 PRD 在解决什么
假设你要做一个“给学生用的番茄钟网页”。PRD 不是一上来写很多页，而是先把这 5 件事写清楚：

| PRD 部分 | 你为什么要写 | 番茄钟示例 |
|---|---|---|
| 背景 | 说明为什么值得做 | 学生想专注 25 分钟学习，但常被手机打断 |
| 用户 | 防止你服务“所有人” | 需要短时专注的学生 |
| 核心流程 | 决定页面和功能主线 | 开始计时 → 专注 → 提醒休息 → 查看完成次数 |
| 范围 | 防止首版乱长 | 首版只做单人计时和完成记录 |
| 验收标准 | 决定什么时候算完成 | 用户能在 10 秒内启动一次专注计时 |

PRD 真正解决的问题是：**这次到底要交付什么，不交付什么。**

<Callout title="场景卡片：为什么 PRD 先写用户和流程" tone="info">
情境：你一开始就想讨论技术选型。做法：先把用户、流程和范围写清。原因：如果目标还没定，后面的技术讨论只是在围着不确定目标打转。
</Callout>

## Step 2: 边填边理解一页 PRD
把番茄钟 PRD 写成一页时，你会发现每一项都在替未来省返工：

```text
项目：学生番茄钟网页
目标用户：需要短时专注学习的学生
核心流程：开始 25 分钟计时 → 结束提醒 → 记录一次完成
P0 范围：开始/暂停/重置计时、完成记录
不做项：登录、社交、排行榜、跨设备同步
验收标准：新用户 10 秒内能启动计时
```

```text
Before：我要做一个学习效率工具。
After：我要做一个给学生使用的番茄钟网页，首版只验证计时和完成记录，不做登录和社交。
```

这就是 PRD 的作用：把“好像想做一个东西”变成“这轮明确要做的一小块东西”。

## Step 3: Implementation Plan 在解决另一类问题
当 PRD 已经回答“做什么”，Plan 才来回答“怎么分步做”。还用番茄钟项目举例：

| Plan 步骤 | 为什么先做这个 | 怎么验证 |
|---|---|---|
| 先确定页面结构 | 让主流程先可见 | 页面能展示开始按钮和计时区 |
| 再做计时逻辑 | 核心能力先跑通 | 倒计时能正确开始和结束 |
| 再做完成记录 | 让核心闭环成立 | 完成一次后能看到记录增加 |
| 最后补交互打磨 | 避免太早陷入细节 | 按钮状态和提示文案正确 |

Plan 真正解决的问题是：**先做哪一步，怎样避免一步跨太大。**

<Callout title="before / after：只有 PRD 没有 Plan vs PRD + Plan" tone="warning">
错误做法：我已经知道要做番茄钟了，直接开始写。正确做法：先把页面结构、计时逻辑、完成记录、交互打磨拆成步骤。差别在于，前者容易一口气冲成大泥球，后者更容易逐步验证。
</Callout>

## Step 4: 胶水编程不是让 AI 发明一切
写 Plan 时还要做一件事：选择成熟模块，而不是默认从零造世界。

| 项目部分 | 更稳的做法 | 风险更高的做法 |
|---|---|---|
| UI | 复用成熟组件 | 全部手搓新组件 |
| 数据层 | 用现有 ORM / 存储方案 | 临时发明数据结构 |
| 验证 | 用已有 lint / typecheck | 完全靠肉眼判断 |

番茄钟项目很小，但原则一样：先选成熟零件，再让 AI 把它们拼起来，这才是胶水编程真正的起点。

## 本单元小结

- PRD 解决“做什么、不做什么、怎样算完成”。
- Plan 解决“先做哪一步、每一步怎么验证”。
- 规划同时也是选成熟模块、降低实现风险的过程。
