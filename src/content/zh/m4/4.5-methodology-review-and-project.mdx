<Callout title="本单元方法论焦点" tone="success">
方法论只有在你能说清‘我在哪一步用了它、它改变了什么结果、哪里还不够’时，才真正变成你的能力。
</Callout>

## Step 1: 用结构化框架回看四大原则
不要只写“我学到了很多”。更有效的回顾方式，是每条原则都回答三个问题：

| 原则 | 我在哪个项目里用到了 | 我做得好的地方 | 还不够的地方 |
|---|---|---|---|
| 规划驱动 | 番茄钟 / CRUD 项目 | 先写 PRD 再动手 | Plan 还不够细 |
| 上下文固化 | `CLAUDE.md`、规则文件 | 固定了技术边界 | 有些规则还太口头 |
| 胶水编程 | 复用成熟组件和框架 | 少走了很多弯路 | 有时还是想从零重做 |
| 闭环交付 | 每步都写验证和 progress | 返工减少 | Verify 还不够稳定 |

这张表的价值在于，它逼你把原则和真实行为对应起来，而不是停留在“听起来懂了”。

## Step 2: 模块项目要交的不只是结果，还有过程资产
M4 模块项目真正要交的是两层内容：

| 层级 | 交付物 | 作用 |
|---|---|---|
| 结果层 | 页面、代码、可运行项目 | 证明你做成了什么 |
| 过程层 | PRD、Plan、`CLAUDE.md`、`progress.md` | 证明你是怎么做成的 |

如果只有结果，没有过程资产，别人很难判断你是不是在用一套可复制的方法工作。

```text
Before：我只需要交一个能跑的结果。
After：我需要同时交结果和过程证据，让别人看见我的工作流。
```

## Step 3: 为 M5 之后的阶段做准备
M5 往后你会进入更偏执行、Debug、交付和协作的阶段。如果 M1-M4 没有打好底座，后面的复杂度只会把混乱放大；如果你已经习惯先写目标、沉淀规则、选成熟模块、保留验证记录，后面的升级会自然很多。

一个很实用的自检框架是：

| 自检问题 | 是 / 否 |
|---|---|
| 我是否会在开工前先写 PRD 或最小需求摘要 |  |
| 我是否会把长期规则写进系统文件而不是反复口头提醒 |  |
| 我是否会优先找成熟模块，而不是默认从零造一遍 |  |
| 我是否会保留 Verify 和 progress 记录 |  |

<Callout title="场景卡片：什么时候算你真的准备好进入下一阶段" tone="info">
情境：你已经能做出页面，但不确定自己是不是准备好做更复杂项目。做法：看你是否已经形成‘先计划、再执行、再验证、再记录’的节奏。原因：后面的复杂度不会奖励灵感型发挥，只会奖励稳定工作流。
</Callout>

## 本单元小结

- 四大原则要通过真实项目回看，不能只停留在抽象概念。
- 模块项目必须同时交结果层和过程层资产。
- M1-M4 的目标不是学会某个工具，而是建立一套稳定工作流。
