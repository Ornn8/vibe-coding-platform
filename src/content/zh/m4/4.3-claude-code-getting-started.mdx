<Callout title="本单元方法论焦点" tone="info">
CLI 工具不是‘更高级的聊天框’，而是更强调上下文、计划和验证的工作环境。
</Callout>

## Step 1: 按系统完成安装与第一次进入项目
先看三个系统的起步关注点：

| 系统 | 你通常会先做什么 | 最常见问题 |
|---|---|---|
| Windows | 打开 PowerShell / Terminal，确认 Node、npm、路径可用 | 路径、权限、命令不在 PATH |
| macOS | 打开 Terminal，确认 shell 可用、目录权限正常 | 终端权限、目录访问限制 |
| Linux | 检查 shell、Node 环境和项目权限 | 依赖缺失、权限不足 |

第一次进入项目后，不要立刻让它改功能。先让它读仓库、看 `AGENTS.md`、看 `CLAUDE.md`、看 README。第一次上手更像“先看图纸再施工”。

<Callout title="场景卡片：第一次打开 Claude Code 时该发什么" tone="success">
情境：你已经在项目目录里启动工具。做法：先让它分析仓库结构、关键约束文件和技术栈，不要先要求改代码。原因：终端代理一旦没有共享上下文，很容易在真实仓库里做出不贴项目的改动。
</Callout>

## Step 2: 学会发第一条‘先分析、不修改’的指令
一条更好的首次指令通常长这样：

```text
先阅读 AGENTS.md、CLAUDE.md 和 README，
总结这个仓库的技术栈、关键目录和主要约束。
先不要改任何文件，只给我一个简短分析。
```

这条指令的价值在于，它先把代理从“执行者”切成“分析者”。

| 指令写法 | 风险 | 更好的替代 |
|---|---|---|
| 帮我修首页 | 容易直接动手且忽略上下文 | 先读约束文件并总结仓库结构 |
| 看看这个项目 | 太泛 | 明确要求技术栈、关键目录、主要约束 |

## Step 3: 从一个小而完整的任务建立信任
第一次真正执行，最好只做一个 15 到 30 分钟能完成的小任务，例如：

| 小任务 | 为什么适合第一次 |
|---|---|
| 改一个空状态文案 | 改动小、验证快 |
| 调整一个按钮文案 | 风险低、容易对照 |
| 补一个说明段落 | 能完整跑一遍分析 → 计划 → 修改 → 验证 |

同时别忘了常见报错排查：

| 常见报错/问题 | 先检查什么 |
|---|---|
| 命令找不到 | Node / npm / 工具是否在 PATH |
| 无法读目录 | 当前终端目录和权限 |
| 改完不敢放行 | 有没有跑 `lint`、`tsc` 或手动验证 |

<Callout title="before / after：第一次就做大改动 vs 先做小闭环" tone="warning">
错误做法：第一次就让 Claude Code 重构大页面。正确做法：先做一个小任务，完整跑一次分析、计划、修改、验证。差别在于，前者在拿仓库冒险，后者在建立协作信任。
</Callout>

## 本单元小结

- 安装和首次接入的重点是路径、权限、依赖和项目目录。
- 第一次进入仓库时，应先分析上下文，而不是立刻执行。
- 小任务更适合建立终端代理协作的第一层信任。
