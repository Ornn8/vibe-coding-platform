## Step 1: 先完成安装与上下文建立
第一次使用 Claude Code 时，不要急着让它改功能。先进入项目目录，让它分析结构、识别关键文件、读取项目约束文件。这个步骤的意义是建立共享上下文，而不是炫耀它能不能立刻写代码。

如果项目里已经有 `CLAUDE.md`、`AGENTS.md` 或 README，这些都是它最应该先读取的内容。上下文越完整，后续动作越稳。

可以把首次接入理解成“先看图纸再施工”。不同系统下的关注点也不同:

| 环境 | 先看什么 |
|---|---|
| Windows | 路径、权限、终端可用性 |
| macOS | 终端权限、项目目录 |
| Linux | 依赖、权限、路径 |

## Step 2: 用 Plan 模式确认修改方案
确认共享上下文后，再让它给修改计划。一个好计划应该告诉你会改哪些文件、为什么改、可能的风险是什么、准备怎么验证。计划阶段越清楚，执行阶段越省心。

如果它给出的方案仍然很粗，你应该继续 Ask，而不是直接放行。

比如“改一下首页”就太粗，而“只修改首页 Hero 区和 CTA 文案，并说明验证方式”就更像可执行计划。

## Step 3: 从小改动开始建立信任
第一次执行，最好只做一个 15 到 30 分钟能完成的小任务，例如改一段文案、补一个空状态、加一个简单组件。重点不在功能大小，而在于你能完整跑一次“分析 -> 计划 -> 执行 -> 验证”。

这种小闭环会比一次大跃进更快帮你建立真正可持续的协作节奏。

典型小任务例子: 把某个卡片标题改短、给空状态补一句解释、给按钮加一个更清楚的文案。

## Step 4: 对比 GUI 工具与 Claude Code
同一个任务，用 GUI 工具和 Claude Code 各做一次，你会很快感受到两者的差别。GUI 更适合快速试验，Claude Code 更适合真实仓库和严格约束。比较不是为了分高下，而是为了知道什么时候该用哪一把武器。

| 工具 | 优势 |
|---|---|
| GUI 工具 | 快速试验 |
| Claude Code | 真实仓库、计划和验证 |

## 本单元小结

- 首次使用先分析上下文，不急着动手。
- Plan 模式要看文件范围、风险和验证。
- 小任务更适合建立稳定协作节奏。
