<Callout title="本单元方法论焦点" tone="warning">
真正的升级，不是让 AI 在同一线程里做更多事，而是设计一个有角色、有检查、有交接边界的协作系统，让产出更可信。
</Callout>

## Step 1: 先看两段“能跑，但其实危险”的代码
下面这两段代码都能跑，但都不该直接信任：

```ts
export async function GET() {
  const users = await prisma.user.findMany();
  return Response.json(users);
}
```

```ts
export async function updateUser(id: string, input: any) {
  return prisma.user.update({ where: { id }, data: input });
}
```

| 代码样例 | 为什么第一眼看起来没问题 | 实际风险在哪里 |
|---|---|---|
| 公开用户接口 | 很快就能返回真实数据 | 没有任何鉴权和访问边界 |
| 直接更新 helper | 很短、很灵活、很省事 | `any` 输入、没有校验、没有权限判断 |

这就是 M7 要解决的问题。现在的问题已经不是“AI 会不会写代码”，而是“怎么防止看起来像成品的代码变成隐藏债务”。

<Callout title="场景卡片：什么时候“能跑”已经不是完成标准" tone="warning">
情境：页面能打开、接口也能返回结果，所以你下意识觉得任务结束了。做法：继续追问这个改动还没跨过哪些审查边界，比如鉴权、校验、可维护性。原因：项目变大之后，隐藏缺陷的成本往往比表面成功更高。
</Callout>

## Step 2: 角色变化，是从执行者变成系统设计者
在 vibe coding 里，你更多是在提需求、接收结果。在 agentic engineering 里，你先设计角色、输入、输出和审查边界，再让系统开始运转。

| 工作方式 | 核心问题 |
|---|---|
| Vibe coding | “我下一步该怎么让 AI 帮我做？” |
| Agentic engineering | “这件事该由哪个角色做，它要交付什么，谁来验证？” |

这不是身份标签的变化，而是你把设计精力放在了哪里。

```text
Before：我让同一个助手在一个线程里同时规划、写代码、调试、审查。
After：我在开始前先把规划、执行、验证拆成不同职责。
```

## Step 3: PEV 没消失，只是被分布到不同角色里
PEV 循环依然成立，只是现在可以拆给不同角色：

| PEV 环节 | 可能承担的角色 | 具体交付物 |
|---|---|---|
| Plan | 规划 agent | 拆解方案、范围、验收标准 |
| Execute | 编码 agent | 代码改动或内容改动 |
| Verify | 审阅 agent / 测试 agent | findings、失败检查、通过或退回 |

<Callout title="场景卡片：一个真实的角色化交接" tone="info">
情境：你要给项目提交页增加 review note 展示。做法：让一个 agent 先写实现计划，一个 agent 修改代码，再由一个 agent 跑 lint、type check 并检查页面回归。原因：重点不是让系统更忙，而是让交接更清楚、闭环更可检查。
</Callout>

## 本单元小结
- AI Slop 往往是“能跑，但有隐藏风险”的产物。
- Agentic engineering 的关键升级是角色和边界设计，而不只是 prompt 技巧。
- PEV 在多角色系统里反而更重要，因为它决定了交接是否清晰。

## 方法论回顾
规划驱动在这里不再只是“先写计划”，而是“先把整套协作系统设计清楚，再让执行发生”。
