<Callout title="本单元方法论焦点" tone="success">
这个项目的重点不是自动化最多步骤，而是证明你能把一条有价值的流程自动化，同时清楚地保留人类仍然必须介入的审查边界。
</Callout>

## Step 1: 先选一条边界清楚的自动化路径
你可以从这些路径里选一条：

| 候选路径 | 为什么适合作为项目起点 |
|---|---|
| Issue -> 计划草稿 | 触发清晰，写入风险低 |
| 计划 -> 实现草稿 | 适合边界稳定的任务 |
| 实现 -> lint / test / 报告 | 价值高，歧义低 |
| 测试结果 -> 审查摘要 | 很适合结构化汇总工作 |

这个项目不需要很大，但必须清晰、受控，而且值得重复。

<Callout title="场景卡片：为什么第一条自动化路径应该尽量短" tone="info">
情境：你很想直接自动化整个 issue 到 merge 的流程。做法：先从“实现 -> lint/test 报告”这种更短的链路开始。原因：链路越短，越容易验证，也越容易暴露隐藏风险。
</Callout>

## Step 2: 给它补上机器守卫和人工检查点
每一条自动化路径都应该同时声明机器检查和停止点。

| 阶段 | 机器守卫 | 人工检查点 |
|---|---|---|
| 生成计划 | 格式和完整性检查 | 人确认范围和风险 |
| 生成代码 | lint、type check、测试 | 人审业务逻辑和 UX 影响 |
| 涉及上线 | 密钥扫描和配置检查 | 人审批上线风险 |

如果你说不清自动化“不该跨过什么”，那这条流程就还没准备好。

<Separator />

## Step 3: 用一个具体框架回顾方法论成长
把这个项目同时当成一次方法论复盘：

| 模块阶段 | 你在复盘时要回答的问题 |
|---|---|
| M1-M2 | 我是否定义对了问题和范围？ |
| M3-M4 | 我是否把任务描述得足够清楚，能让 AI 和我自己都不跑偏？ |
| M5-M6 | 我是否保留了上下文、checkpoint 和可复用资产？ |
| M7 | 我是否只自动化了边界清楚、审查点明确的部分？ |

这一步的意义，是让项目不只是一次实验，而是一次方法论里程碑。

```text
Before：我把自动化项目理解成“能自动化多少就自动化多少”。
After：我把它理解成“只自动化边界清楚的一段，并明确机器守卫和人工回到闭环的入口”。
```

## 本单元小结
- 好的 M7 项目是自动化一条边界清楚的路径，而不是自动化一切。
- 机器检查和人工检查点都必须显式设计出来。
- 最终复盘既要看流程产出，也要看你的方法论成熟度是否真的提高了。

## 方法论回顾
闭环交付在这里进入系统级阶段：闭环不再只是一个人执行完再验证，而是自动化、守卫和人工再进入共同组成的完整系统。
