<Callout title="本单元方法论焦点" tone="warning">
多 Agent 不是为了热闹，而是为了让每一类工作都由最合适的角色来完成。
</Callout>

## Step 1: 典型工作流长什么样
一个基本的多 Agent 工作流通常是：规划 → 编码 → 测试 → 审查。它的价值在于每个角色都只做自己最擅长的那一段，而不是让一个 Agent 包办所有事情。  
你可以把它理解成一个小团队，而不是一个超级员工。
场景对比：如果一个 Agent 从头干到尾，容易在实现和审查之间失去客观性；拆成多个角色后，每个环节都有独立视角。

## Step 2: sub-agent 和 Agent Team 怎么协同
子 Agent 的关键不是数量，而是分工。主 Agent 负责拆任务、汇总结果，子 Agent 负责完成局部工作。  
当你把一个三角色链条设计清楚后，你会发现复杂任务并没有消失，只是被拆到了更合适的位置。
场景对比：主 Agent 像项目经理，子 Agent 像专员。项目经理不直接把每一步都做完，而是把局部目标交给合适的角色。

## Step 3: 自动化把 Issue 变成 PR
多 Agent 工作流最强的一点，是可以和自动化结合：Issue 进来，Agent 接手，输出 PR，再由人审查。  
这时你的工作不再是手工复制粘贴，而是管理一条自动化协作链。
场景对比：手工流程每次都要重复描述需求；自动化链路则能把输入、执行和审核固定成标准路径。

## 本单元小结
- 多 Agent 工作流的核心是分工，不是堆数量。
- 主 Agent 负责编排，子 Agent 负责局部任务。
- 自动化可以把工作流升级成稳定的输入输出管道。

## 方法论回顾
当任务被拆成角色链条后，你就不再只是在和 AI 对话，而是在运行一个小型协作系统。
