<Callout title="本单元方法论焦点" tone="warning">
多 agent 的价值，不是“窗口更多”，而是通过清晰写入范围、明确交接和受控并行，降低协作成本。
</Callout>

## Step 1: 边界不清时，不要急着并行
并行只在子任务有清晰归属时才真的有帮助。

| 任务形态 | 适合多 agent 吗 | 为什么 |
|---|---|---|
| 批量改 8 篇互不相干的 MDX | 适合 | 每个文件都能独立修改和独立审阅 |
| 调一个跨多处的 auth bug | 通常不适合 | 问题路径变化太快，适合紧密单线程 |
| 一个有明确规划和审阅环节的功能 | 适合 | 不同角色之间能形成真实价值 |

如果一个 agent 同时负责规划、实现、测试和审查，你得到的不是系统，而是更快的混乱。

## Step 2: 看一个真实的 sub-agent 工作流
假设你要批量升级一个模块的 lesson 内容：

| 角色 | 输入 | 输出 | 写入范围 |
|---|---|---|---|
| 规划 agent | 修订标准 + 文件清单 | 改写 checklist 和任务拆分 | 不改文件 |
| 改写 agent | 单个 lesson 文件 + checklist | 更新后的 lesson 内容 | 只改这一个文件 |
| 审阅 agent | 更新后的 lesson + 标准 | findings 和漏项 | 不改文件 |

真实流程是这样的：

1. 规划 agent 先读修订标准，并拆分文件任务。
2. 改写 agent 一次只改一个 lesson。
3. 审阅 agent 检查场景卡片、表格、before/after 和 L3 练习。
4. 主 agent 或人类根据 findings 收敛后，再进入下一批。

这才是真实的 sub-agent 工作流。它不是一张框图，而是一组受控的输入、输出和权限关系。

```text
Before：我把规划、实现、测试和审阅全部塞进同一个 agent。
After：我先拆清楚角色边界，再让每个 agent 只处理它能安全负责的那一段。
```

<Callout title="场景卡片：什么时候 sub-agent 是有用的，而不是装饰性的" tone="info">
情境：你想让系统帮你做批量内容升级。做法：给改写 agent 只分配一个文件和一份 checklist，给审阅 agent 只分配只读审查职责。原因：每个角色既知道自己该做什么，也知道自己不能碰什么，系统才会更安全。
</Callout>

## Step 3: 自动化应该建立在稳定流程上，而不是代替判断
当角色模型足够稳定之后，部分流程才适合自动化。

| 工作流阶段 | 适合自动化吗 | 人工检查点 |
|---|---|---|
| 从清晰任务触发流程 | 常常适合 | 人确认范围 |
| 生成计划草稿 | 常常适合 | 人确认边界和风险 |
| 跑 lint / type check | 适合 | 人查看失败项和高风险项 |
| 自动合并业务逻辑改动 | 通常不适合 | 人工审批必须保留 |

重点不是把所有环节都自动化，而是写清自动化的边界。

## 本单元小结
- 多 agent 只有在写入边界和交接都清楚时才真正有价值。
- 真实 sub-agent 流程的关键是输入、输出和权限，不是概念图。
- 自动化只有建立在稳定流程之上才安全。

## 方法论回顾
规划驱动在这里升级成编排能力：先设计角色边界，再决定是否并行和是否自动化。
