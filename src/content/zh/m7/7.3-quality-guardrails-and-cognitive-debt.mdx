agent 产出越快，质量守卫越重要。没有守卫，吞吐量会比团队预期更快地变成认知债务。
</Callout>

## Step 1: 规模会放大缺陷，就像它会放大产出一样
假设一个 agent 工作流每天产出 20 次改动，缺陷率只有 2%。

| 指标 | 数值 |
|---|---|
| 每天改动数 | 20 |
| 缺陷率 | 2% |
| 每天预期坏改动数 | 0.4 |
| 5 天累计坏改动数 | 2 |

一周 2 个坏改动，已经足以带来持续返工、持续回滚和持续不信任。

<Callout title="场景卡片：为什么很低的错误率依然会变贵" tone="warning">
情境：单次缺陷看起来不高，所以你觉得系统应该很安全。做法：把缺陷率乘以一周或一个月的改动总量。原因：系统成本不会出现在一次顺利运行里，而是出现在规模化之后。
</Callout>

## Step 2: 先让机器拦住机械性失败
最小质量守卫循环可以是：

```bash
npm run lint
npx tsc --noEmit
npm test
```

假设 lint 跑出来的是：

```text
src/app/api/profile/route.ts
  12:18  error  Unexpected any. Specify a different type  @typescript-eslint/no-explicit-any
```

然后修复成：

```ts
type ProfileInput = {
  displayName: string;
  bio?: string;
};
```

| 守卫 | 它最擅长抓什么 | 它不擅长判断什么 |
|---|---|---|
| ESLint | 风格问题、不安全模式、部分逻辑味道 | 产品正确性 |
| Type check | 类型不匹配、缺字段 | 用户体验质量 |
| 测试 | 已知行为回归 | 架构取舍 |

这就是质量守卫的核心：把机械性失败尽量挡在自动化层，让人工审查把注意力放在判断题上。

## Step 3: 人类仍然负责风险、含义和长期成本
不是所有通过自动化的改动都应该被自动信任。

| 审查类别 | 最适合的审查者 |
|---|---|
| 语法、风格、类型安全 | 机器 |
| 权限边界和业务规则 | 人 |
| 架构取舍和长期维护成本 | 人 |
| 密钥处理和上线风险 | 机器 + 人 |

```text
Before：只要 lint 和测试过了，我就倾向于相信整个改动。
After：lint 和测试只是在清理机械层；高风险业务含义和边界判断仍然必须由人负责。
```

## 本单元小结
- 很低的缺陷率在规模化流程里也会快速变成系统成本。
- lint、类型检查、测试应该优先拦住机械性失败。
- 人类审查应该保留给风险判断、业务含义和长期维护问题。

## 方法论回顾
闭环交付在这里升级成系统级验证：机器负责挡住可重复失败，人类负责守住仍然需要判断的边界。
