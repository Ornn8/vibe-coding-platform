<Callout title="本单元方法论焦点" tone="warning">
能复盘、能修正、能复用的 prompt 才算资产。一次碰巧有效，不算方法。
</Callout>

## Step 1: 反模式 1，模糊 prompt 会让 AI 全靠猜
看一个失败链路：

| 阶段 | 内容 |
|---|---|
| 原始 prompt | 帮我优化一下课程页 |
| AI 输出 | 改了 Hero、加了营销文案、顺手改了按钮文案 |
| 真正问题 | 你没说优化什么、范围在哪 |
| 修正写法 | 只检查 Hero 标题、简介和 CTA，让初学者 10 秒内看懂本单元学什么，不改其他区域 |
| 修正后效果 | 改动范围收窄，输出更可审查 |

这类反模式的问题不是“字数太少”，而是缺目标、缺范围、缺完成标准。

## Step 2: 反模式 2，贪心 prompt 会把重点摊薄
再看一个链路：

| 阶段 | 内容 |
|---|---|
| 原始 prompt | 帮我把首页、登录、评论、支付、会员都做出来 |
| AI 输出 | 每个点都做了一点，但没有一个点真的稳 |
| 真正问题 | 你把多个阶段的事塞进了一轮 |
| 修正写法 | 本轮只做登录和支付前的商品详情页，不做评论和会员 |
| 修正后效果 | AI 更容易聚焦一个闭环 |

```text
Before：把所有想到的功能都塞进一轮。
After：只围绕一个闭环推进，并明确不做项。
```

<Callout title="场景卡片：为什么贪心 prompt 特别容易让人误判" tone="warning">
情境：AI 一下给了很多功能，看起来很值。做法：先问自己有没有一个核心闭环真正做稳。原因：功能多不等于交付稳，贪心 prompt 很容易制造‘看起来完成’的幻觉。
</Callout>

## Step 3: 反模式 3，没有验证要求就很难形成闭环
最后看第三类失败链路：

| 阶段 | 内容 |
|---|---|
| 原始 prompt | 修好这个 bug |
| AI 输出 | 给出一段看起来像修复的代码 |
| 真正问题 | 你没要求它说明怎么验证 |
| 修正写法 | 修好这个 bug，并说明你会如何验证，至少包含一条手动验证或命令检查 |
| 修正后效果 | 结果不再停留在“看起来像修好了” |

没有验证要求的 prompt，最大的问题是你会把“输出出现了”误当成“问题解决了”。

## Step 4: 把有效 prompt 沉淀成 Portfolio
当你开始记录“任务类型、prompt、结果、适用场景、常见坑、验证方式”，你就不是在保存聊天记录，而是在管理 Prompt Portfolio。

| 资产字段 | 应该记录什么 |
|---|---|
| 任务类型 | 内容审阅、需求拆解、代码修复等 |
| prompt 本体 | 真正有效的版本 |
| 适用场景 | 什么时候用 |
| 常见坑 | 容易跑偏在哪里 |
| 验证方式 | 怎样判断这次结果算过关 |

<Callout title="before / after：聊天记录 vs Prompt Portfolio" tone="info">
错误做法：觉得‘我记得这条 prompt 以前好像有用过’。正确做法：把 prompt 连同场景、问题和验证方式一起记下来。差别在于，前者靠记忆，后者才是资产。
</Callout>

<Exercise id="ex-3.5.4.1" level="L3" title="整理你的 Prompt Portfolio" description="从你真实使用过的 prompt 里挑 5 条，按任务类型整理成小型 Portfolio，并为每条补上适用场景、常见坑和验证方式。" />

## 本单元小结

- 模糊、贪心、缺验证是最常见的三类反模式。
- 反模式要看完整失败链路，才能真正知道该怎么改。
- Portfolio 让 prompt 从一次性对话变成可迁移的工作资产。

## 方法论回顾

本单元把闭环交付扩展到了 prompt 层：不仅要有输出，还要能复盘、修正和复用。
