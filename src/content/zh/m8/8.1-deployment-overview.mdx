<Callout title="本单元方法论焦点" tone="warning">
部署不是收尾时顺手点一下按钮，而是交付闭环的最后一段：本地能跑，到了真实构建、真实配置和真实运行环境里还能不能成立。
</Callout>

## Step 1: 先建立一次具体的部署体验
以一个 Next.js 项目部署到 Vercel 为例：

1. 打开 Vercel 控制台，点击 `Add New...` -> `Project`。
2. 选择你的 Git 仓库。
3. 在导入页面确认框架被识别为 `Next.js`。
4. 展开环境变量区域，填写 `DATABASE_URL`、`AUTH_SECRET` 之类的配置。
5. 点击 `Deploy`。
6. 观察构建日志依次经过 install、build、route generation。
7. 打开 preview URL，验证真实线上页面。

这条“打开 -> 选择 -> 填值 -> 部署 -> 验证”的路径，才是学习者真正需要脑补出来的部署过程。

| 步骤 | 你要点什么 / 填什么 | 你应该看到什么 |
|---|---|---|
| 导入仓库 | 选择 Git 仓库 | 自动识别框架和项目设置 |
| 填环境变量 | 粘贴运行时配置和密钥 | 变量在部署前被列出来 |
| 点击部署 | 点击 deploy 按钮 | 构建日志和 preview URL |

<Callout title="场景卡片：第一次部署最常见的坑" tone="warning">
情境：构建成功了，但页面运行时报错。做法：先检查部署控制台里是否真的填了运行时环境变量，而不是只在本地 `.env` 里写过。原因：很多第一次部署失败，本质上不是代码问题，而是配置没带上去。
</Callout>

## Step 2: 用硬信息来选平台，而不是凭感觉
平台选择应该基于真实代价和控制力取舍。

| 平台类型 | 启动成本 | 控制力 | 更适合什么项目 |
|---|---|---|---|
| Vercel / Netlify | 低 | 较低 | 前端为主、Next.js 站点、快速上线 |
| Railway / Render | 中 | 中 | 小型全栈应用和托管服务 |
| Docker + VPS | 高 | 高 | 需要自定义基础设施且能承担运维成本的团队 |

正确答案不是“最强的平台”，而是“对这个项目来说，运维成本仍然合理的平台”。

## Step 3: 部署失败时，按层排查
部署出错时，不要乱跳，按层排查：

| 层级 | 常见症状 | 第一检查点 |
|---|---|---|
| Build 层 | 安装或编译失败 | 构建日志、依赖缺失 |
| Environment 层 | 应用启动即崩 | 环境变量、密钥、回调配置 |
| Service 层 | API 或数据库调用失败 | 连接串、网络规则、服务状态 |
| Application 层 | 页面能开但业务逻辑不对 | runtime log 和真实用户流程 |

```text
Before：部署失败后，我在控制台和代码之间来回乱点。

```

## 本单元小结
- 学部署不能只记平台名字，必须能脑补一条具体操作路径。
- 选平台要看启动成本、控制力和项目适配度。
- 部署失败按层排查，比到处乱试更快定位问题。

## 方法论回顾
闭环交付在这里跨过最后一道门槛：构建、配置和运行时验证一起决定本地项目能否变成真实产品。
