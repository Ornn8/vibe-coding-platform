<Callout title="本单元方法论焦点" tone="warning">
AI 不只会帮你更快交付功能，也会帮你更快复制风险。安全检查不是上线前的装饰，而是负责任交付的一部分。
</Callout>

## Step 1: 先看漏洞代码和修复代码的直接对比
这是一个硬编码密钥的错误示例：

```ts
const stripe = new Stripe("sk_live_123");
```

更安全的版本：

```ts
const stripeKey = process.env.STRIPE_SECRET_KEY;

if (!stripeKey) {
  throw new Error("Missing STRIPE_SECRET_KEY");
}

const stripe = new Stripe(stripeKey);
```

这是一个权限边界过宽的示例：

```ts
export async function GET() {
  return Response.json(await prisma.review.findMany());
}
```

更安全的版本：

```ts
export async function GET() {
  const session = await auth();

  if (!session?.user || session.user.role !== "ADMIN") {
    return Response.json({ error: "Forbidden" }, { status: 403 });
  }

  return Response.json(await prisma.review.findMany());
}
```

| 风险类型 | 有漏洞的写法 | 修复后的写法 |
|---|---|---|
| 密钥管理 | 把真实 key 写进源码 | 从 env 读取，并在缺失时安全失败 |
| 权限边界 | 敏感数据公开读取 | 显式鉴权 + 角色校验 |

## Step 2: 把密钥和输入处理当成正式上线工作
另一个常见问题，是过早信任输入：

```ts
await prisma.post.create({ data: req.body });
```

更安全的版本：

```ts
const parsed = postSchema.safeParse(await req.json());

if (!parsed.success) {
  return Response.json({ error: "Invalid input" }, { status: 400 });
}

await prisma.post.create({ data: parsed.data });
```

| 审查项 | 上线前要确认什么 |
|---|---|
| 密钥 | 没有真实密钥被提交或硬编码 |
| `.env` 处理 | `.env` 被忽略，`.env.example` 只保留占位符 |
| 输入校验 | 外部输入在使用前被解析和校验 |
| 错误处理 | 不向用户暴露敏感实现细节 |

<Callout title="场景卡片：为什么本地 happy path 正常，不代表能安全上线" tone="info">
情境：表单在本地提交流程里一切正常，所以你觉得它已经安全。做法：继续检查异常输入、缺失密钥和未授权访问时的行为。原因：很多安全问题在 happy path 里根本不会暴露出来。
</Callout>

## Step 3: 把安全变成可重复的上线 checklist
第一天不需要企业级安全体系，但必须有一套每次上线前都能重复执行的最小检查。

| Checklist 项 | 为什么必须上线前确认 |
|---|---|
| 密钥已外置 | 避免凭证泄漏 |
| 敏感路由有权限边界 | 避免数据误暴露 |
| 输入已校验 | 避免危险写入和崩溃 |
| 日志和错误信息已审查 | 避免内部实现暴露 |

```text
Before：上线前我只会检查“页面能不能打开、功能能不能跑通”。
After：我会继续检查密钥、输入校验、权限边界和错误暴露，再决定这个版本是否真的能上线。
```

## 本单元小结
- 安全更容易通过“漏洞代码 vs 修复代码”的方式被真正理解。
- 密钥处理、权限边界和输入校验必须在上线前检查，而不是上线后补救。
- 一份短但可重复的 checklist，比空泛的安全说教更有价值。

## 方法论回顾
闭环交付在这里增加了上线安全门：不只问“能不能用”，还要问“给真实用户用之前是否足够安全”。
