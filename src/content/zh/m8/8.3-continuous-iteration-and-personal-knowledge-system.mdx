<Callout title="本单元方法论焦点" tone="success">
真正强的构建者，不只是做完一个项目，而是会把每个项目沉淀成下一次还能复用的规划、执行和复盘资产。
</Callout>

## Step 1: 先看一份真实的 `feature-implementation.md`
下面是一份具体示例：

```md
# Feature: Review Notes on Project Submissions

## Goal
Allow reviewers to attach notes to each submission so students can see actionable feedback.

## Scope
- Add review-note field to reviewer workflow
- Show the note on the student project page
- Do not redesign the whole review dashboard

## Acceptance
- Reviewer can save one note
- Student can read the latest note
- `npm run lint` and `npx tsc --noEmit` pass

## Risks
- Notes may expose admin-only wording to students
- Existing review flow must remain unchanged

## Verification
- Test reviewer save path
- Test student visibility path
- Confirm old submissions still render safely
```

| 部分 | 为什么重要 |
|---|---|
| Goal | 让功能始终绑定一个真实结果 |
| Scope | 防止顺手扩展和需求漂移 |
| Acceptance | 让验证变具体 |
| Risks | 强迫你把隐藏失败模式写出来 |
| Verification | 定义这个功能怎么才算可信 |

## Step 2: 把项目资产组织成个人知识体系
到这里为止，你已经积累了 prompts、feature 文档、Skill、模板和复盘。

| 资产类型 | 更适合放在哪里 |
|---|---|
| Feature 文档 | `/features/` 或项目 planning 目录 |
| Skill | `/skills/` 或可复用工作流库 |
| 复盘 | `/retrospectives/` 或项目回顾归档 |
| Prompt / 模板 | `/templates/` 或个人 playbook |

如果这些内容继续埋在旧对话或散落文件夹里，它们就不会真正复利。

<Callout title="场景卡片：什么时候一次迭代已经值得沉淀成资产" tone="info">
情境：你刚完成一个功能，脑子里还记得哪些 prompt 最有效、哪些检查最容易漏。做法：立刻把它整理成 feature 文档、Checklist 或 Skill 草稿。原因：经验最容易在迭代刚结束时被抓住，也最容易在几天后彻底散掉。
</Callout>

```text
Before：有用的 prompt 和计划都被埋在旧对话里。
After：这些资产按类型归档，所以下一个项目能从可复用材料开始，而不是从记忆开始。
```

## Step 3: 成熟构建者知道 Vibe Coding 的边界在哪里
不是所有项目都适合同样程度的松弛开发。

| 项目类型 | Vibe coding 适配度 | 为什么 |
|---|---|---|
| Landing page 原型 | 高 | 迭代快、风险低 |
| 内部运营后台 | 中 | 速度有价值，但数据边界仍然重要 |
| 医疗或金融流程 | 低 | 错误的现实代价太高 |

真正成熟的判断，不是“AI 能不能生成”，而是“这类工作是否适合依赖快速、偏松的迭代方式”。

## 本单元小结
- 完整 feature 文档是实用规划资产，不是形式主义。
- prompt、文档、Skill 和复盘只有被系统化保存，才会真正复利。
- 成熟构建者会判断 Vibe Coding 何时该停、严格工程何时该上场。

## 方法论回顾
上下文固化在这里变成长期个人系统：每个项目都应该留下能复用的规划和复盘资产，而不是消失在旧对话里。
