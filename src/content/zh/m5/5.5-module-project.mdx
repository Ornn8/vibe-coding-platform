<Callout title="本单元方法论焦点" tone="warning">
持续迭代的重点不是加更多功能，而是让每次新增能力都能沿着依赖顺序稳稳落下去。
</Callout>

## Step 1: 先列变更清单，再判断依赖关系
这次项目不是推倒重来，而是在 M4 的 CRUD 基础上继续升级，比如补认证、数据迁移、权限边界或交互改进。

| 改动项 | 它依赖什么 | 风险在哪里 |
|---|---|---|
| 加认证 | 用户模型、登录态存储 | 影响权限和页面访问 |
| 数据迁移 | 现有 schema、旧数据兼容 | 可能影响已有功能 |
| 权限页保护 | 认证状态是否稳定 | 页面可能出现未授权错误 |

如果你不先画出依赖图，很容易把“加一个功能”做成“顺手重构整个项目”。

<Callout title="场景卡片：为什么依赖关系先于实现顺序" tone="info">
情境：你准备先做登录页，因为它最显眼。做法：先看它依不依赖用户表、权限字段和数据结构。原因：真正决定先后顺序的不是页面显不显眼，而是依赖链是否已经成立。
</Callout>

## Step 2: 按依赖顺序写迭代计划
同样是“加认证和迁移”，顺序可能完全不同。

| 情况 | 更合理的顺序 |
|---|---|
| 认证依赖用户字段 | 先数据迁移，再接认证 |
| 页面保护依赖登录态 | 先认证流程，再加页面守卫 |
| 新表单依赖权限判断 | 先权限，再开表单入口 |

```text
Before：按感觉排序，想到哪做哪。
After：先处理被别人依赖的底层，再处理建立在它上面的页面和交互。
```

这一步不是为了把计划写漂亮，而是为了减少返工。依赖顺序一旦写反，后面每一轮都会更贵。

## Step 3: 提交的不只是代码，而是完整证据链
这个项目真正要交的是：

| 交付物 | 为什么必须交 |
|---|---|
| 代码结果 | 证明你做成了什么 |
| Git 历史 / checkpoint | 证明迭代路径可回溯 |
| Memory Bank | 证明上下文可以跨会话继续 |
| Debug 记录 | 证明出错时你能闭环修复 |

如果只有代码，没有过程资产，别人很难判断你是靠方法稳稳推进，还是靠运气碰出来。

<Callout title="before / after：只交成品 vs 交证据链" tone="warning">
错误做法：只交一个能跑的结果。正确做法：同时交代码、checkpoint、Memory Bank 和 debug 记录。差别在于，前者只能展示结果，后者才能展示你是否具备持续迭代能力。
</Callout>

## 本单元小结
- 模块项目的关键是持续迭代，不是从零重做。
- 先看依赖关系，再决定实施顺序。
- 完整提交必须同时展示结果和证据链。

## 方法论回顾
闭环交付在这里进入“持续演进”阶段：项目不只要能完成，还要能一轮轮稳定长大。
