<Callout title="本单元方法论焦点" tone="warning">
版本控制不是 AI 编程外面的附属技能，而是让快速迭代仍然可恢复、可比较、可审查的最低安全系统。
</Callout>

## Step 1: 先看一个真实的“AI 把代码改崩了”故事
假设一个课程首页已经能正常工作。你让 AI 连续改三轮：

1. 改 Hero 文案。
2. 调导航结构。
3. 让整体留白更舒展。

第三轮结束后，页面白屏了。你知道哪里坏了，但你不知道问题是第一轮、第二轮还是第三轮引入的。

| 状态 | 你手里有什么 | 接下来会发生什么 |
|---|---|---|
| 没有 checkpoint | 只有模糊记忆 | 只能靠猜测回滚，返工成本高 |
| 每个已验证步骤都 commit | 清晰的版本历史 | 能快速定位是哪一次改动引入了问题 |

```text
Before：我让 AI 连续改了三轮，但没有留下任何稳定状态点。
After：每完成一个已验证的小步骤，我都会先 commit，再进入下一轮。
```

Git 在这里的价值不是“更专业”，而是把“我感觉是后面才坏的”变成“就是这个 commit 引入了问题”。

<Callout title="场景卡片：什么时候最该先 commit 再继续" tone="warning">
情境：当前功能已经通过页面检查和终端验证，你正准备让 AI 继续下一轮。做法：先把当前稳定状态 commit 下来。原因：一旦下一轮改坏，你需要一个真实可回退的落点，而不是依赖记忆。
</Callout>

## Step 2: 按工作节奏选工具，而不是按习惯选工具
Claude Code、Codex 和 GUI 工具不是谁替代谁，而是分别适合不同节奏的工作。

| 工具 | 启动成本 | 更适合的阶段 | 典型任务 | 具体动作或命令 |
|---|---|---|---|---|
| Claude Code | 低，已经在真实仓库里就能直接开始 | 实时实现与调试阶段 | 读代码、改代码、验证、继续迭代 | `npm run lint`、检查文件、修一个真实 bug |
| Codex | 中，需要先把任务边界写清楚 | 边界明确的异步执行阶段 | 批量重写、独立模块生成、按 spec 落地 | 交付一个清晰任务说明，等待异步完成 |
| GUI 原型工具 | 低，适合快速探索方向 | 最早期的界面探索阶段 | 试页面风格、试布局、试视觉方向 | 直接 prompt 一个 landing page 或 dashboard 原型 |

错误的问题是“我最喜欢哪个工具”。更好的问题是“这次任务需要什么工作节奏”。

<Callout title="场景卡片：同一个项目里为什么常常要混用工具" tone="info">
情境：你既要改 lesson 页视觉，也要修一个真实表单 bug，还要批量重写结构化文档。做法：先用 GUI 工具探索页面方向，用 Claude Code 修真实 bug，再把边界清楚的文档改写交给 Codex。原因：这些任务的反馈回路不同，最安全的工具也会跟着变化。
</Callout>

## Step 3: 真正的重点是让闭环更顺，而不是工具更多
好的工具协作，不是“工具越多越好”，而是“减少打断，同时保留可回退性”。

| 场景 | 更顺的组合 |
|---|---|
| 快速试一个视觉方向 | GUI 工具 -> 确认方向 -> 再回到真实仓库 |
| 在真实仓库里安全改动 | Claude Code -> 验证 -> Git checkpoint |
| 交付一个边界明确的后台任务 | 写清 spec -> Codex 异步执行 -> 人工审阅 -> checkpoint |

如果你把实时调试丢给异步工具，反馈回路会变慢。如果你把边界很清楚的批量任务放进长对话里，反而会让上下文越来越吵。工具选择本身就是交付设计的一部分。

<Callout title="before / after：按偏好选工具 vs 按工作流选工具" tone="info">
错误做法：因为习惯某个工具，就所有任务都用它。正确做法：先判断这次任务更像探索、实时协作，还是异步生成，再决定工具。前者优化的是习惯，后者优化的是交付。
</Callout>

## 本单元小结
- Git checkpoint 是 AI 协作里让改动可回退的安全网。
- Claude Code、Codex 和 GUI 工具分别适合不同工作节奏。
- 工具选择本身也是闭环交付设计的一部分，因为它会影响下一步验证是否顺畅。

## 方法论回顾
闭环交付在这里多了一层要求：不只是“能不能快”，还要保证每一次快之后都能安全比较、恢复并继续推进。
