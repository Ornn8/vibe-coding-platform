<Callout type="methodology" title="Methodology Focus">
Reliable AI collaboration is not AI running in one long burst. It is a visible loop with verification at the end of every round.
</Callout>

## Step 1: Read One Full PEV Execution Record
Imagine the task is "add a completion counter to the tomato-timer page." A full PEV record might look like this:

| Stage | Real content |
|---|---|
| Plan | Put the counter on the homepage, do not add another page, and keep the count in local state for now |
| Execute | Add a stats card and increment the count when one timer cycle completes |
| Verify | Complete one full timer cycle, check whether the count increases, then reload and confirm whether reset behavior matches the plan |

That loop is simple, but it forces two pauses: one before action and one after action. A lot of rework comes from skipping the second one.

<Callout type="scenario" title="Why PEV is safer than one long run">
Situation: AI already produced the feature and you want to rush into the next step. Action: stop and verify whether the current result truly matches the original plan. Why: small mistakes become expensive when you carry them into later steps.
</Callout>

## Step 2: Ask, Plan, and Direct Execute Are Timing Decisions
These three modes are easier to understand as timing choices:

| Mode | When to use it | Example |
|---|---|---|
| Ask | The target is still unclear or the solution space is wide | Should the stats card live on the homepage or in a detail view? |
| Plan | The target is clear, but the change carries risk | Show me the change scope and the verification steps first |
| Execute directly | The task is tiny, low-risk, and already bounded | Change the button label from "Start" to "Start Focus" |

```text
Before: every task goes straight to execution.
After: unclear target means Ask, risky path means Plan, and only tiny bounded work goes straight to Execute.
```

## Step 3: Large Work Needs Nested PEV Loops
PEV is not only for whole projects. It should also appear inside substeps. Even "work on the database" can be split like this:

| Subtask | Plan | Execute | Verify |
|---|---|---|---|
| Design fields | Decide field names and types | Write the schema | Check whether the fields cover the requirement |
| Create the table | Confirm migration approach | Run the migration | Confirm the table exists correctly |
| Query data | Decide the returned shape | Write the query | Confirm the page gets the right data |

That is why engineering workflows look slower at first while staying much more stable over time.

<Callout type="scenario" title="One large action vs nested PEV">
Wrong move: treat "do the database" as one black-box step. Better move: separate field design, migration, and querying, and verify each part. One path is hard to debug. The other creates usable checkpoints.
</Callout>

## Unit Summary

- The most important letter in PEV is often V.
- Ask helps with direction, Plan helps with path, and direct Execute belongs only to small bounded work.
- Large work becomes safer when you nest smaller PEV loops inside it.
