## Step 1: Plan, Execute, Verify

PEV stands for Plan, Execute, Verify. It means you do not let the model run blindly. You state the task, you run the action, and then you check the result before moving on.

That rhythm is what keeps the work from drifting.

For a small task, the plan might be "rename one component, run lint, confirm the page still builds." For a bigger task, it might be "outline the change first, then execute in stages, then verify each stage."

## Step 2: Ask or Plan Before You Act

Not every task should be executed immediately. Sometimes you should ask the model to explain its plan first. Sometimes you should ask clarifying questions. Sometimes you should simply proceed.

Ask/Plan mode is the judgment call that helps you decide which path is right.

Examples:

- Ask when the task is unclear or could change the scope.
- Plan when the task is important but needs a safe sequence.
- Execute directly when the change is tiny and the risk is low.

## Step 3: Nest PEV Inside Bigger Work

A large task can be broken into smaller ones, and each smaller one can still follow PEV. That is how the rhythm scales. You do not abandon verification when the task gets bigger; you make verification more explicit.

That nested structure is one of the reasons AI collaboration becomes more stable over time.

For example, "add an edit form" can become: plan the fields, implement the form, verify the save path. Each of those substeps can also be checked with PEV.

<Exercise id="ex-4.2.3.1" level="L3" title="Expand one step into a mini PEV" description="Take one larger task and break one step into a smaller Plan, Execute, Verify sequence." />

The bigger the task, the more important it is to make the plan visible before you start changing things.

## Unit Summary

- PEV makes execution predictable.
- Ask/Plan is a judgment layer that helps you decide how to proceed.
- Larger tasks become manageable when you nest PEV inside smaller steps.
