<Callout type="methodology" title="Methodology Focus">
PRDs are not paperwork for other people, and plans are not ceremony. Both are forward decisions that reduce later rework.
</Callout>

## Step 1: Feel What a PRD Is Solving Through One Tomato-Timer Example
Imagine you want to build a tomato-timer page for students. A PRD does not begin by filling many pages. It begins by making five things explicit:

| PRD section | Why it exists | Tomato-timer example |
|---|---|---|
| Background | Explains why the project matters | Students want 25-minute focus sessions but get distracted easily |
| User | Prevents you from designing for "everyone" | Students who need short focus blocks |
| Core flow | Defines the page and feature spine | Start timer -> focus -> break reminder -> review completions |
| Scope | Prevents first-version sprawl | v1 only includes solo timing and completion tracking |
| Acceptance criteria | Defines what done means | A new user can start a focus session within 10 seconds |

The real job of the PRD is simple: decide what this round will deliver and what it will not.

<Callout type="scenario" title="Why PRDs start with user and flow">
Situation: you want to jump straight into stack discussions. Action: write the user, the flow, and the scope first. Why: if the target is still unstable, technical discussion is only orbiting uncertainty.
</Callout>

## Step 2: Fill a One-Page PRD and Explain Each Part
Once you compress the tomato-timer example into a one-page PRD, each line is visibly saving future rework:

```text
Project: student tomato timer page
Target user: students who need short focus sessions
Core flow: start a 25-minute timer -> finish -> record one completion
P0 scope: start / pause / reset timer, completion count
Exclusions: login, social features, leaderboards, multi-device sync
Acceptance criteria: a new user can start a timer within 10 seconds
```

```text
Before: I want to build a productivity tool.
After: I want to build a tomato-timer page for students, and v1 only validates timing and completion tracking, not login or social features.
```

That is the value of the PRD: it turns "I think I want to build something" into "this is the exact small thing this round is building."

## Step 3: The Implementation Plan Solves a Different Problem
Once the PRD answers what to build, the plan answers how to move through the build in a safe order. Use the same tomato-timer project:

| Plan step | Why it comes first | How to verify it |
|---|---|---|
| Define the page structure | The core flow needs a visible shell | The page shows the timer area and start control |
| Build timer logic | The main capability must work first | Countdown starts and finishes correctly |
| Add completion tracking | The core loop becomes meaningful | One completion appears after one finished session |
| Add interaction polish | Details should come after the core loop | Button states and helper text behave correctly |

The plan's job is to answer: what comes first, and how do we avoid making each step too large?

<Callout type="scenario" title="PRD without a plan vs PRD plus a plan">
Wrong move: know what the project is and jump straight into building. Better move: split the page shell, timer logic, completion tracking, and polish into separate steps. One path creates a blob. The other creates checkpoints.
</Callout>

## Step 4: Glue Coding Starts Here, Not Later
Writing the plan also means choosing mature building blocks instead of defaulting to invention.

| Project area | Lower-risk move | Higher-risk move |
|---|---|---|
| UI | Reuse established components | Hand-build every component from scratch |
| Data layer | Use the existing ORM or storage pattern | Invent a temporary data shape |
| Verification | Reuse lint and typecheck | Rely only on visual judgment |

The tomato-timer app is small, but the principle is the same: choose reliable parts first, then let AI connect them. That is where glue coding really begins.

## Unit Summary

- PRDs answer what to build, what not to build, and what success looks like.
- Plans answer what order to build in and how to verify each step.
- Planning is also where you reduce risk by choosing mature building blocks.
