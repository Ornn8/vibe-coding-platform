## Step 1: A Failure Case Shows Why Planning Comes First

Imagine a learner asking, "help me build a blog." The first draft looks fine, then the scope expands, the prompt changes again and again, and after 15 rounds the project is still unstable. The failure is not a single bad prompt. The failure is that nobody defined the target well enough at the start.

That is why planning-driven work starts with failure analysis. The point is to feel the cost of weak planning before learning the fix.

## Step 2: PRDs Answer What to Build

A PRD defines the user, the problem, the core flow, the scope, and the acceptance criteria. It does not explain every implementation detail. Its job is to make sure humans and AI agree on what the delivery should be.

Good PRDs do not need to be long. They need to answer why this exists, who it is for, what the first version includes, and how success will be judged.

Use a simple template:

- Background
- User
- Core flow
- Scope
- Acceptance criteria

For example, a tomato timer PRD might say the user is a student who wants short focus sessions, the core flow is start timer -> break reminder -> history, and the scope excludes login and social sharing.

## Step 3: Implementation Plans Answer How to Build

Once the PRD is clear, the next step is to turn it into a sequence of executable steps. The value of an implementation plan is that it breaks a big goal into smaller pieces and pairs each piece with a verification action.

Planning is not about predicting everything. It is about reducing chaos and making change manageable.

For the same tomato timer example, the plan might split into data model, timer UI, reminder flow, history page, and final verification. Each step should be small enough to complete and check.

## Step 4: Choose Mature Building Blocks

Planning is also where glue coding starts. Once the target is clear, you can choose stable libraries, frameworks, and patterns instead of asking AI to invent everything from scratch.

The goal is not to use the most novel stack. The goal is to reduce risk by connecting dependable pieces around a well-defined core.

For example, you might reuse Next.js, Prisma, and shadcn/ui instead of asking AI to invent your own routing system, database layer, or component library.

## Unit Summary

- PRDs define goals and boundaries; plans define sequence and actions.
- Good plans are broken into small, verifiable steps.
- Separating requirements and plans makes later collaboration easier.
