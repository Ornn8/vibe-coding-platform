<Callout type="methodology" title="Methodology Focus">
Methodology becomes real only when you can say where you used it, how it changed the outcome, and what is still weak.
</Callout>

## Step 1: Review the Four Principles with a Structured Frame
Do not stop at "I learned a lot." A stronger review asks the same three questions for each principle:

| Principle | Where I used it | What I did well | What is still weak |
|---|---|---|---|
| Planning-driven work | Tomato timer or CRUD project | I wrote the PRD before building | My plan still needs finer steps |
| Context stabilization | `CLAUDE.md`, repo rules | I made technical boundaries explicit | Some rules are still only verbal |
| Glue coding | Reused mature components and libraries | I avoided unnecessary reinvention | I still sometimes want to rebuild too much |
| Closed-loop delivery | Verification steps and progress logs | I reduced rework | My Verify habit is not fully consistent yet |

That table matters because it forces each principle to connect to a real behavior, not just a definition.

## Step 2: The Module Project Ships Results and Process Evidence
The M4 module project should be delivered in two layers:

| Layer | Artifacts | Why it matters |
|---|---|---|
| Result layer | Code, pages, running project | Shows what you built |
| Process layer | PRD, plan, `CLAUDE.md`, `progress.md` | Shows how you built it |

If only the result exists, no one can tell whether the work came from a repeatable method or from luck.

```text
Before: I only need to submit something that runs.
After: I need to submit both the result and the evidence of the workflow that produced it.
```

<Exercise id="ex-4.5.2.1" level="L2" title="Audit your delivery package by layer" description="Review your project package through the result layer and process layer, and identify which type of evidence is still missing." />

## Step 3: Use the Review to Prepare for M5 and Beyond
From M5 onward, the work becomes more execution-heavy, debug-heavy, and collaboration-heavy. If M1-M4 did not build a stable base, later complexity will only amplify the chaos. If you already have the habit of writing the target first, storing rules, choosing mature pieces, and keeping verification logs, the next stages become much more natural.

A practical self-check looks like this:

| Self-check question | Yes / No |
|---|---|
| Do I write a PRD or at least a minimum requirement summary before starting? |  |
| Do I store long-running rules in system files instead of repeating them verbally? |  |
| Do I look for mature building blocks before asking AI to invent new ones? |  |
| Do I keep verification and progress records? |  |

<Callout type="scenario" title="How to know whether you are ready for the next stage">
Situation: you can already build pages, but you are unsure whether you are ready for more complex project work. Action: check whether you already work in the rhythm of plan, execute, verify, and record. Why: later complexity rewards stable workflow, not improvisation.
</Callout>
