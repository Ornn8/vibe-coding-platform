## Step 1: Prepare the Documents, Then Start with the Data Model
Before the build starts, prepare at least the PRD, the implementation plan, and `CLAUDE.md`.

| Document | What it answers |
|---|---|
| PRD | What the project is supposed to deliver |
| Plan | What order the work should follow |
| `CLAUDE.md` | What long-running rules must stay in place |

Then begin with the **data model**, not the page, because the data layer defines the CRUD skeleton.

| Phase | What this round should achieve | How to verify it |
|---|---|---|
| Data model | Define fields, relationships, and the minimum storage shape | The schema makes sense and the migration runs |

## Step 2: Build the Backend API Before You Touch the Page
Once the model is clear, move to the **backend API**. In a note-taking CRUD app, this phase only asks whether create, read, update, and delete can work at the data boundary.

| Phase | What this round should achieve | How to verify it |
|---|---|---|
| Backend API | Get the create, read, update, and delete flows working | Confirm responses through requests, logs, or simple checks |

```text
Before: build the whole CRUD app in one round.
After: define the data model, make the API work, build the pages, then polish the interactions.
```

This phase matters because "data works" should be validated separately from "the UI looks right."

<Separator />

## Step 3: Build the Frontend Pages, Then Add Interaction Details
Only after the data and API layers feel stable should you move to the **frontend pages** and the **interaction layer**.

| Phase | What this round should achieve | How to verify it |
|---|---|---|
| Frontend pages | Render list, detail, and form views | The pages display real data correctly |
| Interaction polish | Add loading states, validation, delete confirmation, and error feedback | Real user actions produce the expected feedback |

That is why the content plan insists on four explicit CRUD stages. Later phases touch UI, data, and behavior at the same time, so step-skipping becomes even more expensive.

<Callout type="scenario" title="Why a visible page is not enough">
Situation: the list page already looks correct, so you want to keep polishing the UI. Action: stop and verify whether the page is showing the right data first. Why: "looks right" is not the same as "the data flow is right."
</Callout>

<Exercise id="ex-4.4.3.1" level="L2" title="Separate your frontend phase from your interaction phase" description="For your CRUD project, define the goal, boundary, and verification action for the frontend-page phase and the interaction-polish phase separately." />

<Separator />

## Step 4: Use a Memory Log to Keep the Workflow Coherent
After the project reaches its third phase, both humans and AI start forgetting what was already decided. That is when `progress.md` or a memory bank becomes essential.

| Field | What it should capture |
|---|---|
| Done | Which of the four phases are completed |
| Verified | What commands or manual checks have already run |
| Current state | Where the project is stuck or paused |
| Next step | What should happen in the next round |

```text
Done: note model, list endpoint, create endpoint
Verified: API responses are correct, the list page renders real data
Current state: the edit form is not connected yet
Next step: add the edit endpoint and verify form submission
```

<Callout type="scenario" title="No memory log vs visible progress">
Wrong move: rely on old chat messages to remember the project state. Better move: keep a progress log with done, verified, current, and next. The first loses context quickly. The second gives the workflow durable memory.
</Callout>

## Unit Summary

- CRUD should still move in four phases: data model, backend, frontend, and interactions.
- Each phase needs its own verification, instead of passing problems forward.
- `progress.md` or a memory bank gives the project a durable collaboration memory.
