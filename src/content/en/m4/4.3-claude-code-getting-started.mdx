CLI tools are not just "more advanced chat." They are environments where context, plans, and verification matter much more explicitly.
</Callout>

## Step 1: Install It and Enter the Project Correctly
The first setup path depends on the operating system:

| System | What you usually do first | Most common issue |
|---|---|---|
| Windows | Open PowerShell or Terminal and confirm Node, npm, and PATH | Path or permission issues |
| macOS | Open Terminal and confirm shell access and directory permissions | Terminal and folder access issues |
| Linux | Check shell, Node environment, and repo permissions | Missing dependencies or permission problems |

Once the tool opens in the project directory, do not rush into feature work. First let it inspect the repo, read `AGENTS.md`, read `CLAUDE.md`, and orient itself. The first run is more like "read the drawings before building."

<Callout type="scenario" title="What the first prompt should do">
Situation: the tool is open inside the project directory. Action: ask it to inspect the repository structure, key constraint files, and the stack before making any change. Why: without shared context, a terminal agent is much more likely to act in ways that do not fit the repo.
</Callout>

## Step 2: Learn the First 'Analyze, Do Not Edit' Prompt
A stronger kickoff prompt often looks like this:

```text
Read AGENTS.md, CLAUDE.md, and README first.
Summarize the repository stack, key directories, and main constraints.
Do not edit any files yet. Only give me a short analysis.
```

That prompt changes the agent from "executor" into "analyst" before any risk is introduced.

| Prompt style | Risk | Better alternative |
|---|---|---|
| Fix the homepage | The agent may act before it understands context | Read the constraint files and summarize the repo first |
| Look at this project | Too vague | Ask for stack, key directories, and main constraints explicitly |

## Step 3: Build Trust with One Small Complete Task
The first real execution task should be small enough to finish in 15 to 30 minutes.

| Small task | Why it works well for a first run |
|---|---|
| Change an empty-state sentence | Low risk and easy to verify |
| Rewrite one button label | Small blast radius and clear before/after |
| Add one short helper paragraph | Lets you run analysis -> plan -> edit -> verify once end to end |

Also remember the most common troubleshooting checks:

| Problem | What to inspect first |
|---|---|
| Command not found | Whether Node, npm, and the tool are on PATH |
| Cannot read the repo | Current directory and folder permissions |
| Unsure whether the change is safe | Run `lint`, `tsc`, or a manual check before trusting it |

<Callout type="scenario" title="Large first task vs small first loop">
Wrong move: ask Claude Code to refactor a large page on the first run. Better move: complete one small task through analyze, plan, edit, and verify. One approach gambles with the repository. The other builds collaboration trust.
</Callout>

## Unit Summary

- Setup success depends on PATH, permissions, dependencies, and project location.
- The first repo interaction should be analysis-first, not edit-first.
- Small tasks are the safest way to build early trust in a terminal agent workflow.
