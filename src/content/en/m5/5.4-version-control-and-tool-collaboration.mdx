<Callout title="Method Focus" tone="warning">
Version control is not a bonus skill around AI coding. It is the minimum safety system that makes fast iteration recoverable, comparable, and reviewable.
</Callout>

## Step 1: Start with a real "AI broke my code" story
Imagine a course homepage that already works. You ask AI for three quick rounds of edits:

1. Update the hero copy.
2. Clean up the navigation.
3. Make the layout feel more spacious.

After the third round, the page turns white. You know something broke, but you do not know whether the bug came from the first, second, or third edit.

| State | What you have in your hands | What happens next |
|---|---|---|
| No checkpoints | A vague memory of what changed | You guess, retrace, and lose time |
| One commit after every verified step | A clean version history | You can compare and isolate the exact bad change |

```text
Before: I let AI make three rounds of edits without saving a stable point.
After: I commit after every meaningful, verified step before asking for the next round.
```

Git matters here because it turns "I think it broke later" into "this specific commit introduced the break."

<Callout title="Scenario: when you should commit before continuing" tone="warning">
Situation: the current feature passes the page check and your terminal validation, and you are about to ask AI for the next edit. Action: commit the stable state first. Why: once AI starts the next round, you need a clean recovery point instead of relying on memory.
</Callout>

## Step 2: Choose tools by workflow, not by habit
Claude Code, Codex, and GUI tools are not enemies. They are better at different rhythms of work.

| Tool | Startup cost | Best project stage | Typical job | Example command or action |
|---|---|---|---|---|
| Claude Code | Low if you are already in the repo | Live implementation and debugging | Read code, change code, verify, iterate | `npm run lint`, inspect files, patch a bug |
| Codex | Medium because the task must be scoped clearly | Well-bounded async execution | Rewrite a contained module or produce a spec-driven change | Hand off a clean task with file scope and expected output |
| GUI prototyping tools | Low for visual exploration | Earliest concept or layout stage | Try interface directions quickly | Prompt a landing page or dashboard mock |

The wrong question is "Which tool do I like most?" The better question is "What rhythm does this task need right now?"

<Callout title="Scenario: one task, three different tool fits" tone="info">
Situation: you need to redesign a lesson page, debug a broken form, and bulk-rewrite several structured docs. Action: use a GUI tool to explore the page direction, use Claude Code for the broken form, and use Codex for the contained document rewrite. Why: the tasks have different feedback loops, so the safest tool also changes.
</Callout>

## Step 3: The real goal is a smoother closed loop
Good collaboration is not "use more tools." It is "reduce handoff friction while keeping the work reversible."

| Scenario | Smoother sequence |
|---|---|
| Try a visual direction fast | GUI tool -> confirm direction -> move into repo |
| Make a real repo change safely | Claude Code -> verify -> Git checkpoint |
| Ship a bounded background task | Write the spec -> Codex executes -> human review -> checkpoint |

If you send a live debugging task to an async tool, the loop slows down. If you keep a large, well-scoped batch rewrite inside an overly chatty loop, the context gets noisy. Tool choice is part of delivery design.

<Callout title="before / after: choosing by preference vs choosing by workflow" tone="info">
Wrong move: use the same favorite tool for every job. Better move: decide whether the current work is exploratory, interactive, or asynchronous, then choose the tool that matches that pace. Preference optimizes comfort. Workflow fit optimizes delivery.
</Callout>

## Summary
- Git checkpoints are the safety net that makes AI collaboration reversible.
- Claude Code, Codex, and GUI tools each fit different work rhythms.
- Tool choice is part of closed-loop delivery because it changes how easy the next verification step becomes.

## Methodology Review
Closed-loop delivery adds one more discipline here: not just "can we move fast," but "can we recover, compare, and continue safely after every fast step?"
