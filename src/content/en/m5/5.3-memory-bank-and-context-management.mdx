<Callout title="Method Focus" tone="warning">
Context management is not mainly about sentence phrasing. It is about where project memory lives and how the assistant re-reads it.
</Callout>

## Step 1: Why AI Forgets What Happened Yesterday
AI chat context is like a limited whiteboard. As the thread grows, old constraints and earlier decisions get pushed off the surface. That is why long projects cannot rely on chat history alone.

<Callout title="Scenario: when forgetting is most likely" tone="warning">
Situation: you reopen the project two days later and expect AI to continue from step six. Action: do not assume it still remembers the constraint history. Give it readable project memory files first. Why: session changes and time gaps are where forgetting shows up most often.
</Callout>

| Situation | What AI is most likely to forget |
|---|---|
| Same-day short thread | Some early details, but the main goal may still hold |
| New session on the next day | Constraints, current stage, and completed work |
| Many tasks mixed in one conversation | What the current work actually is |

## Step 2: Look at a Real `progress.md`
Here is a simple but genuinely useful `progress.md` example:

```md
# Progress

## Done
- Built the course module list page
- Connected MDX rendering on the lesson page
- `npm run lint` passes

## Current State
- Working on removing the lesson-page bottom exercise entry
- Module-project submission flow has not been changed yet

## Verified
- The `/zh/modules/m1/...` lesson page renders correctly
- `npx tsc --noEmit` passes

## Next
- Rewrite the Chinese M1-M2 content
- Sync the English versions
```

| Section | Why it helps |
|---|---|
| Done | Prevents AI from repeating finished work |
| Current State | Shows where the project is paused |
| Verified | Shows what is already trustworthy |
| Next | Gives AI a clean place to resume from |

## Step 3: `context.md` and `decisions.md` Solve Different Problems
`progress.md` alone is not enough. You usually also need:

| File | Main question it answers |
|---|---|
| `context.md` | What kind of project this is and why it exists |
| `decisions.md` | Why one approach was chosen over another |
| `progress.md` | Where the work is now and what comes next |

All three together form the useful core of a memory bank. Otherwise you may know what step you are on without knowing why the project is shaped this way.

<Callout title="before / after: progress without decision context" tone="info">
Wrong move: only record what step the project has reached. Better move: also record the project background and the key decisions that shaped it. One restores activity. The other restores judgment.
</Callout>

<Exercise id="ex-5.3.3" level="L3" title="Create a minimum viable memory bank" description="In a real project, create `context.md`, `decisions.md`, and `progress.md`, then use them to restore one new session by having AI read the files before continuing." />

## Summary
- Long projects cannot rely on chat memory alone.
- `progress.md` stores state, `context.md` stores background, and `decisions.md` stores judgment.
- A real memory bank helps AI restore both project status and the reasoning behind it.

## Methodology Review
This is context stabilization as a file-driven workflow: not "remember it," but "write it down and re-read it."
