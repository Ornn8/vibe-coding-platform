<Callout title="Method Focus" tone="warning">
Fast debugging does not start with guessing. It starts with structuring evidence so AI can work on facts instead of on your frustration.
</Callout>

## Step 1: Runtime Bugs Need an Error Chain, Not Emotion
Start with the classic runtime bug loop:

| Part of the loop | Content |
|---|---|
| Error signal | Clicking Save returns a 500 |
| Evidence | Console shows `Cannot read properties of undefined` |
| Your hypothesis | A missing null guard on one form field |
| What you send to AI | The error message, trigger action, and relevant code snippet |
| Verification after the fix | Click Save again and confirm the 500 is gone |

If you only say "the page is broken," AI can only guess. If you send the error, the trigger, and the expected behavior together, AI can narrow the search space immediately.

## Step 2: Visual Bugs Need Screenshots and a Clear Target State
Visual bugs are different. They usually need the current look and the intended look.

| Part of the loop | Content |
|---|---|
| Current state | The primary button wraps to a second line on mobile and cards are misaligned |
| Evidence | A screenshot at the broken width |
| Expected state | The button should stay on one line and align with the title baseline |
| What you send to AI | Screenshot, device or width information, and the expected layout |
| Verification after the fix | Re-open the same width and compare again |

```text
Before: This page looks a little off.
After: At iPhone width, the primary button wraps onto a second line. I want it to stay on one line and align with the card title.
```

<Callout title="Scenario: why visual debugging hates abstraction" tone="warning">
Situation: you know the page looks wrong, but you cannot describe the deviation clearly. Action: capture the screenshot first, then write which element is wrong, on which viewport, and how it should look instead. Why: without a visual reference and an explicit target, AI drifts badly on visual fixes.
</Callout>

## Step 3: Logic Bugs Need Expected vs Actual Behavior
Logic bugs are often the hardest to see because the page may still render. For example, "delete succeeds but the list does not refresh" or "the completion counter increases twice."

| Part of the loop | Content |
|---|---|
| Expected behavior | After deleting one item, the list should show one fewer item |
| Actual behavior | The success toast appears, but the list stays unchanged |
| What you send to AI | Expected vs actual, trigger steps, and any important state information |
| Verification after the fix | Repeat the same interaction and compare again |

If you only say "the logic is wrong," AI has no way to know whether the issue is data flow, state updates, or permissions.

<Exercise id="ex-5.2.3" level="L3" title="Document one structured logic-debug loop" description="Choose one real bug, capture the evidence, write your hypothesis, record AI's repair suggestion, and verify the final result with an explicit expected-vs-actual comparison." />

## Summary
- Runtime bugs need the error chain.
- Visual bugs need screenshots plus the intended state.
- Logic bugs need an explicit expected-vs-actual comparison.

## Methodology Review
Closed-loop delivery also applies to failure handling: locate, repair, and verify, instead of trusting a fix that merely looks plausible.
