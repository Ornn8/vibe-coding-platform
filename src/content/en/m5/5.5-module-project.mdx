<Callout title="Method Focus" tone="warning">
The real challenge in iterative delivery is not adding more features. It is making each new capability land in dependency order without breaking the base that already works.
</Callout>

## Step 1: List the changes before you touch the code
This project is not a new app. It is the M4 CRUD foundation plus another round of capability, such as authentication, data migration, permission boundaries, or interaction improvements.

| Change item | What it depends on | Where the risk lives |
|---|---|---|
| Add authentication | User model, session handling, route protection | Broken login flow or wrong access control |
| Migrate data | Existing schema and compatibility plan | Regressions in current features |
| Protect admin-only pages | Reliable auth state and role data | Users seeing the wrong page or getting blocked incorrectly |

If you skip this list, "add one feature" quickly turns into "accidentally redesign the whole app."

<Callout title="Scenario: why dependency mapping comes before implementation order" tone="info">
Situation: you want to build the login screen first because it is the most visible task. Action: check whether it depends on user data, roles, or route protection that do not exist yet. Why: visibility does not decide order. Dependency does.
</Callout>

## Step 2: Write an iteration plan in dependency order
The same goal can have very different safe sequences.

| Situation | Better order |
|---|---|
| Auth needs new user fields | Migrate the data model first, then connect auth |
| Route protection depends on stable login state | Finish auth flow first, then guard the pages |
| A new form depends on permissions | Define the permission rule first, then expose the form |

```text
Before: I order the work by what feels exciting.
After: I fix the lowest-level dependency first, then build the page or interaction that relies on it.
```

This is what keeps iteration cheap. If the order is wrong, every later step becomes more fragile and more expensive to rework.

## Step 3: Submit evidence, not only a result
The final project package should prove both the app result and the delivery method.

| Delivery artifact | Why it must be included |
|---|---|
| Code result | Proves what changed |
| Git history and checkpoints | Proves the iteration path is traceable |
| Memory bank | Proves the work can resume across sessions |
| Debug log | Proves you closed the loop when things went wrong |

If you submit only code, other people can see the result. They cannot see whether the delivery process was controlled or just lucky.

<Callout title="before / after: shipping code vs shipping an evidence chain" tone="warning">
Wrong move: submit only the working repository. Better move: submit the repository, checkpoints, memory files, and structured debug record together. One shows the outcome. The other shows that you can deliver reliably more than once.
</Callout>

## Summary
- The module project is about extending an existing app, not rebuilding it.
- Dependency order should control the iteration plan.
- A complete submission shows both the product result and the evidence chain behind it.

## Methodology Review
Closed-loop delivery enters a more mature phase here: not just building a feature, but proving that the feature was added in a stable, inspectable, and repeatable way.
