## Step 1: Understand the project change in detail

The curriculum says to add authentication and a database migration on top of the existing M4 project. This is an iteration, not a rewrite. The goal is to add capability to a working codebase, not to start over because the new task looks larger than the old one.

### Scenario

Before:
- A CRUD app with no login and no persistent user state

After:
- The same app, but now with authentication and a database change that supports user data

The difference matters because the project must remain recognizable while it grows.

## Step 2: Design the iteration plan by dependency order

The curriculum's dependency question is explicit: should authentication come first or should the database change come first? The answer is not random. You order the work by dependency, so each step can build on a valid base.

### Scenario

If user records must be stored before the auth flow can be trusted, then the database migration must happen before the final auth behavior. If auth needs to be tested before the data model changes, you sequence the work differently. The point is to reason about the dependency, not the vibe.

## Step 3: Submit the full delivery record

The curriculum requires more than code. You must submit the repository history, the memory-bank files, at least two checkpoints, and a complete debug log. That submission proves the process was controlled, not accidental.

### Scenario

A good submission lets another person understand:
- what changed
- why it changed
- how the change was verified
- where the project state is now

## Summary
- Iteration is about adding capability to an existing system.
- Dependency order should drive the plan, not intuition alone.
- The submission should prove both the result and the process.

## Methodology Review
This unit combines closed-loop delivery with context stabilization: the app changes, and the record of the change stays recoverable.
