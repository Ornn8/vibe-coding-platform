<Callout type="methodology" title="Methodology Focus">
Closed-loop delivery starts in the requirement stage. A journey map is the bridge between user problems and implementation paths.
</Callout>

## Step 1: Draw the journey before listing features

Take a reading-club signup page as an example. Do not begin with page names such as home, signup form, or FAQ. Begin with the user's action chain.

| User action | What the user sees | What the user feels | Design implication |
|---|---|---|---|
| Opens a shared link | A hero section that explains the reading club and event time | I need to know if this is worth my time | The first screen must explain value and schedule fast |
| Scrolls to learn more | Highlights, flow, speaker info | I need to know whether this fits me | Order the content from value to detail |
| Decides to sign up | A clear CTA and short form | I want the action to be easy | The CTA must be visible and the form must stay short |
| Finishes signup | Confirmation and next-step guidance | I want certainty about what happens next | The success state must feel final and clear |

<Callout type="scenario" title="Scenario: Why the journey map beats a feature list">
Situation: You listed many features, but you still do not know what belongs on the first screen.

Action: Start with what the user does, sees, and feels.

Why: AI understands a user flow better than a random list of pages.
</Callout>

## Step 2: Turn the journey into requirement text

When you rewrite the journey as requirements, include at least the target user, core flow, boundary, and exclusions.

```text
Build a signup page for first-time reading-club participants.
The user should first understand the value and time of the event, then view the flow and speaker info, and finally complete signup.
This first version only includes the event intro, flow explanation, and signup form.
Do not add payments, comments, or community features.
```

| Requirement part | Minimum level of clarity |
|---|---|
| User | Who opens the page |
| Flow | The key steps from open to completion |
| Boundary | What sections or pages belong in this round |
| Exclusions | What must not be built yet |

<Callout type="warning" title="Before / After: Vague requirement vs executable requirement">
Wrong: Build a better reading-club page with more complete features.

Right: Build a signup page for first-time reading-club participants. Include event intro, flow, and signup form only. Do not add payments or community features.

Difference: The first version has no user and no boundary. The second version can be executed.
</Callout>

## Step 3: Spot the most common requirement mistakes

The most common mistakes are: too vague, too greedy, outcome-only with no constraints, talking about technology before the user problem, and mixing future ideas into the first version.

| Mistake type | Typical example | How to fix it |
|---|---|---|
| Too vague | Build a better expense app | Add the user and the core flow |
| Too greedy | First version includes auth, social, and rankings | Shrink to P0 only |
| Missing constraints | Build an event page | Add user, flow, and exclusions |
| Technology first | Use the latest framework to build a system | Start with the user problem |
| Future ideas mixed in | Build phase two features now as well | Separate first version from later ideas |

<Callout type="scenario" title="Scenario: One very common bad requirement">
Bad requirement: Build a better expense app with lots of complete features and a premium design.

Why it is bad: There is no user, no flow, no boundary, and no clear success criterion.

How to improve it: Add who it is for, what the key action is, and what is explicitly out of scope.
</Callout>

## Unit Summary

- Journey maps are closer to real user behavior than feature lists.
- Good requirement text must include user, flow, and boundary.
- Requirement mistakes are cheapest to fix early.

**Methodology Review**: Closed-loop delivery starts earlier than shipping. It starts with executable requirements.
