## Step 1: MVP Is the Shortest Path to Validation

An MVP is not a smaller version of everything. It is the shortest path to proving or disproving the core hypothesis. If the first release cannot validate anything, it is not really an MVP.

This is where the planning habit changes from "what could we build?" to "what do we need to learn first?"

One practical test: if removing a feature does not affect your ability to learn whether the idea works, that feature is probably not part of the MVP.

## Step 2: Use P0, P1, and P2

A useful prioritization system keeps the first release honest. P0 means the product cannot function without it. P1 means the experience gets better but the product can still ship without it. P2 means it is nice to have, but not necessary yet.

If you have too many P0 items, the scope is probably too big. If everything is P2, the project probably has no real core yet.

## Step 3: Keep AI Focused on the Core

The biggest enemy of Vibe Coding is a giant pile of features. If you ask AI to do too many things at once, the result often becomes shallow or unstable. The fix is not a better model. The fix is to define what must exist first.

That is why prioritization is part of engineering, not an afterthought.

The glue-coding mindset helps here: pick mature building blocks only after the core has been narrowed down. Do not ask AI to invent every part of the stack if an existing module already solves the hard part.

For example, a simple habit tracker might start with "add task," "mark done," and "view today" before it ever touches streaks, sharing, or analytics.
