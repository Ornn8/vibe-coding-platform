<Callout type="methodology" title="Methodology Focus">
An MVP is not a shrunken product. It is the shortest path to validating the core assumption.
</Callout>

## Step 1: An MVP validates a core assumption

If your product is an expense app, the assumption may be "users will open the tool every day to record one expense quickly." If your product is a blog system, the assumption may be "authors want a lightweight writing and publishing flow." The MVP should validate that assumption, not attempt to complete the whole product universe.

<Callout type="scenario" title="Scenario: Two products, two different MVPs">
Expense app: The first version only needs fast entry and a same-day list review.

Blog system: The first version only needs writing, publishing, and reading a post.

Why: The product types are different, so the shortest validation path is different too.
</Callout>

| Product type | Core assumption | MVP-like first version | Easy feature trap |
|---|---|---|---|
| Expense app | Users will record spending quickly | Add a record and review today's list | Charts, points, themes |
| Blog system | Authors want a smooth publish flow | Write, publish, and read a post | Comments, tag clouds, analytics |

## Step 2: Use P0, P1, and P2 to cut scope

P0 means the feature is required to validate the assumption. P1 improves the experience but can wait. P2 is optional polish.

| Priority | Expense app example | Blog system example |
|---|---|---|
| P0 | Add expense, review today's entries | Write a post, publish it, read it |
| P1 | Category stats, monthly summary | Drafts, cover image, filtered article list |
| P2 | Theme switcher, points, share card | Comments, likes, ranking panels |

<Callout type="warning" title="Before / After: Stuff everything in vs only build P0">
Wrong: The first version includes charts, points, sharing, and achievements.

Right: The first version only supports adding one record and reviewing today's list.

Difference: The wrong path blurs the validation target. The right path makes the result testable.
</Callout>

## Step 3: Turn P0 into an AI-executable first-version prompt

When you write the first-version prompt, specify both what to build and what not to build.

```text
Build the first version of an expense tracking tool.
Target user: a general user who wants to log daily expenses quickly.
This round only includes: add one record and review today's list.
Do not build: charts, sharing, points, or theme switching.
```

<Callout type="scenario" title="Scenario: Actually compressing the scope">
Situation: You have many ideas and do not want to lose them.

Action: Put P1 and P2 into the "do not build this round" section instead of leaving them implied.

Why: AI is very good at filling in imagined product worlds. Boundaries must be explicit.
</Callout>

| Prompt style | What AI is more likely to do |
|---|---|
| Build a complete expense app | Expand the feature set automatically |
| Build only add-record plus today's list, and exclude charts and sharing | Stay closer to the validation target |

## Unit Summary

- An MVP validates a core assumption, not feature count.
- P0/P1/P2 is the most practical language for scope control.
- A good prompt says both what to build and what not to build.
