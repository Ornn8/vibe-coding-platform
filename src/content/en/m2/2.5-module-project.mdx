<Callout type="methodology" title="Methodology Focus">
This module project asks you to turn product thinking into something executable. The best proof of understanding is a document plus a working MVP.
</Callout>

## Step 1: Write the Requirement Document

Use the soul questions, the MVP scope, and the user journey to create a short requirement document. It should clearly state the user, the pain point, the solution, the scope, and what counts as success.

The document is not for decoration. It is the brief that lets AI and humans stay aligned while building.

Think of it as the bridge between product thinking and implementation: the more concrete the requirement document is, the less guessing the tool has to do later.

<Exercise id="ex-2.5.1.1" level="L3" title="Submit the requirement doc" description="Write a one-page requirement document that combines the soul questions, MVP boundary, and user journey." />

<Separator />

## Step 2: Build the MVP With a GUI Tool

After the requirement is clear, build the first version with a GUI-first tool or other fast prototyping stack. The goal is to create something testable, not something polished enough to be final.

Make sure the build stays close to the original hypothesis. If it starts drifting, the requirement document should help pull it back.

Use the document as a guardrail:

- If the tool adds a feature that does not support the hypothesis, remove it.
- If the design starts looking busy, return to the smallest testable version.
- If the build feels stuck, compare the current output with the requirement and ask what changed.

<Exercise id="ex-2.5.2.1" level="L3" title="Build the MVP" description="Create the first working version and record the key prompts or choices you used." />

<Separator />

## Step 3: Package the Result

The point of this project is not only the output, but also the trace of how you got there. Package the requirement document, the deployment link or screenshots, and a short retrospective that explains what you learned.

That complete package is what makes the work reviewable and reusable later.

<Exercise id="ex-2.5.3.1" level="L3" title="Submit the full package" description="Submit the requirement doc, the MVP link or screenshots, and a short retrospective." />

## Unit Summary

- A good module project turns product thinking into an executable brief and a working MVP.
- The requirement document keeps the build aligned with the hypothesis.
- The final package should include both the result and the process.
