<Callout type="methodology" title="Methodology Focus">
The requirement document is not a side artifact. It is the construction drawing for the MVP.
</Callout>

## Step 1: Write a requirement document that AI can actually use

This document does not need to be long. It does need to be complete. At minimum, include:

- Project background
- Target user
- Core pain
- Core flow
- P0 / P1 / P2 boundaries
- Acceptance criteria

| Section | What question it must answer |
|---|---|
| Background | Why this project exists now |
| User | Who it is for |
| Pain | What inefficiency or frustration the user currently tolerates |
| Flow | What happens from open to completion |
| Scope | What this round includes and excludes |
| Acceptance | What counts as a successful first version |

<Callout type="scenario" title="Scenario: The right length for a first requirement doc">
Situation: You worry that a short document will look unprofessional, but a long one will be unreadable.

Action: Write the strongest one-page version first.

Why: The goal is not formal documentation. The goal is a stable execution input for the next step.
</Callout>

## Step 2: Use the document to drive the first MVP

When you reopen a GUI tool, your first prompt should not begin from memory. It should begin from a requirement summary.

```text
Build the first MVP from this requirement:
Target user: first-time reading-club participants.
Core flow: understand the event value, review the flow and schedule, then sign up.
This round only includes: event intro, process explanation, and signup form.
Do not build: payments, comments, or personal accounts.
Acceptance criterion: the user can understand the event and complete signup within one minute.
```

<Callout type="warning" title="Before / After: Improvised prompting vs document-driven prompting">
Wrong: Open the tool and improvise the request from memory.

Right: Extract a first-version summary directly from the requirement doc and use that as the prompt baseline.

Difference: Improvisation changes direction every round. The document gives every round the same baseline.
</Callout>

| Execution style | What AI is more likely to do |
|---|---|
| Prompt from memory | Drift in structure and scope |
| Prompt from the requirement doc | Stay closer to the intended boundary |

## Step 3: Submit both the result and the process

This module does not ask for a link only. It asks for a project pack:

- Requirement doc
- MVP link or screenshots
- First prompt
- Key iteration notes
- Short retrospective

<Callout type="scenario" title="Scenario: Why process evidence matters">
Situation: Two learners both built a page, but only one can explain how the scope stayed under control.

Action: Keep the requirement doc, prompt log, and iteration evidence together with the final result.

Why: The course is training reviewable delivery, not just visible output.
</Callout>

| Deliverable | Minimum requirement |
|---|---|
| Requirement doc | One-page readable version with boundaries and acceptance criteria |
| MVP result | Link or screenshots that show the core flow |
| Prompt log | First prompt plus at least one boundary-correction round |
| Retrospective | What worked, and where drift still appeared |
