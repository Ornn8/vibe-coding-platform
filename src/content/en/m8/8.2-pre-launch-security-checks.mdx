## Step 1: The most common AI-era security mistakes

Hardcoded secrets, weak input validation, unclear permission boundaries, overexposed errors, and risky query or integration patterns are all common. The danger is that these issues often still "work" until they are deployed.

### Scenario

The curriculum's warning is simple: code that appears functional can still be unsafe. A hardcoded API key, an unvalidated form, or a wide-open permission boundary may not break the app right away, but it can still break the release.

## Step 2: Treat environment and credential management seriously

`.env` should never be committed. Deployment environments need the right variables in the right place. Many developers can build features. Far fewer can handle credentials safely enough to launch.

### Scenario

Before:
- secrets live in source files or in ad hoc notes

After:
- secrets live in environment variables
- `.env.example` shows only safe placeholders
- deployment settings match the runtime environment

## Step 3: Build a repeatable release checklist

You do not need enterprise-grade security on day one, but you do need a repeatable minimum: HTTPS, auth, input validation, error handling, access control, logs, and rollback thinking.

### Scenario

A release checklist matters because it is reusable. The curriculum wants the same safety check to work before every launch, not just when a problem is already visible.

<Exercise id="ex-8.2.3.1" level="L3" title="Run one pre-launch audit" description="Use your launch checklist on a real project and record the issues, risk level, and planned fixes." />

## Summary
- The biggest release risk is often the issue that still seems to work.
- Credentials and permissions are easy to neglect and expensive to neglect.
- Security checklists are valuable because they are repeatable, not because they are long.

This unit applies closed-loop delivery to release safety by turning security checks into explicit, repeatable gates before launch.
