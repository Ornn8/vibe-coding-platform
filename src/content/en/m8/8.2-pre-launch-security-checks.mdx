<Callout title="Method Focus" tone="warning">
AI can help you ship risk faster as easily as it helps you ship features faster. Security checks are part of responsible delivery, not optional polish.
</Callout>

## Step 1: Compare vulnerable code with fixed code
Here is a hardcoded-secret mistake:

```ts
const stripe = new Stripe("sk_live_123");
```

And the safer version:

```ts
const stripeKey = process.env.STRIPE_SECRET_KEY;

if (!stripeKey) {
  throw new Error("Missing STRIPE_SECRET_KEY");
}

const stripe = new Stripe(stripeKey);
```

Here is an unsafe permission boundary:

```ts
export async function GET() {
  return Response.json(await prisma.review.findMany());
}
```

And the safer version:

```ts
export async function GET() {
  const session = await auth();

  if (!session?.user || session.user.role !== "ADMIN") {
    return Response.json({ error: "Forbidden" }, { status: 403 });
  }

  return Response.json(await prisma.review.findMany());
}
```

| Risk type | Vulnerable pattern | Fixed pattern |
|---|---|---|
| Secrets | Hardcoded key in source | Read from env and fail safely |
| Permissions | Open access to sensitive data | Explicit auth and role check |

## Step 2: Treat credential and input handling as first-class release work
Another common AI-era mistake is trusting input too early.

```ts
await prisma.post.create({ data: req.body });
```

Safer:

```ts
const parsed = postSchema.safeParse(await req.json());

if (!parsed.success) {
  return Response.json({ error: "Invalid input" }, { status: 400 });
}

await prisma.post.create({ data: parsed.data });
```

| Review item | What to verify before launch |
|---|---|
| Secrets | No real secret is committed or hardcoded |
| `.env` handling | `.env` is ignored and `.env.example` contains placeholders only |
| Input validation | External input is parsed before use |
| Error handling | Sensitive implementation details are not leaked to users |

<Callout title="Scenario: why working code can still be release-unsafe" tone="info">
Situation: the form submits successfully in local development. Action: check how the app handles malformed input, missing secrets, and unauthorized access before shipping. Why: many unsafe paths never show up in the happy-path demo.
</Callout>

## Step 3: Turn security into a repeatable launch checklist
The point is not enterprise perfection on day one. The point is to make the same minimum safety review repeatable.

| Checklist item | Why it belongs before launch |
|---|---|
| Secrets are externalized | Prevents credential leakage |
| Sensitive routes have auth boundaries | Prevents accidental data exposure |
| Inputs are validated | Prevents unsafe writes and crashes |
| Logs and error messages are reviewed | Prevents overexposed internals |

```text
Before: before launch, I only checked whether the page opened and the feature seemed to work.
After: I also inspect secrets, input validation, permission boundaries, and error exposure before deciding the release is actually safe enough.
```

<Exercise id="ex-8.2.3" level="L3" title="Run one pre-launch audit on a real project" description="Take a real project, run a launch-security checklist against it, and record the issues you found, their risk level, and which ones must be fixed before release." />

## Summary
- Security becomes easier to learn when you compare vulnerable code directly with the safer version.
- Secret handling, auth boundaries, and input validation should be checked before launch, not after.
- A short repeatable checklist is more valuable than a long security lecture.

## Methodology Review
Closed-loop delivery now includes release safety gates: not just "does it work," but "is it safe enough to put in front of real users?"
