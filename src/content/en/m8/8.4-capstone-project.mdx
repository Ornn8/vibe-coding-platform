## Step 1: Choose a topic that is real, bounded, and demoable

A strong capstone does not need to be huge, but it does need to be complete. It should demonstrate the full path from requirements and plans through context setup, implementation, verification, deployment, and review.

### Scenario

The curriculum gives you a selection problem, not a blank page. Pick a topic that can be explained clearly, finished in scope, and demonstrated in a short, convincing session.

Use three questions to narrow the topic:
- Who is it for?
- What problem does it solve?
- How will you demo it clearly?

## Step 2: The deliverable is more than code

The final package should include the repository, deployment link, PRD or planning docs, CLAUDE.md, Skills, memory-bank or progress records, retrospective, and demo materials. Those artifacts show that the project was built methodically rather than accidentally.

### Scenario

The curriculum's deliverable list makes process visible:
- code shows what was built
- plans show how it was designed
- context files show how it stayed consistent
- retrospective shows what should improve next

<Separator />

## Step 3: The retrospective determines whether you really leveled up

After the build is done, the most valuable part is often the review. What methods worked best? Where did rework accumulate? Which rules should become permanent assets for the next project?

### Scenario

A capstone only becomes growth when the review changes the next project. If the same mistakes would happen again, the capstone is not yet a learning artifact.

- The capstone measures full-loop delivery, not isolated technical polish.
