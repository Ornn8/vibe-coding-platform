<Callout title="Method Focus" tone="success">
Strong builders do not only finish projects. They convert each project into reusable planning, execution, and review assets for the next one.
</Callout>

## Step 1: Look at a real `feature-implementation.md`
Here is a concrete example:

```md
# Feature: Review Notes on Project Submissions

## Goal
Allow reviewers to attach notes to each submission so students can see actionable feedback.

## Scope
- Add review-note field to reviewer workflow
- Show the note on the student project page
- Do not redesign the whole review dashboard

## Acceptance
- Reviewer can save one note
- Student can read the latest note
- `npm run lint` and `npx tsc --noEmit` pass

## Risks
- Notes may expose admin-only wording to students
- Existing review flow must remain unchanged

## Verification
- Test reviewer save path
- Test student visibility path
- Confirm old submissions still render safely
```

| Section | Why it matters |
|---|---|
| Goal | Keeps the feature tied to one real outcome |
| Scope | Prevents side quests and redesign drift |
| Acceptance | Makes verification concrete |
| Risks | Forces you to surface hidden failure modes |
| Verification | Defines how the feature earns trust |

## Step 2: Turn project artifacts into a personal knowledge system
By now you have accumulated prompts, feature docs, Skills, templates, and retrospectives.

| Asset type | Good home in your system |
|---|---|
| Feature docs | `/features/` or project planning folder |
| Skills | `/skills/` or reusable workflow library |
| Retrospectives | `/retrospectives/` or project-review archive |
| Reusable prompts and templates | `/templates/` or playbook folder |

If these stay buried in chats or scattered folders, they do not compound.

<Callout title="Scenario Card: when one iteration is already worth preserving as an asset" tone="info">
Situation: you just finished a feature and still remember which prompts worked, which checks were easy to forget, and where rework happened. Action: turn that fresh knowledge into a feature doc, checklist, or Skill draft immediately. Why: iteration knowledge is easiest to capture right after the work ends and easiest to lose a few days later.
</Callout>

```text
Before: useful prompts and plans stayed trapped inside old conversations.
After: the assets are stored by type, so the next project can start from reusable material instead of memory.
```

## Step 3: Mature builders know where vibe coding stops fitting
Not every project should be built with the same level of looseness.

| Project type | Vibe coding fit | Why |
|---|---|---|
| Landing page prototype | High | Fast iteration is low risk |
| Internal ops dashboard | Medium | Speed helps, but data boundaries still matter |
| Medical or financial workflow | Low | Errors carry high real-world cost |

That boundary is part of maturity. The question is not "Can AI generate it?" It is "Should this workflow be allowed to rely on fast, loose iteration?"

## Summary
- A full feature document is a practical planning asset, not paperwork.
- Knowledge compounds only when prompts, docs, Skills, and retrospectives are stored systematically.
- Mature builders learn where vibe coding should stop and stricter engineering should begin.

Context stabilization becomes a long-term personal system here: each project should leave behind reusable planning and review assets instead of disappearing into old chats.
