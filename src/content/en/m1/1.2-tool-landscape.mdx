<Callout type="methodology" title="Methodology Focus">
Do not chase the strongest tool. Match tool complexity to task complexity.
</Callout>

## Step 1: Why Browser Tools Are Great for Starting
Open `bolt.new` or `lovable.dev` and you usually see one large input box and a preview area. Type:

```text
Build a cafe menu page with latte, americano, and cappuccino.
Show prices, categories, and a pickup button. Keep it modern and clean.
```

In under a minute, you often get a full page in the preview. That "open it, ask once, see something real" loop is the biggest strength of browser tools. They are great for landing pages, small demos, and first MVP validation.

<Callout type="scenario" title="When browser tools should win">
Situation: you are still validating whether a page direction is worth pursuing. Action: start with a browser tool and get a visible draft. Why: at this stage you need low setup cost and fast feedback more than deep engineering control.
</Callout>

| Tool category | Setup cost | Time to first use | Best project size | Typical jobs |
|---|---|---|---|---|
| Browser tool | Low, just sign in | 1-3 minutes | Single pages, demos, first MVPs | Landing pages, personal sites, quick concept pages |
| IDE tool | Medium, local setup needed | 5-15 minutes | Real codebases, small to medium projects | Edit real files, reuse components |
| CLI tool | Medium to high, terminal and repo context needed | 10-20 minutes | Multi-step delivery, verified project work | Plan, execute, review, verify |

## Step 2: What IDE and CLI Tools Change
Once you move from "try an idea" to "keep changing real work," browser tools become less stable. If you need to inspect files, reuse existing components, control dependencies, or run checks, IDE and CLI tools become much more useful.

Cursor and Windsurf feel like "editors that can talk back." You read code while asking AI to modify code. Claude Code and Codex feel more like "terminal partners that can operate a workflow." They do not just edit. They read plans, inspect repos, run checks, and summarize results.

```text
Before: IDE and CLI both felt like "more advanced AI tools."
After: I can choose them by task. If I need visual file editing, IDE makes sense. If I need plan-execute-verify delivery, CLI is usually stronger.
```

| Task | Browser tool | IDE tool | CLI tool |
|---|---|---|---|
| First draft of a course landing page | Best speed | Usable but heavier | Too heavy |
| Add a page inside an existing Next.js repo | Weak | Strong fit | Strong fit |
| Build CRUD in four verified steps | Poor fit | Possible | Best fit |
| Run `npm run lint` and `npx tsc --noEmit` and fix issues | Weak | Possible | Strong fit |

## Step 3: Make a Real First-Tool Decision
Three questions are enough. First, am I validating an idea or editing a real project? Second, do I need AI to understand the whole repository? Third, do I need commands, checks, or step-by-step execution right now?

If the answer is "I am validating an idea, I do not need repo context, and I do not need commands," start with a browser tool. If you already need code control, dependency control, or verification, do not force a browser tool to do a CLI job.

<Callout type="scenario" title="What picking the wrong tool feels like">
Wrong move: choose the terminal first just to feel professional. Better move: start with the lightest tool that can validate the task, then upgrade only when the complexity truly rises. Let the task drive the tool, not the other way around.
</Callout>

| Current state | Recommended starting tool | Your next move |
|---|---|---|
| I only want to test one page idea | Browser tool | Generate a first draft today |
| I already have a local project and want to edit files | IDE tool | Open the repo and make a targeted change |
| I need planning, execution, and verification | CLI tool | Read context first, then work in steps |

## Unit Summary

- Browser tools win on low friction and fast feedback.
- IDE tools are best when you want to keep editing real code with visual context.
- CLI tools are strongest when planning, execution, and verification all matter.
