<Callout type="methodology" title="Methodology Focus">
Even a tiny page deserves a tiny plan. Write the target, the structure, and the style before you ask for the first draft.
</Callout>

## Step 1: Generate a First Draft You Can Actually Show
Open your chosen browser-based AI tool, but do not type immediately. First write a three-line mini plan:

```text
Audience: future employers and collaborators
Core information: who I am, what I can do, what I recently built
Structure: Hero, skills section, project showcase, contact section
```

Then turn that into a prompt:

```text
Please generate a personal introduction page.
Audience: future employers and collaborators.
Core information: intro, skills, recent projects, contact details.
Style: clean, modern, mobile-first, plenty of whitespace.
Structure: Hero, skill cards, project showcase, contact section.
```

You usually see a loading state first, then a real page in the preview. The key check is not whether it looks impressive. The key check is whether it follows the three-line plan you wrote first.

<Callout type="scenario" title="What matters in round one">
Situation: the first page already looks decent. Action: inspect whether the audience, core information, and section structure actually match your plan. Why: if the direction is off in round one, later iterations get more expensive.
</Callout>

```text
Before: Build me a nice homepage.
After: Build a personal page for future employers with a Hero, skill cards, project showcase, and a contact section. Keep it mobile-first.
```

## Step 2: Replace "Make It Better" with Two Clear Iterations
Most beginners do not fail in round one. They fail in round two, when they say things like "make it more premium" or "make it nicer." AI can only guess from that. A stronger method is to change one dimension per round.

For example, use the second round only for visual changes:

```text
Change the main color to a calmer dark blue, tighten the card radius slightly,
and turn the CTA into a solid primary button.
```

Use the third round only for information structure:

```text
Turn the project section into three project cards, each with a title,
a 30-word summary, and one view button.
```

| Iteration style | What AI can understand | Likely result |
|---|---|---|
| Make it more premium | Mostly guesses | Drift and rework |
| Change color and button style | Clear visual target | More stable visual refinement |
| Change project-section structure | Clear layout target | Better information hierarchy |

<Callout type="scenario" title="Vague feedback vs precise feedback">
Wrong move: say "make it nicer." Better move: say "change the palette to dark blue, add a clear Contact Me button, and reduce how much scrolling the first mobile screen needs." The difference is that the second request names the object and the success condition.
</Callout>

## Step 3: Start Reading the File Structure AI Produced
After the page is generated, open the file tree. You do not need to understand every file yet. You only need to start saying, "this file probably controls page structure" or "this folder probably holds reusable pieces." You might see files such as `page.tsx`, `components`, `globals.css`, or `public`.

The point is not framework mastery. The point is observation. Later, when you use IDE or CLI tools, you will need to judge what AI changed, where it changed it, and whether the change belongs there.

| File or folder you may see | Typical responsibility | What to inspect first |
|---|---|---|
| `page.tsx` / `index` | Page structure and section order | Whether the page follows your plan |
| `components` | Reusable cards, buttons, sections | Whether AI separated repeated blocks |
| `styles.css` / `globals.css` | Colors, spacing, typography | Where visual tweaks landed |
| `public` | Images, icons, static assets | Whether the asset set is sensible |

## Unit Summary

- Write a three-line plan before the first prompt.
- Use clear, single-dimension iteration rounds instead of vague aesthetic requests.
- Reading file structure is the first step from "I can generate" to "I can control."

## Methodology Review

This unit turns planning-driven work into a repeatable habit: define the target, iterate intentionally, then inspect the result.
