Vibe Coding is not the opposite of planning. It is a low-friction way to validate ideas first, then move into the right level of control.

## Step 1: Understand Vibe Coding Through One Task
Imagine you want to build a simple weather page. You open a browser-based AI tool and type: "Build a Chinese weather page where users enter a city and see today's temperature, an icon, and a short clothing suggestion. Keep it clean and mobile-friendly." In about 30 seconds, you usually get a visible first version. You did not start by wiring a project, defining folders, or hand-writing the UI. That is the core Vibe Coding experience: describe the outcome first, then react to what the AI produces.

The point is not "never read code again." The point is to lower the cost of the first useful result. When you are still testing an idea, exploring structure, or feeling out a page direction, this mode is extremely efficient.

| Approach | What you do first | Typical startup time | Best fit |
|---|---|---|---|
| Traditional hand-coding | Create project, install deps, scaffold UI | 30-90 min | You already know the plan |
| Vibe Coding | Describe the target outcome | 1-5 min | You are still validating the idea |

<Callout title="Scenario Card: The Right First Expectation" tone="info">
If your first generated page is imperfect, that is not failure. The job of Vibe Coding is not one-shot perfection. It is fast clarification.
</Callout>

## Step 2: The Same Task in Three Modes
Keep the weather page task. In Vibe mode, you ask the AI to make a visible first version. In Spec mode, you first define the user, the modules on the page, the data source, the mobile-first constraint, and what not to build yet. In Agentic mode, you split the work across roles: a planning agent outlines the solution, an execution agent builds it, and a review agent checks whether it respects mobile UX and accessibility.

These are not rankings. They are cost and fit decisions. A weekend side tool does not need Agentic overhead. A payment page should not rely on unbounded Vibe iteration.

| Mode | What happens first | Strength | Main risk |
|---|---|---|---|
| Vibe | Direct outcome request | Fast start, fast feedback | Scope drift |
| Spec | Requirements and plan first | More stable output | More upfront thinking |
| Agentic | Roles and review workflow first | Best for complex parallel work | Highest coordination cost |

**Before / After**

Weak judgment: "AI is strong, so I should always start by asking it to build."

Stronger judgment: "I first decide whether this task is idea validation, spec execution, or multi-role delivery, then choose Vibe, Spec, or Agentic."

## Step 3: Why This Is the Right Time to Learn
This is a strong learning window because three things are true at once: models can hold more context, tools connect generation, editing, preview, and deploy into one loop, and the community has already discovered repeatable patterns. The opportunity is not just "I used AI early." The opportunity is "I learned how not to be dragged around by it."

The course path reflects that. M1-M2 are about defining what to build and why. M3 stabilizes prompts and context. M4 and beyond move into planning, verification, terminal agents, and multi-agent workflows. This is not hype-following. It is capability-building.

| Your current state | What you most need | Immediate payoff |
|---|---|---|
| You only try tools | Mode selection and scope judgment | Fewer wasted starts |
| You can prototype but often derail | Requirement definition and context | Less rework |
| You want to work in real repos | Planning, verification, agent workflows | More reliable delivery |

## Unit Summary
- Vibe Coding turns ideas into visible results quickly.
- Vibe, Spec, and Agentic are different collaboration modes on the same growth path.
- The real skill is choosing the right mode for the task.
