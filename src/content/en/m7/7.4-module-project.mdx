<Callout title="Method Focus" tone="success">
This project is not about automating the maximum number of steps. It is about proving that you can automate one useful path without hiding where human review still belongs.
</Callout>

## Step 1: Choose one bounded automation flow
Pick one path such as:

| Candidate flow | Why it is a good project choice |
|---|---|
| Issue -> plan draft | Clear trigger and low write risk |
| Plan -> implementation draft | Useful when the task boundary is stable |
| Implementation -> lint/test/report | High value and low ambiguity |
| Test result -> review summary | Good for structured reporting work |

The project does not need to be huge. It needs to be clear, bounded, and worth repeating.

<Callout title="Scenario: choosing a safe first automation target" tone="info">
Situation: you are tempted to automate the entire issue-to-merge path. Action: begin with one shorter chain such as implementation to lint-and-test report. Why: the shorter flow is easier to verify and makes hidden risk easier to see.
</Callout>

## Step 2: Add machine guardrails and human checkpoints
Every automation path should declare both its machine checks and its stop points.

| Stage | Machine guardrail | Human checkpoint |
|---|---|---|
| Plan generation | Format and completeness checks | Human confirms scope and risk |
| Code generation | Lint, type check, tests | Human reviews business logic and UX impact |
| Release-related changes | Secret scanning and config checks | Human approves launch risk |

If you cannot explain what the automation must not cross, the workflow is not ready.

<Separator />

## Step 3: Review the methodology progression with a concrete framework
Use this project to review the full progression:

| Module stage | Question to answer in your retrospective |
|---|---|
| M1-M2 | Did I define the right problem and scope? |
| M3-M4 | Did I frame the work clearly enough for AI and for myself? |
| M5-M6 | Did I preserve context, checkpoints, and reusable operating assets? |
| M7 | Did I automate only what had clear boundaries and review checkpoints? |

That review is what turns the project into a methodology milestone instead of a single experiment.

```text
Before: I treated the automation project as "automate as many steps as possible."
After: I treat it as "automate only the bounded segment, then make machine guardrails and human re-entry points explicit."
```

## Summary
- A good M7 project automates one bounded path rather than the whole universe.
- Machine checks and human checkpoints must both be explicit.
- The retrospective should evaluate not only the workflow output, but also whether your methodology maturity actually increased.

## Methodology Review
Closed-loop delivery becomes system-level delivery here: the loop now includes automation design, guardrail design, and explicit human re-entry points.
