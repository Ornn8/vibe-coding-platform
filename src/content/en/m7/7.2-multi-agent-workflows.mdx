<Callout title="Method Focus" tone="warning">
The value of multiple agents is not "more windows." It is lower coordination cost through clear write scope, explicit handoffs, and bounded parallel work.
</Callout>

## Step 1: Do not split work until boundaries are real
Parallel work only helps when subtasks have clean ownership.

| Task shape | Good for multiple agents? | Why |
|---|---|---|
| Rewrite eight unrelated MDX files | Yes | Each file can be owned and reviewed independently |
| Debug one cross-cutting auth bug | Usually no | The diagnosis path changes too quickly |
| Add a feature with separate planning and review steps | Yes | Different roles add value without overwriting each other |

If one agent is asked to plan, code, test, and review all at once, you do not get a system. You get faster confusion.

## Step 2: Look at one real sub-agent workflow
Imagine you are upgrading lesson content across a module.

| Role | Input | Output | Write scope |
|---|---|---|---|
| Planning agent | Revision rules and file list | Rewrite checklist and task split | No file edits |
| Rewrite agent | One lesson file plus checklist | Updated lesson content | One lesson file |
| Review agent | Updated lesson plus standards | Findings and missed requirements | No file edits |

Here is the actual flow:

1. The planning agent reads the revision standard and splits the file list.
2. The rewrite agent updates one lesson at a time.
3. The review agent checks scenarios, tables, before/after, and L3 exercises.
4. A human or primary agent resolves findings before the next batch starts.

That is a real sub-agent workflow. It is not just a box diagram. It is a bounded exchange of inputs, outputs, and permissions.

```text
Before: I put planning, implementation, testing, and review into one agent.
After: I split the role boundaries first, then let each agent handle only the segment it can own safely.
```

<Callout title="Scenario: what makes a sub-agent useful instead of decorative" tone="info">
Situation: you want help on a batch rewrite. Action: give the rewrite agent only one file and one checklist, and give the review agent read-only responsibility. Why: the system gets safer when each role knows both what it should do and what it must not touch.
</Callout>

## Step 3: Automation should follow a stable workflow, not replace judgment
Once a role pattern becomes stable, parts of it can be automated.

| Workflow stage | Safe to automate? | Human checkpoint |
|---|---|---|
| Trigger task from a clear request | Often yes | Human confirms the scope |
| Generate a plan draft | Often yes | Human checks boundary and risk |
| Run lint and type checks | Yes | Human reviews failed or risky cases |
| Auto-merge business logic changes | Usually no | Human approval stays mandatory |

The goal is not maximum automation. The goal is explicit automation boundaries.

<Exercise id="ex-7.2.3" level="L3" title="Map one real workflow from trigger to approval" description="Write or diagram one workflow from task trigger to final review, then mark which steps can be automated and which still require human approval in a real project." />

## Summary
- Multi-agent work helps only when write boundaries and handoffs are clear.
- A real sub-agent flow is defined by inputs, outputs, and permissions, not a concept sketch.
- Automation becomes useful only after the workflow is already stable enough to trust.

## Methodology Review
Planning-driven work becomes orchestration here: the system is designed around role boundaries first, then parallel work and automation are added on top of that structure.
