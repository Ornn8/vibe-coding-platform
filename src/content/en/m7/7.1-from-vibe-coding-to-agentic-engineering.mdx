<Callout title="Method Focus" tone="warning">
The real upgrade is not asking AI to do more in one thread. It is designing a collaboration system where roles, checks, and handoffs make the output more trustworthy.
</Callout>

## Step 1: Start with code that runs but is still dangerous
Here are two examples of AI slop. Both can run. Neither should be trusted yet.

```ts
export async function GET() {
  const users = await prisma.user.findMany();
  return Response.json(users);
}
```

```ts
export async function updateUser(id: string, input: any) {
  return prisma.user.update({ where: { id }, data: input });
}
```

| Code sample | Why it looks fine at first | Why it is actually risky |
|---|---|---|
| Open user endpoint | It returns real data fast | No auth or access boundary |
| Direct update helper | It is short and flexible | `any` input, no validation, no permission check |

That is the problem M7 solves. The issue is no longer "Can AI generate code?" The issue is "How do we stop plausible-looking code from becoming hidden debt?"

<Callout title="Scenario: when 'it works' is no longer a good enough standard" tone="warning">
Situation: the page renders and the endpoint responds, so the task feels done. Action: ask what review boundary has not been crossed yet, such as auth, validation, or maintainability. Why: in larger projects, hidden defects matter more than surface success.
</Callout>

## Step 2: The role shift is from executor to system designer
In vibe coding, you mainly issue requests and react to outputs. In agentic engineering, you define roles, inputs, outputs, and review boundaries before the system starts moving.

| Working style | Main question |
|---|---|
| Vibe coding | "What should I ask AI to build next?" |
| Agentic engineering | "What role should handle this, what should it output, and who verifies it?" |

The difference is not status or jargon. It is where you spend design effort.

```text
Before: I asked one assistant to plan, code, debug, and review in the same thread.
After: I separated planning, implementation, and verification into different responsibilities before the work started.
```

## Step 3: PEV still matters, but now it can be distributed
The PEV loop did not disappear. It became a multi-role workflow.

| PEV step | Possible role | Concrete output |
|---|---|---|
| Plan | Planning agent | Task breakdown, scope, acceptance checks |
| Execute | Coding agent | Code changes or content changes |
| Verify | Review or test agent | Findings, failed checks, approval or rejection |

<Callout title="Scenario: a real role-based handoff" tone="info">
Situation: you want to add review notes to a project submission page. Action: one agent writes the implementation plan, one agent updates the code, and one agent checks lint, types, and user-facing regressions. Why: the point is not more activity. The point is cleaner handoffs and a more inspectable loop.
</Callout>

<Exercise id="ex-7.1.3" level="L3" title="Design a real PEV workflow with separate roles" description="Choose one real feature or lesson rewrite, define who handles Plan, Execute, and Verify, and then run at least one of those roles in a real tool or repository workflow." />

## Summary
- AI slop is often code that runs but carries hidden risk.
- Agentic engineering shifts the focus from prompting to role and boundary design.
- PEV becomes stronger when planning, execution, and verification are explicitly separated.

## Methodology Review
Planning-driven work scales here from "write a plan first" into "design the whole collaboration system first, then let execution happen inside it."
