<Callout title="Method Focus" tone="warning">
The faster agents generate changes, the more important guardrails become. Without them, throughput turns into cognitive debt faster than teams expect.
</Callout>

## Step 1: Scale multiplies defects as easily as output
Suppose an agent workflow produces 20 changes per day and only 2 percent are defective.

| Metric | Number |
|---|---|
| Changes per day | 20 |
| Defect rate | 2% |
| Expected bad changes per day | 0.4 |
| Expected bad changes in 5 days | 2 |

Two bad changes per week is already enough to create repeated review friction, rollback work, and trust problems.

<Callout title="Scenario: why low error rates still become expensive" tone="warning">
Situation: each individual defect feels rare, so the workflow seems safe. Action: multiply the defect rate by the number of generated changes over a week or month. Why: the system cost appears at scale, not in a single happy-path run.
</Callout>

## Step 2: Let machines catch the mechanical failures first
Here is a minimal guardrail loop:

```bash
npm run lint
npx tsc --noEmit
npm test
```

Now imagine lint returns this:

```text
src/app/api/profile/route.ts
  12:18  error  Unexpected any. Specify a different type  @typescript-eslint/no-explicit-any
```

And the fix becomes:

```ts
type ProfileInput = {
  displayName: string;
  bio?: string;
};
```

| Guardrail | What it catches well | What it does not judge well |
|---|---|---|
| ESLint | Style, unsafe patterns, some logic smells | Product correctness |
| Type check | Type mismatches and missing fields | UX quality |
| Tests | Regressions in expected behavior | Architecture tradeoffs |

This is the core quality-guardrail lesson: push mechanical failures into automation so human review can spend time on judgment instead.

## Step 3: Humans still review risk, meaning, and long-term cost
Not everything should be auto-approved.

| Review category | Best reviewer |
|---|---|
| Syntax, style, type safety | Machine |
| Permission boundaries and business rules | Human |
| Architecture tradeoffs and future maintenance cost | Human |
| Secret handling and release safety | Machine plus human |

```text
Before: if lint and tests passed, I felt ready to trust the whole change.
After: lint and tests only clear the mechanical layer; risky meaning and business impact still need a person.
```

## Summary
- Small defect rates become meaningful system cost when workflows scale.
- Lint, types, and tests should remove mechanical failures before human review.
- Human attention should stay focused on risk, meaning, and long-term maintainability.

## Methodology Review
Closed-loop delivery grows into system-level verification here: machines block repeatable failures, while humans protect the boundaries that still require judgment.
