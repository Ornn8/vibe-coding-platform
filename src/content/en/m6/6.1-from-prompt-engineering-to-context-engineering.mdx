<Callout title="Method Focus" tone="success">
Prompt engineering improves a single turn. Context engineering keeps long-term collaboration stable. The longer the project gets, the more the center of gravity shifts from phrasing to memory design.
</Callout>

## Step 1: Watch a long project slowly degrade
Imagine a course-platform project that continues for 20 rounds of collaboration.

| Round | What AI feels like | What is actually happening |
|---|---|---|
| 1 | Clear, aligned, fast | The task is short and the context is still intact |
| 5 | Mostly good, but occasionally forgetful | Early constraints start falling out of view |
| 10 | Style drifts and old decisions get questioned again | Project-level memory is no longer stable |
| 20 | It behaves like a new project | The thread has lost durable continuity |

This is the moment where "I just need a stronger prompt" stops being a good diagnosis.

<Callout title="Scenario: when you should stop patching prompts and fix the system" tone="warning">
Situation: you have reminded AI for the tenth time not to change the database schema. Action: stop adding one more sentence to the prompt and move that rule into the system or project layer. Why: repeated reminders are a signal that the missing thing is durable context, not one more clever phrasing tweak.
</Callout>

## Step 2: Context engineering manages multiple layers
The most practical model is a four-layer stack:

| Layer | What it stores |
|---|---|
| System layer | Long-lived principles, guardrails, and collaboration rules |
| Project layer | Background, goals, architecture, and known constraints |
| Task layer | What this round needs to produce right now |
| Tool layer | What capabilities and integrations are currently available |

Prompt engineering mostly strengthens the task layer. Context engineering manages how all four layers stay available together.

<Callout title="Scenario: same failure, different broken layer" tone="info">
Situation: AI keeps ignoring a no-schema-change rule. Action: decide whether the rule was never written at the system layer, buried in project context, or missing from the current task handoff. Why: the fix depends on which layer failed, not on how annoyed you feel.
</Callout>

## Step 3: When context decays, repair the right layer
Common decay signals include repeated mistakes, forgotten project rules, style drift, and answers that target the wrong problem.

| Decay signal | Most likely weak layer |
|---|---|
| Stable rules keep getting ignored | System layer |
| AI forgets the project goal or past decision | Project layer |
| The current assignment is misunderstood | Task layer |
| AI does not use available capabilities well | Tool layer |

```text
Before: when AI drifted, I just kept adding more text to the prompt.
After: I first diagnosed which layer had degraded, then repaired that exact layer.
```

That shift matters because random history stuffing makes threads heavier, not clearer.

<Exercise id="ex-6.1.3" level="L3" title="Analyze and repair one context-decay case" description="Choose one long conversation or long project you have seen, identify the decay signal, decide which layer failed, and write a repair plan for that specific layer." />

## Summary
- Long projects fail when durable context is not managed across time.
- The four-layer model gives you a practical diagnostic map.
- When collaboration degrades, repair the missing layer instead of only extending the prompt.

## Methodology Review
Context stabilization becomes context engineering here: the goal is no longer to help AI remember more text, but to organize the right information in the right layer for the next round of work.
