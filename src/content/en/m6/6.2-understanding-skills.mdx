<Callout title="Method Focus" tone="success">
The main job of a Skill is not to make AI "know more." It is to make AI behave more consistently on one class of work by packaging triggers, boundaries, workflow, and examples.
</Callout>

## Step 1: Compare behavior before and after a Skill is loaded
Imagine the task is "rewrite an MDX lesson into the platform's standard teaching structure."

| State | What AI usually does |
|---|---|
| No Skill loaded | It may improve the lesson, but miss the exercise levels, forget the before/after block, or skip the scenario card |
| Content-revision Skill loaded | It is much more likely to check method focus, three steps, scenario cards, before/after, and L1/L2/L3 exercises systematically |

```text
Before: Please improve this lesson.
After: Load the content-revision Skill, then rewrite the lesson using method focus, three steps, scenario cards, before/after, and L1/L2/L3 exercises.
```

The difference is not magic. The Skill turns "how this task should usually be done" into reusable operating guidance.

<Exercise id="ex-6.2.1" level="L1" title="Describe the behavior gap before and after a Skill" description="Pick one familiar task and write two likely behavior differences between AI with no Skill and AI with a well-targeted Skill." />

## Step 2: A good Skill is structured, not merely long
A useful Skill usually has four core parts:

| Part | What it does |
|---|---|
| Trigger | Tells AI when the Skill should activate |
| Boundaries | Tells AI what must not happen |
| Workflow | Tells AI what order to follow |
| Example | Tells AI what good output looks like |

If a Skill is only a background paragraph, AI still has to guess when to apply it and how strict it should be.

<Separator />

## Step 3: Skills become valuable when they reduce repeated explanations
If you keep telling AI the same rules across multiple tasks, such as "read the project constraints first," "do not touch the database schema," or "list findings before suggesting fixes," you are already looking at the shape of a future Skill.

<Callout title="Scenario: when repeated instructions should become a Skill" tone="info">
Situation: you have restated the same collaboration rules in three or more different tasks. Action: promote those rules into a Skill. Why: once a workflow is repeated, it should stop living only in chat and start living in reusable operating guidance.
</Callout>

| Repeated instruction | Better place to store it |
|---|---|
| Read constraints before editing | Workflow step |
| Do not change the schema | Boundary |
| Present findings before fixes | Output example or workflow rule |

<Exercise id="ex-6.2.3" level="L3" title="Turn repeated instructions into a first Skill draft" description="Choose one set of instructions you have repeated many times and turn it into a small Skill draft with at least a trigger, two boundaries, and a workflow." />

## Summary
- Skills improve repeatability more than raw knowledge.
- Strong Skills are clear about trigger, boundaries, workflow, and examples.
- Repeated explanations are often a sign that a reusable Skill should exist.

## Methodology Review
Glue coding levels up here by turning repeated operating advice into reusable assets instead of making you re-teach the same workflow every time.
