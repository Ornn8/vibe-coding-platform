<Callout title="Method Focus" tone="success">
Writing a Skill is not a trick. It is the act of compressing a verified way of working into a reusable workflow that AI can execute again later.
</Callout>

## Step 1: Start with one concrete, repeated theme
We will use `Tailwind CSS` as the example topic because it creates the right kind of repeated guidance:

| Repeated instruction | Why it belongs in a Skill |
|---|---|
| Reuse existing utility patterns first | It is a stable rule, not a one-off note |
| Do not add large custom CSS casually | It is a clear boundary |
| Check mobile layout after spacing edits | It is a repeatable verification action |

That is the signal you want. The problem is no longer "How do I phrase this once?" It is "How do I make this workflow reusable?"

<Exercise id="ex-6.3.1" level="L1" title="Find a repeated task that deserves a Skill" description="List three rules you repeat in the same type of task and decide whether they are stable enough to package into a Skill." />

<Separator />

## Step 2: Draft a minimum viable `SKILL.md`
Here is a minimal but usable skeleton:

```md
# Tailwind Layout Editing

## Trigger
- When editing existing Tailwind-based UI

## Boundaries
- Reuse the current utility style first
- Avoid custom CSS unless necessary
- Keep mobile-first responsive behavior

## Workflow
1. Read the current component structure
2. Identify the layout or spacing issue
3. Prefer utility-class edits
4. Check mobile and desktop behavior

## Example
- Fix uneven card spacing without rewriting the component tree
```

This draft is not perfect yet, but it already answers three practical questions: when should the Skill run, what should it avoid, and what sequence should it follow?

## Step 3: Strengthen the draft with examples and prohibitions
The draft becomes much more executable when you add trigger examples and explicit forbidden moves.

```text
Before: When editing UI, try to use Tailwind.
After: When the task involves layout, spacing, or responsive issues in an existing Tailwind component, activate this Skill. Do not create a new CSS file or restructure the component tree unless the request explicitly requires it.
```

<Callout title="Scenario: why boundaries cannot stay vague" tone="warning">
Situation: AI only needed to adjust utility classes, but it also rewrote the component structure and added custom CSS. Action: write those forbidden moves directly into the Skill. Why: if the boundary stays fuzzy, AI treats "helpful" as permission to keep expanding scope.
</Callout>

| Extra section to add | What it improves |
|---|---|
| Trigger examples | Helps AI recognize the right situation faster |
| Explicit prohibitions | Prevents scope creep and over-editing |
| Good-output example | Makes the expected result easier to imitate |

## Step 4: A Skill only becomes real after testing
Skills should be tested like code.

| Test question | What to observe |
|---|---|
| Does AI drift less? | It stops changing unrelated parts |
| Do you repeat yourself less? | The old explanation burden goes down |
| Is the output more stable? | The result follows the same workflow more consistently |

<Exercise id="ex-6.3.4" level="L3" title="Use your Skill on a real task and iterate once" description="Apply your Skill to one real task, record what mistakes disappeared and what ambiguity remains, then revise the Skill one more time." />

## Summary
- The best Skill topics are repeated, rule-heavy tasks.
- A minimal `SKILL.md` needs Trigger, Boundaries, Workflow, and Example.
- Testing is what turns a draft into a real reusable asset.

## Methodology Review
Glue coding becomes "productizing experience" here: not retelling the same rules, but turning them into something future tasks can actually inherit.
