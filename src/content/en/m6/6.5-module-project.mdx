This module project is not mainly a showcase of how many tools you can name. It is proof that you can organize rules, capabilities, and memory into one stable workflow system.
</Callout>

## Step 1: Design the toolchain as a system, not a pile
A mature project usually has at least four asset types:

| Asset | What it is responsible for | What counts as done |
|---|---|---|
| `CLAUDE.md` | Long-lived rules and boundaries | AI can follow the project's baseline behavior more consistently |
| Self-authored Skills | Repeated high-frequency workflows | They reduce repeated explanation in real tasks |
| MCP connections | External capability access | At least one real system can be accessed in a useful workflow |
| Memory bank | Project continuity | A new session can restore context from files instead of guesswork |

When these four layers are clearly assigned, the toolchain starts behaving like a system instead of a pile of disconnected habits.

## Step 2: Define completion standards for each layer
This is the part teams skip most often. A file existing is not the same as the file working.

| Project part | Weak definition of done | Stronger definition of done |
|---|---|---|
| `CLAUDE.md` | The file exists | AI actually follows the boundaries more reliably |
| Skill | There is a draft | It has been used on a real task at least once |
| MCP | The connection is configured | One real workflow has been completed and explained |
| Memory bank | The folder exists | A new session can recover project state from it |

```text
Before: I created these files, so the toolchain should be complete.
After: These assets count only when they measurably change how collaboration behaves.
```

<Separator />

## Step 3: Submit both the product layer and the capability layer
The final project pack should include:

| Layer | What you submit |
|---|---|
| Product layer | A working app, page set, or repository result |
| Capability layer | `CLAUDE.md`, at least two self-authored Skills, at least one MCP-backed workflow, and a memory bank |

If you ship only the app, people can see that the result exists. If you ship the capability layer too, they can see why the next round of work should be easier, safer, and cheaper.

<Callout title="Scenario: why the capability layer is part of the deliverable" tone="info">
Situation: you feel that only the page and features should count as project output. Action: treat `CLAUDE.md`, Skills, MCP workflows, and memory-bank files as first-class deliverables too. Why: those assets determine whether the next round of collaboration starts from a system or from scratch.
</Callout>

## Summary
- The key to the toolchain is clear division of responsibility, not tool count.
- Every layer needs a real definition of done.
- A complete project submission should show both the product result and the capability system behind it.

## Methodology Review
Closed-loop delivery becomes capability-system delivery here: not just handing over the app, but also handing over the rules, tools, and memory that make the next iteration stable.
