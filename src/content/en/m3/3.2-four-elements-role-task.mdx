<Callout type="methodology" title="Methodology Focus">
Role changes the way AI thinks. Task changes the boundary of what it is allowed to do. If either one is vague, later constraints are just cleanup work.
</Callout>

## Step 1: The Same Task Changes When the Role Changes
Use the task "improve the lesson-page intro area." Keep the task almost the same, but swap the role:

| Role setting | What AI is more likely to focus on |
|---|---|
| Frontend engineer | Layout, components, responsive structure |
| Content editor | Wording, clarity, information order |
| Course designer | Learning path, cognitive load, exercise progression |
| Code reviewer | Risk, regressions, implementation boundaries |

```text
Role 1: You are a senior frontend engineer. Improve the structure
and mobile hierarchy of the lesson-page hero.

Role 2: You are a curriculum editor. Improve the copy and
information order of the lesson-page hero.

Role 3: You are a course designer. Check whether the hero helps
beginners understand the unit goal quickly.

Role 4: You are a code reviewer. Find the most likely regressions
that a hero redesign could introduce.
```

The task family is almost identical, but the model's default attention changes with the role. Role is not there to sound impressive. It is there to preset the judgment angle.

## Step 2: Vague Roles Barely Help
Statements like "you are a top expert" sound strong, but they do not actually constrain AI in a useful way. A good role is directly tied to the work.

```text
Before: You are a very powerful AI.
After: You are a curriculum editor responsible for a learning-platform content upgrade, and you should prioritize structure completeness, scenario vividness, and exercise progression.
```

| Role phrasing | Problem | Better direction |
|---|---|---|
| You are a top expert | No concrete working boundary | Name the responsibility and criteria |
| You are a content editor | Still wide | Add what content and what standards matter |
| You are the curriculum editor for this lesson upgrade | More executable | The role and target start to align |

<Callout type="scenario" title="When to rewrite the role first">
Situation: AI keeps giving reasonable-looking output, but it still focuses on the wrong things. Action: rewrite the role before piling on more task detail. Why: if the role is wrong, the model's starting perspective is already off.
</Callout>

## Step 3: The Task Has to Be Executable
Task statements cannot stop at "improve this" or "check this page." A stronger task needs at least three things:

| Task element | What it should say |
|---|---|
| Goal | What should be completed |
| Scope | Which parts are in play |
| Completion standard | What counts as acceptable |

```text
Weak: Improve this lesson page.
Strong: Only review the hero title, description, and CTA of this lesson page. The goal is that a beginner should understand what this unit teaches within 10 seconds. Do not change any other section.
```

<Callout type="scenario" title="Vague task vs executable task">
Wrong move: ask AI to "improve it." Better move: limit the work to the hero title, description, and CTA, state the beginner-understanding goal, and exclude the rest of the page. The second version already carries goal, scope, and a finish line.
</Callout>

## Step 4: Combine Role and Task into a Prompt You Can Actually Test
Once both parts are clear, you can combine them into a prompt you can test, review, and reuse:

```text
You are a curriculum editor working on a learning-platform content upgrade.
Only review Step 1 and Step 2 of this MDX file.
Your goal is to help Chinese beginners enter the scenario faster.
Output the result as problems first, then revision suggestions.
```

That kind of prompt is no longer guesswork. It is a reusable working template.

## Unit Summary

- Role shapes the model's working perspective.
- Task shapes the boundary of action.
- Vague roles and vague tasks both create outputs that look right while still drifting.
