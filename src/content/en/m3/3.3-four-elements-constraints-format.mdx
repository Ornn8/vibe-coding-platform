<Callout type="methodology" title="Methodology Focus">
Constraints prevent drift. Format makes the result reusable. You need both if you want output that fits into a real workflow.
</Callout>

## Step 1: Technical Constraints Stop AI from Inventing the Wrong Implementation
Technical constraints tell AI what technical boundary it must stay inside.

```text
Before: Help me build a lesson page.
After: Update the lesson page inside the existing Next.js App Router project, keep MDX rendering, do not change route structure, and do not introduce new frameworks.
```

| Constraint type | Example | What it prevents |
|---|---|---|
| Technical constraint | Keep using Next.js App Router | Prevents AI from changing architecture casually |
| Technical constraint | Do not modify the database schema | Prevents scope expansion |
| Technical constraint | Reuse existing components first | Prevents unnecessary reinvention |

If you skip technical constraints, AI may casually add libraries, patterns, or structures that the repository never asked for.

## Step 2: Style Constraints and Negative Constraints Change the Tone and Boundary
Style constraints define what the result should feel or sound like. Negative constraints define what must not happen.

```text
Style constraint before:
Rewrite the lesson intro.

Style constraint after:
Rewrite it in a calm, instructional tone for Chinese beginners.
Do not make it sound like marketing copy.
```

```text
Negative constraint before:
Improve this page.

Negative constraint after:
Only improve the hero and summary. Do not move the buttons,
do not add new features, and do not touch any other section.
```

| Constraint type | before | after |
|---|---|---|
| Style constraint | Make it more professional | Use a calm, instructional, low-marketing tone |
| Negative constraint | Improve the page | Only change the hero; do not add features or touch other blocks |

<Callout type="scenario" title="When negative constraints matter most">
Situation: you already know some things must not change, such as the database, routing, or unrelated parts of the page. Action: write the forbidden zone explicitly. Why: AI respects explicit prohibitions much more reliably than unstated assumptions.
</Callout>

## Step 3: Format Decides Whether the Result Can Be Picked Up by the Next Step
If you do not ask for a format, AI may still answer well, but the answer might arrive as a wall of text when what you really needed was a checklist, a findings list, or a table.

| Task | Best format | Why |
|---|---|---|
| Content review | Findings list plus revision suggestions | Easier to fix issue by issue |
| Requirement synthesis | Table plus priority column | Easier to decide and compare |
| Execution plan | Step list plus verification notes | Easier to act on directly |

```text
Weak: Please review this content.
Strong: Review this content and output problems first, then revision suggestions, ordered by severity.
```

<Callout type="scenario" title="No format vs useful format">
Wrong move: ask AI to "review this" and hope the response arrives in a workable shape. Better move: request a findings-first structure so the answer can flow directly into the next task. One approach feels like chat. The other behaves like a workflow node.
</Callout>
