<Callout type="methodology" title="Methodology Focus">
A prompt is not a magic spell. The real variable is whether AI has enough stable, decision-shaping context to work with.
</Callout>

## Step 1: The Same Task Looks Different Under Weak and Strong Context
Use the task "build a lesson page." Start with two prompts:

```text
Weak context:
Help me build a lesson page.
```

```text
Strong context:
Build a lesson page for a Chinese-first Next.js learning platform.
The page is for one lesson, must preserve chapter navigation,
the progress action, and the MDX content area.
Do not change the database structure or introduce new routes.
Keep it mobile-first.
```

Both prompts ask for a lesson page, but the AI workload is completely different. In the weak version, the model must guess the stack, the language, the layout, and the audience. In the strong version, those decisions are already made.

| Prompt version | What AI still has to guess | What usually happens |
|---|---|---|
| Weak context | Stack, language, layout, component boundaries | The output looks complete but misses the project |
| Strong context | Mostly implementation details | The output fits the repo more closely |

<Callout type="scenario" title="What to inspect when AI 'doesn't get it'">
Situation: AI gives you something usable-looking, but the language, structure, or style do not fit the project. Action: check whether you specified the stack, audience, page purpose, and explicit exclusions. Why: most 'misunderstanding' is actually context omission.
</Callout>

## Step 2: Real Output Differences Make the Value Obvious
Imagine AI starts working from both prompts. You usually get differences like these:

| Dimension | Weak-context output | Strong-context output |
|---|---|---|
| Page structure | AI invents hero blocks, marketing sections, and extra modules | The layout stays closer to lesson content and navigation |
| Technical style | AI may use patterns the repo does not use | AI is more likely to preserve the existing structure |
| Language | It may default to English or a mixed tone | It stays closer to the Chinese learning context |
| Completeness | It looks polished but drifts | It stays narrower and needs less rework |

```text
Before: I thought "more context" just meant AI would be easier to control.
After: I understand that context directly changes structure, language, component choices, and rework cost.
```

## Step 3: Not All Context Belongs in One Prompt
The most practical way to split context is into three layers:

| Layer | What it is for | Example |
|---|---|---|
| Task layer | What this specific round is doing | Only rewrite Step 1 and Step 2 of the lesson |
| Project layer | What kind of project this is | A Chinese-first Next.js learning platform |
| System layer | Stable long-running rules | Do not touch the database schema; avoid `any`; reuse existing components |

The task layer belongs in the current prompt. Stable project and system information often belongs in `CLAUDE.md`, `AGENTS.md`, or repository instructions.

<Callout type="scenario" title="Everything in one prompt vs layered context">
Wrong move: rewrite the project background, stable rules, and task details from scratch every time. Better move: store stable information in a reusable system layer and use the prompt for what changes this round. The second approach is less noisy and much more reusable.
</Callout>

## Unit Summary

- Output differences are often context differences in disguise.
- Stronger context changes AI structure choices and rework cost directly.
- Stable information should be layered instead of rewritten every time.

## Methodology Review

This unit is the first concrete practice of context stabilization: take stable information out of temporary chat and turn it into reusable collaboration infrastructure.
