## Step 1: What a System Prompt Is

A system prompt is not just a fancy instruction. It is the project-level agreement that tells AI how to behave over time: what its role is, what principles matter, and what should remain consistent.

That is why files like `CLAUDE.md`, `AGENTS.md`, or similar project instructions matter so much. They reduce repetition and help collaboration stay stable.

## Step 2: What Belongs in CLAUDE.md

Useful system prompts usually define role, principles, guardrails, and delivery expectations. They may also list stack choices, communication habits, verification rules, or project-specific constraints.

The goal is to turn invisible team knowledge into something that AI can actually use.

Use a structure like this:

- Role: who AI should be in the project
- Principles: the stable habits it should follow
- Boundaries: what it should not change without asking
- Delivery rules: how work should be verified and handed off

For example, a small CLAUDE.md might say: "You are a careful pair-programming assistant for this repository. Keep the current route structure intact, do not change file names without permission, run lint before finalizing, and summarize the change in one short paragraph."

## Step 3: Keep It Maintainable

A good system prompt is not overloaded with every possible detail. It should stay readable and stable. If it becomes too long or too specific, people stop using it.

The best version is the one that actually helps you start work faster and stay aligned longer.

## Step 4: Apply Construction Principles

System prompt construction should follow a few simple principles: keep it clear, keep it consistent, and keep the boundaries explicit. That is how a reusable project manual stays useful over time.

When a system prompt is well built, it becomes a shared memory layer instead of a pile of reminders.
