<Callout type="methodology" title="Methodology Focus">
The value of a system prompt is not that it looks professional. The value is that it turns long-running project knowledge into reusable rules.
</Callout>

## Step 1: `CLAUDE.md` Is a Project Manual, Not a Template Demo
When you place `CLAUDE.md` in a repository, you are really telling AI three things: what the project is, what long-term rules matter, and which boundaries must not be crossed casually.

| Section type | What question it answers |
|---|---|
| Project background | What kind of project this is and who it serves |
| Stack declaration | Which frameworks and patterns should stay in place |
| File and boundary rules | What should not be changed lightly |
| Verification and delivery rules | How a change should prove itself |

Without those sections, AI has to re-guess the project in every new task.

## Step 2: Why the Stack Declaration Needs to Be Specific
Look at a system-level statement like this:

```text
This project uses Next.js App Router, TypeScript strict mode,
Tailwind CSS, and shadcn/ui.
Preserve the current route structure and do not introduce new frameworks
or state-management patterns without approval.
```

That text is not there to show off stack names. It exists to stop AI from making silent architectural choices that the repository did not ask for.

```text
Before: Please follow the project rules.
After: This project uses Next.js App Router and TypeScript strict mode. Do not change the route structure or introduce a new framework casually.
```

<Callout type="scenario" title="What the stack declaration is really preventing">
Situation: you only wanted a page change, but AI starts suggesting new state libraries or a different routing style. Action: write the stack and the preserve rules directly into the system prompt. Why: the stack declaration is not decoration; it is an anti-drift mechanism.
</Callout>

## Step 3: Keep Long-Running Rules Separate from One-Off Exceptions
`CLAUDE.md` should hold rules that remain useful across many tasks.

| Good for `CLAUDE.md` | Better left in a one-off prompt |
|---|---|
| Read the project constraints before editing | Only change the hero in this task |
| Explain verification before finalizing | Use a table for this particular answer |
| Do not modify the database schema casually | Leave the CTA position alone for this round |

If you keep promoting one-time instructions into the system prompt, the file quickly becomes bloated and noisy.

<Callout type="scenario" title="Temporary exception vs reusable rule">
Wrong move: take something said once in a temporary task and upgrade it into a permanent system rule. Better move: only promote rules that keep proving useful across tasks. Otherwise the system file gets louder while becoming less trustworthy.
</Callout>

## Step 4: Use Real Failures to Discover Missing System Rules
The most practical feedback loop is this: when AI keeps making the same class of mistake, ask whether the missing fix is actually a system-layer rule.

| Repeated AI mistake | Missing system rule that might help |
|---|---|
| Keeps trying to add new libraries | Reuse existing dependencies unless explicitly asked otherwise |
| Keeps skipping checks | Explain verification steps before finalizing |
| Keeps editing beyond scope | Confirm file scope and exclusions before making changes |

## Unit Summary

- `CLAUDE.md` is a project collaboration manual, not a decorative template.
- Stack declarations exist to prevent silent architectural drift.
- Only rules that stay useful across tasks should become system-level guidance.

## Methodology Review

This unit is pure context stabilization: take recurring project knowledge and turn it into explicit, reusable rules.
