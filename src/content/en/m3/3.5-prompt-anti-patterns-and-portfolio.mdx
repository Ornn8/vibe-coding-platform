<Callout type="methodology" title="Methodology Focus">
A prompt only becomes an asset when it can be reviewed, repaired, and reused. One lucky success is not a method.
</Callout>

## Step 1: Anti-Pattern 1, Vague Prompts Force AI to Guess
Look at one full failure chain:

| Stage | Content |
|---|---|
| Original prompt | Improve this lesson page |
| AI output | Changed the hero, added marketing language, rewrote button copy |
| Real issue | You never said what "improve" meant or where the scope ended |
| Better rewrite | Only review the hero title, summary, and CTA so beginners understand the unit in 10 seconds. Do not change any other section |
| Result after the fix | The scope narrows and the output becomes more reviewable |

The problem is not that the prompt is short. The problem is that it lacks a goal, a scope, and a finish line.

## Step 2: Anti-Pattern 2, Greedy Prompts Spread Attention Too Thin
Now look at another chain:

| Stage | Content |
|---|---|
| Original prompt | Build the homepage, login, comments, payments, and memberships |
| AI output | Touched every area a little, but nothing felt stable |
| Real issue | You tried to collapse multiple product stages into one round |
| Better rewrite | This round only builds login and the product-detail flow before payment. Do not build comments or memberships yet |
| Result after the fix | AI can focus on one closed loop instead of five half-loops |

```text
Before: Put every idea into one prompt.
After: Drive one closed loop at a time and write explicit exclusions.
```

<Callout type="scenario" title="Why greedy prompts are easy to misread as progress">
Situation: AI generates many features in one round, and the output feels impressive. Action: ask whether one meaningful closed loop is actually stable. Why: feature count is not the same as delivery quality.
</Callout>

<Separator />

## Step 3: Anti-Pattern 3, No Verification Means No Closure
Here is the third chain:

| Stage | Content |
|---|---|
| Original prompt | Fix this bug |
| AI output | Produced code that looks like a fix |
| Real issue | You never asked how the fix should be verified |
| Better rewrite | Fix this bug and explain how you would verify the change, including at least one command, manual check, or comparison |
| Result after the fix | The work no longer stops at "looks fixed" |

Without verification, it is too easy to confuse "the model produced output" with "the problem is solved."

<Exercise id="ex-3.5.3.1" level="L2" title="Add verification to weak prompts" description="Take three prompts that lack verification and rewrite them so each one includes at least one concrete validation method." />

<Separator />

## Step 4: Turn Good Prompts into a Portfolio
Once you start storing the task type, the prompt, the result, the fit conditions, the common failure modes, and the verification method, you are no longer saving chat history. You are building a Prompt Portfolio.

| Portfolio field | What it should capture |
|---|---|
| Task type | Content review, requirement breakdown, bug fix, and so on |
| Prompt body | The version that actually works |
| Fit conditions | When to use it |
| Common drift | Where it tends to go wrong |
| Verification | How to decide whether the result is good enough |

<Callout type="scenario" title="Chat memory vs portfolio asset">
Wrong move: think "I kind of remember that this prompt worked before." Better move: store the prompt together with the scenario, the risks, and the verification method. One depends on memory. The other becomes a usable asset.
</Callout>

## Unit Summary

- Vagueness, greed, and missing verification are three of the most common prompt anti-patterns.
- Anti-patterns become teachable when you trace the whole failure chain.
- A portfolio turns prompts from one-off chat into reusable work assets.
