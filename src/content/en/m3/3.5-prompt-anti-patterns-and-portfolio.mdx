## Step 1: Recognize the Common Anti-Patterns

Three common prompt problems show up again and again: vagueness, over-assumption, and no verification. A vague prompt gives AI too little to work with. An over-assuming prompt lets AI decide too much for you. A prompt with no verification trusts the output without checking it.

Most prompt failures are some combination of those three.

Compare the patterns:

- Weak: "Improve this"
- Better: "Improve this lesson page by tightening the introduction and keeping the route structure unchanged"

## Step 2: Build Your Prompt Library

Once you find prompts that actually work, keep them. Group them by task type, project, or result. A small prompt library turns ad hoc effort into a reusable asset.

This is where prompt writing starts to look like engineering rather than improvisation.

Prompt library entries are most useful when they include a stable use case, a known result, and the context they depend on.

For example, you might keep one prompt for "rewrite lesson intros," one for "generate PRD outlines," and one for "review route-safe edits."

Two higher-level anti-patterns to watch for are context pollution and hallucination bait. A prompt like "use everything we have discussed so far and make it amazing" often mixes too much noise into the context, while "guess the missing requirement" invites the model to invent details.

## Step 3: Make the Library Reviewable

Good prompt assets should include context, the prompt itself, the result, and any lessons learned. That way the prompt is not just saved, but actually explainable and improvable.

If a prompt cannot be reviewed later, it is not really an asset yet.
