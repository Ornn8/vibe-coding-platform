# 课程架构 v3 — 方法论内核 + 学习单元制

> 解决两个核心问题：
> 1. vibe-coding-cn 的方法论没有真正融入课程
> 2. 40 节课的扁平列表造成学习压力，缺乏连续性

---

## 一、方法论内核：四大原则贯穿全程

以下四个原则来自 vibe-coding-cn，是整个课程的思想主线。
它们不是"某一节课的内容"，而是"每节课都必须体现的行为准则"。

### 原则 1：规划驱动

> 核心：规划就是一切。不谨慎规划就让 AI 开干，代码库会变成无法管理的混乱。

在课程中的体现：
- M1-M2：教用户在动手前先"想清楚要做什么"（灵魂三问、MVP、旅程地图）
- M3：教用户在写 prompt 前先"组织好上下文"（四要素、系统提示词）
- M4 起：每个实操任务必须先写 Plan 再执行，不允许"上来就干"
- 每个 Unit 的练习都要求用户先提交计划/思路，再提交产出

### 原则 2：上下文固定

> 核心：通过 CLAUDE.md、memory-bank、进度文件把项目的"记忆"固化下来，AI 才能在长期协作中保持一致。

在课程中的体现：
- M3：教 CLAUDE.md 的编写不是"学一个文件格式"，而是"学会把隐性知识显性化"
- M4-M5：每个项目必须维护 memory-bank 和 progress.md
- 模块项目的提交清单必须包含 CLAUDE.md

### 原则 3：胶水编程

> 核心：不让 AI 从零生成代码（幻觉根源），而是让 AI 把成熟的、经过验证的模块连接起来。人审查的是连接方式。

在课程中的体现：
- M2：教用户从"我要做一个 XX"转变为"我要用哪些现有模块组合出 XX"
- M4 起：实操项目引导用户选择成熟库/框架，让 AI 做"集成"而非"发明"
- M6：Skill 和 MCP 就是胶水编程在工具层面的体现——AI 通过标准接口连接外部能力

### 原则 4：闭环交付

> 核心：需求 → 上下文文档 → 实施计划 → 分步实现 → 自测 → 进度记录，全程可复盘、可移交。

在课程中的体现：
- 每个模块项目的提交物必须包含完整闭环的全部产出（不只是代码）
- 进度记录（progress.md）从 M4 起是必交项

### 方法论在课程中的渗透方式

不单独开课讲方法论，而是通过以下机制渗透：

| 渗透机制 | 说明 |
|----------|------|
| 每个 Unit 开头的"本节方法论要点" | 1-2 句话提示这个 Unit 对应哪个原则 |
| 练习中的行为约束 | "请先写出你的计划，再开始实操"——强制规划驱动 |
| 项目提交清单 | 必须提交 CLAUDE.md / Plan / 进度记录——强制闭环交付 |
| 场景示例中的对比 | 每次 before/after 对比都暗含方法论——为什么"好的做法"好？因为遵循了原则 |
| 阶段性"方法论回顾"Unit | 每个模块的最后一个 Unit 做回顾：这个模块你践行了哪些原则？ |

## 二、学习单元制（Unit-Step 结构）

### 问题
40 节课 → 用户看到导航栏里密密麻麻的链接 → 产生压力 → 不想开始

### 解决方案
引入 **Unit（学习单元）** 作为中间层。

```
Module（模块）
└── Unit（学习单元，导航栏中可见，4-5 个/模块）
    └── Step（步骤，Unit 页面内的区块，2-4 个/Unit）
        └── Exercise（练习，嵌入在 Step 中）
```

### 用户体验变化

| 维度 | 旧设计（v2） | 新设计（v3） |
|------|-------------|-------------|
| 导航栏看到的 | M3 下面 12 个链接 | M3 下面 5 个 Unit |
| 点进一个链接 | 一篇 800 字文章 + 3 个练习 | 一个 30-45 分钟的学习单元，包含 2-4 个 Step，连续滚动 |
| 视觉感受 | "好多课要上" | "每个模块就几个单元，每个单元一口气读完" |
| 学习节奏 | 频繁在页面间跳转 | 在一个页面里连续流动，像读一篇好文章 |
| 颗粒度 | Step 级别的细练习 | 不变——练习仍然在 Step 级别 |

### 单个 Unit 页面的内部结构

```
┌─────────────────────────────────────────────┐
│ Unit 标题                                    │
│ 预计时长：35 分钟 │ 方法论要点：规划驱动      │
├─────────────────────────────────────────────┤
│                                             │
│ Step 1：[知识点标题]                          │
│ [讲解 + 场景示例 + 对比]                      │
│ 练习 1.1（识别题）                           │
│ 练习 1.2（模仿题）                           │
│                                             │
│ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  分隔线  ─ ─ ─ ─ ─ │
│                                             │
│ Step 2：[知识点标题]                          │
│ [讲解 + 场景示例 + 对比]                      │
│ 练习 2.1（模仿题）                           │
│ 练习 2.2（应用题）                           │
│                                             │
│ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  分隔线  ─ ─ ─ ─ ─ │
│                                             │
│ Step 3：[知识点标题]                          │
│ [讲解 + 场景示例]                            │
│ 练习 3.1（应用题）                           │
│ 练习 3.2（创造题，可选）                      │
│                                             │
├─────────────────────────────────────────────┤
│ 本单元小结 + 方法论回顾                       │
│ 下一单元预告                                 │
│ [上一单元] [下一单元]                         │
└─────────────────────────────────────────────┘
```

### 关键设计细节

1. **一个 Unit = 一个 MDX 文件**（不是多个）。Step 是文件内的 `##` 标题区块
2. **导航栏只展示 Unit 级别**，不展示 Step。用户不需要知道一个 Unit 里有几个 Step
3. **Unit 内部有锚点导航**（页面右侧的目录），用户可以跳转到特定 Step
4. **每个 Unit 页面顶部显示**：预计时长、本单元的方法论要点标签、进度（3/3 Step 完成）
5. **Step 完成状态**：每个 Step 的最后一个练习提交后，该 Step 标记为完成。全部 Step 完成 → Unit 完成

## 三、M1-M4 重新编排：Unit-Step 结构

### M1：觉醒与认知（4 个 Unit）

#### Unit 1.1：Vibe Coding 是什么，为什么现在要学
预计时长：25 分钟

| Step | 知识点 | 练习 |
|------|--------|------|
| Step 1 | Vibe Coding 的起源和定义 | L1：用自己的话解释给朋友听 |
| Step 2 | 三种模式对比（Vibe / Spec / Agentic） | L1：场景判断题（3 个场景选模式）；L2：为一个新场景判断适合的模式 |
| Step 3 | 为什么 2026 年是最好的学习时机 | L2：去搜索一条最近的 vibe coding 新闻，用灵魂三问分析它 |

方法论渗透：引出"规划驱动"概念——Vibe Coding 好用但容易失控，本课程教你如何有控制地 vibe

#### Unit 1.2：工具地图——选对你的第一把武器
预计时长：30 分钟

| Step | 知识点 | 练习 |
|------|--------|------|
| Step 1 | 浏览器工具：Bolt.new / Lovable / Replit（功能、适用场景、免费额度） | L1：对比表格填空——每个工具最适合什么场景 |
| Step 2 | IDE 工具：Cursor / Windsurf（和浏览器工具的区别、什么时候该"升级"） | L1：判断题——给定场景，选浏览器工具还是 IDE |
| Step 3 | CLI 工具：Claude Code / Codex（简要介绍，不上手，让用户知道"后面会学到"） | L2：选择你的起步工具并注册，写出选择理由 |

方法论渗透：工具选择本身就是"规划"的一部分——用什么工具取决于你要做什么

#### Unit 1.3：动手——你的第一个 AI 生成网页
预计时长：35 分钟

| Step | 知识点 | 练习 |
|------|--------|------|
| Step 1 | 跟着做：用 Bolt.new 生成一个个人介绍页面（完整步骤 + prompt 示例） | L2：照着示例 prompt 生成你自己的页面 |
| Step 2 | 学会迭代：对 AI 说"改一下"（改颜色、加元素、调布局的具体 prompt 示例） | L2：对你的页面做 3 次修改，每次记录 prompt 和结果 |
| Step 3 | 观察 AI 的产出：它生成了什么文件？每个文件是干什么的？（不需要全懂） | L3：描述你看到的文件结构，猜测每个文件的作用 |

方法论渗透：即使是最简单的任务，"先想清楚要什么再动手"也比"随便试试"效果好

#### Unit 1.4：模块项目——做一个完整的静态网站
预计时长：40 分钟（指导 + 开始做，实际项目可能需要更多时间）

| Step | 知识点 | 练习 |
|------|--------|------|
| Step 1 | 项目要求详解 + 优秀示例展示（展示 2-3 个学员作品截图和 prompt 记录） | L1：分析一个示例作品——它做对了什么 |
| Step 2 | 常见问题预防：一次给太多需求、风格描述太模糊、不知道怎么分页 | L2：写出你的项目计划（选题 + 3 个页面的内容描述） |
| Step 3 | 动手开始 + 提交指导 | L3：完成项目并提交（部署链接 + prompt 记录） |

方法论渗透：这是第一次体验"先计划再执行"的节奏——Step 2 的计划就是 Step 3 的指南

### M2：需求定义与产品思维（5 个 Unit）

#### Unit 2.1：为什么你的 Vibe Coding 总是翻车
预计时长：25 分钟

| Step | 知识点 | 练习 |
|------|--------|------|
| Step 1 | 一个真实的翻车案例：从"帮我做一个博客"到 15 轮后项目崩溃（完整过程复盘） | L1：这个案例犯了哪些错误？（选择题） |
| Step 2 | 翻车的根因分析：不是工具不行，是需求不清——引出"需求定义"的重要性 | L2：回忆你自己的一次翻车经历，分析根因 |

方法论渗透：正式引入"规划驱动"原则——用翻车案例让用户"痛过"之后才讲方法

#### Unit 2.2：灵魂三问——用户、痛点、方案
预计时长：40 分钟

| Step | 知识点 | 练习 |
|------|--------|------|
| Step 1 | 第一问：用户是谁（怎么画一个具体的用户画像，not "所有人"） | L1：判断题——哪个用户描述更好？；L2：为"背单词 App"写一个用户画像 |
| Step 2 | 第二问：痛点在哪（真痛点 vs 伪需求，用户现在怎么解决） | L1：区分真痛点和伪需求；L2：分析你选的 App idea 的用户痛点 |
| Step 3 | 第三问：为何用你（差异化，不需要"比淘宝好"，只需要"在某个点上不同"） | L2：写出你的 App 和现有方案的差异点 |
| Step 4 | 综合实践：把三个答案组合成一段"AI 能读懂的需求描述" | L3：完整回答灵魂三问，写成一段连贯的需求描述 |

方法论渗透：灵魂三问就是"规划驱动"在需求阶段的具体表现

#### Unit 2.3：MVP 与功能优先级
预计时长：35 分钟

| Step | 知识点 | 练习 |
|------|--------|------|
| Step 1 | MVP 是什么：不是缩水版，是"验证核心假设的最短路径"（用具体产品的 MVP 故事说明） | L1：判断题——哪个是 MVP，哪个是"缩水版" |
| Step 2 | P0/P1/P2 分级法：怎么判断（没有它不能用 = P0 / 体验更好 = P1 / 锦上添花 = P2） | L1：给出 10 个功能列表，做分级；L2：用同样方法给自己的项目分级 |
| Step 3 | 为什么功能堆砌是 vibe coding 的大敌（AI 一次处理太多功能会失控——联系翻车案例） | L3：把你的 P0 功能列表写成一段 prompt，测试 AI 的反应 |

方法论渗透：P0/P1/P2 就是"胶水编程"思维的前置——先定义要连接什么，而不是让 AI 发明什么

#### Unit 2.4：用户旅程地图与需求表达
预计时长：35 分钟

| Step | 知识点 | 练习 |
|------|--------|------|
| Step 1 | 用户旅程地图：从用户视角描述"打开→使用→完成"（和功能列表的区别） | L1：同一个 App，功能列表和旅程地图的对比——哪个 AI 更容易理解？ |
| Step 2 | 怎么画旅程地图（模板 + 记账 App 完整示例） | L2：照着模板为你的项目画旅程地图 |
| Step 3 | 需求表达的五种常见错误（太模糊/太贪心/只要结果/没约束/跳过需求定技术） | L1：识别 5 个错误案例；L2：改写 3 个差需求 |

方法论渗透："闭环交付"的起点——旅程地图就是从需求到实施计划的桥梁

#### Unit 2.5：模块项目——需求文档 + MVP 实现
预计时长：45 分钟

| Step | 知识点 | 练习 |
|------|--------|------|
| Step 1 | 需求文档模板详解 + 示例（番茄钟 / 记账 App 的完整需求文档） | L1：分析示例文档——它包含了灵魂三问的哪些内容 |
| Step 2 | 你的需求文档：带着模板一步步填写 | L3：完成你的项目需求文档 |
| Step 3 | 用 GUI 工具实现 MVP + 提交指导 | L3：提交需求文档 + MVP 部署链接 + prompt 日志 |

方法论渗透：第一次完整体验"闭环交付"——从需求文档到可运行产品

### M3：提示词工程（5 个 Unit）

#### Unit 3.1：上下文决定一切
预计时长：30 分钟

| Step | 知识点 | 练习 |
|------|--------|------|
| Step 1 | 对比实验：同一个需求，差 prompt vs 好 prompt 的真实产出（给出具体代码/截图对比） | L1：看完对比，列出 3 点差异 |
| Step 2 | 什么是上下文：AI 做决策的全部信息来源（任务级/项目级/系统级三层模型） | L1：判断题——以下信息属于哪个层级 |
| Step 3 | 你动手试：用差 prompt 和好 prompt 分别生成同一个功能，记录对比 | L3：提交两个 prompt 和对比分析 |

方法论渗透："上下文固定"原则的入门——理解 AI 的产出质量由上下文决定

#### Unit 3.2：提示词四要素（上）——角色 + 任务
预计时长：40 分钟

| Step | 知识点 | 练习 |
|------|--------|------|
| Step 1 | 角色设定：告诉 AI 用什么身份思考（日常比喻 + 3 个场景对比） | L1：哪个角色设定更好？；L2：为 3 个任务写角色设定 |
| Step 2 | 角色设定的写法模板 + 常见错误（"你是专家"没有意义） | L2：用模板改写 3 个差的角色设定 |
| Step 3 | 任务描述：把"你想要什么"说具体（模糊 vs 具体的 5 组 before/after） | L1：判断哪个任务描述更具体；L2：改写 3 个模糊描述 |
| Step 4 | 综合练习：角色 + 任务组合 | L3：为自己的项目写一个包含角色+任务的 prompt，测试效果 |

方法论渗透：角色设定就是"上下文固定"在单次对话中的应用

#### Unit 3.3：提示词四要素（下）——约束 + 格式
预计时长：40 分钟

| Step | 知识点 | 练习 |
|------|--------|------|
| Step 1 | 约束条件：给 AI 画框框（技术约束/风格约束/"不要做什么"三类 + 场景） | L1：识别缺少约束的 prompt；L2：为 3 个 prompt 补充约束 |
| Step 2 | 输出格式：要代码还是文档？什么结构？分几个文件？ | L2：为 3 个不同任务指定输出格式 |
| Step 3 | 四要素组装：一个完整 prompt 的构建过程（从零到完整的演示） | L3：用四要素写一个完整 prompt 并让 AI 执行 |
| Step 4 | 四要素检查清单 + 常见遗漏 | L3：用检查清单审查你在 M2 项目中写过的 prompt，改进 3 个 |

方法论渗透：四要素就是"规划驱动"在 prompt 层面的体现——写 prompt 也需要先规划

#### Unit 3.4：系统提示词与 CLAUDE.md
预计时长：40 分钟

| Step | 知识点 | 练习 |
|------|--------|------|
| Step 1 | 什么是系统提示词：项目级的"工作手册"（在不同工具中的形态对比） | L1：CLAUDE.md / AGENTS.md / .cursorrules 对比——它们解决什么问题 |
| Step 2 | CLAUDE.md 的结构详解（用本平台自身的 CLAUDE.md 作为真实案例，逐段讲解） | L2：分析本平台的 CLAUDE.md——每一段为什么这样写 |
| Step 3 | 动手写你的 CLAUDE.md（模板 + 一步步填写指导） | L3：为你的 M2 项目写一份 CLAUDE.md |
| Step 4 | 系统提示词构建原则（来自 vibe-coding-cn 的方法论：清晰性、一致性、约束边界） | L3：用构建原则检查并改进你的 CLAUDE.md |

方法论渗透：这就是"上下文固定"原则的核心实践——CLAUDE.md 是把隐性知识显性化的工具

#### Unit 3.5：Prompt 反模式与个人提示词库
预计时长：35 分钟

| Step | 知识点 | 练习 |
|------|--------|------|
| Step 1 | 三大反模式（模糊/贪心/无约束）+ 每个都有 before/after 对比和真实失败场景 | L1：识别反模式；L2：改写 3 个反模式 prompt |
| Step 2 | 两大高级反模式（上下文污染/幻觉引导）+ 对策 | L1：判断题；L2：描述你遇到过的类似情况 |
| Step 3 | Prompt Portfolio：怎么收集、分类、迭代你的 prompt 资产 + 四大常用模板 | L3：整理你学习至今最好的 5 个 prompt 到 Portfolio |

方法论渗透：Prompt Portfolio 是"闭环交付"中"可复盘、可移交"的 prompt 层面体现

### M4：规划驱动与工作流（5 个 Unit）

#### Unit 4.1：规划就是一切——PRD 与 Implementation Plan
预计时长：45 分钟

| Step | 知识点 | 练习 |
|------|--------|------|
| Step 1 | 翻车案例深度分析：15 轮对话从能用到崩溃的完整复盘 | L1：在每一轮标注"这里应该怎么做" |
| Step 2 | PRD：写给 AI 看的需求文档（模板 + 番茄钟完整示例） | L2：参照模板为一个新项目写 PRD |
| Step 3 | Implementation Plan：把 PRD 变成执行步骤（模板 + 同一个番茄钟的拆解示例） | L2：把你的 PRD 拆成 5-8 步 |
| Step 4 | 胶水编程思维：选择成熟模块而不是让 AI 发明（介绍"选库"的思路） | L3：为你的项目列出可以使用的成熟库/框架 |

## 四、数据模型调整

### Prisma Schema 变更

需要新增 Unit 层：

```prisma
model Unit {
  id       String @id @default(cuid())
  slug     String  // "1.1-what-is-vibe-coding"
  order    Int
  moduleId String

  title           Json    // { "zh": "...", "en": "..." }
  description     Json?
  estimatedMins   Int     @default(30)
  methodologyTag  String? // "规划驱动" / "上下文固定" / "胶水编程" / "闭环交付"

  module Module @relation(fields: [moduleId], references: [id], onDelete: Cascade)
  // Steps 不在数据库中，是 MDX 文件内的 ## 标题区块
  // 练习仍然通过 Exercise 表管理，关联到 Unit

  exercises Exercise[]
  progress  UnitProgress[]

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@unique([moduleId, slug])
  @@unique([moduleId, order])
  @@map("units")
}

// 替换 LessonProgress
model UnitProgress {
  id     String @id @default(cuid())
  userId String
  unitId String

  completed   Boolean   @default(false)
  completedAt DateTime?
  studyMins   Int       @default(0)

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)
  unit Unit @relation(fields: [unitId], references: [id], onDelete: Cascade)

  @@unique([userId, unitId])
  @@map("unit_progress")
}
```

原有的 Lesson 表替换为 Unit 表。Step 不入库，是 MDX 内的结构。

### MDX 文件结构变更

```
src/content/zh/m1/
├── 1.1-what-is-vibe-coding.mdx          # Unit 1.1（内含 3 个 Step）
├── 1.2-tool-landscape.mdx               # Unit 1.2（内含 3 个 Step）

└── 1.4-module-project.mdx               # Unit 1.4（内含 3 个 Step）
```

每个 MDX 文件对应一个 Unit，内部用 `## Step 1: ...` 分隔。
